# 4.3 Tool Execution in OpenClaw

**How OpenClaw's tools work, how they compose, and how the agent recovers from errors**

---

## The Four Core Tools

OpenClaw renamed Pi's `bash` tool to `exec`, but the behavior is identical. Everything else carries over.

### 1. `read(path)`

Reads a file or lists a directory. Under the hood: `cat /path/to/file` or `ls /path/to/dir`.

```json
{"tool": "read", "path": "workspace/reports/"}
```

Returns a directory listing if the path is a directory. Returns file contents if it's a file. The agent reads files to understand data, verify output, or load a skill's documentation.

---

### 2. `write(path, content)`

Creates or overwrites a file with the given content.

```json
{
  "tool": "write",
  "path": "workspace/tmp/analyze.py",
  "content": "#!/usr/bin/env python3\nimport json, sys\n..."
}
```

The agent uses `write` to create scripts before running them, to save intermediate data, and to produce final output files. **Write does not execute anything** — it just puts bytes on disk.

---

### 3. `edit(path, old_string, new_string)`

Finds `old_string` in the file and replaces it with `new_string`. More token-efficient than rewriting an entire file when you only need to change a few lines.

```json
{
  "tool": "edit",
  "path": "workspace/tmp/query.sql",
  "old_string": "WHERE month = '2026-01'",
  "new_string": "WHERE month = '2026-02'"
}
```

The agent uses `edit` to update scripts it already wrote, fix bugs in generated code, or tweak configurations.

---

### 4. `exec(command)`

Runs a bash command. Captures stdout, stderr, and exit code. Returns all of it to the agent.

```json
{
  "tool": "exec",
  "command": "python3 workspace/tmp/analyze.py workspace/tmp/claims-raw.csv"
}
```

**This is where computation happens.** Everything the agent can't do with read/write/edit, it does through `exec`. Database queries, API calls, data processing, format conversion, file manipulation — all of it goes through exec.

---

## Additional Tools

### `message(text)`

Sends a message back to the user via whatever surface they're on (Telegram, web chat, etc.). This is how the agent communicates completion or asks clarifying questions.

```json
{"tool": "message", "text": "Done! February report saved to reports/2026-02-claims.html"}
```

In Pi, the agent just stops calling tools and prints to terminal. In OpenClaw, the agent explicitly calls `message` to notify you. The distinction matters: you might be away from the computer when the job finishes.

---

### `memory_search(query)`

Semantic search across workspace memory files. The agent uses this when it needs to find relevant past context but doesn't know which file it's in.

```json
{"tool": "memory_search", "query": "NPI lookup API credentials"}
```

Returns the most semantically similar passages from your memory files. Slower and more expensive than `read`, so the agent typically tries `read(AGENTS.md)` first and falls back to `memory_search` for fuzzier recall.

---

### `browser(action, url, ...)`

Web automation tool. Optional — not all OpenClaw installations enable it. Handles cases where a plain `curl` or `wget` isn't enough (JavaScript-rendered pages, login flows, etc.).

```json
{"tool": "browser", "action": "navigate", "url": "https://npiregistry.cms.hhs.gov/search"}
```

---

## How `exec` Works in Detail

When the agent calls `exec`, OpenClaw runs:

```bash
bash -c "your command here"
```

The result returned to the agent looks like:

```
stdout: [everything printed to stdout]
stderr: [everything printed to stderr]
exit_code: 0
```

**The agent sees all three.** This is important: if your script prints warnings to stderr but succeeds (exit code 0), the agent sees the warnings and can decide if they matter. If the script fails (non-zero exit code), the agent sees exactly what went wrong.

---

## Error Handling: Non-Zero Exit Codes

When a command fails, the agent does not stop. It sees the failure and reasons about what to do next.

**Example:**

```
Turn 4:
  exec: psql -h localhost -d claims -f query.sql

Result:
  stderr: psql: error: connection to server on socket
          "/var/run/postgresql/.s.PGSQL.5432" failed:
          No such file or directory
  exit_code: 1
```

**Agent's next action:** It sees the connection error. It might:
- Try a different connection string (if AGENTS.md has alternatives)
- Read AGENTS.md to find the correct DB host
- Call `exec("pg_isready -h db.internal")` to check connectivity
- Message the user: "Database connection failed — is the DB accessible from this machine?"

The agent adapts. It doesn't just give up. This is why the while loop architecture works: errors are just more information.

---

## Sequential vs. Parallel Execution

OpenClaw's agent loop is **sequential by default** — one tool call per turn, wait for result, then decide next action. This is by design: the LLM needs to see each result before choosing the next step.

**Sequential (normal):**
```
Turn 1: exec(query_claims.sh) → wait → see result
Turn 2: exec(analyze.py) → wait → see result
Turn 3: exec(render_report.py) → wait → see result
```

**When the agent fakes parallelism:** If two operations are truly independent, the agent can write a shell script that runs them in parallel and call that script as one exec:

```bash
# Agent writes this as a single script:
python3 analyze_claims.py claims.csv &
python3 analyze_denials.py denials.csv &
wait
echo "Both complete"
```

This is a composability pattern: use exec to run a script that itself uses parallelism. The agent's loop stays sequential; the parallelism lives in the bash layer.

## File-Based Composition

The most important tool composition pattern in OpenClaw: **pass data between steps via files, not pipes**.

### Why not pipes?

```bash
# Agent could call this:
exec("cat claims.csv | python analyze.py | pandoc -o report.html")
```

This works but has problems:
- If `python analyze.py` fails, the agent can't inspect the intermediate data
- The analysis output disappears — it was never saved
- You can't resume from the middle if something goes wrong

### File-based alternative:

```
Turn 1: exec(query_claims.sh > tmp/claims.csv)
Turn 2: exec(python analyze.py tmp/claims.csv > tmp/stats.json)
Turn 3: read(tmp/stats.json)                    ← agent verifies data
Turn 4: exec(render_report.py tmp/stats.json > tmp/report.md)
Turn 5: exec(pandoc tmp/report.md -o reports/feb.html)
```

Each intermediate result is on disk. The agent can read `tmp/stats.json` to verify the numbers before rendering. If the render fails, `tmp/stats.json` still exists — you can restart from Turn 4. You can inspect any intermediate state by opening the file yourself.

> **Rule:** Write intermediate results to files. Files are checkpoints. Pipes are black boxes.

---

## Output Truncation

Bash commands can produce huge output. A database query might return 50,000 lines. OpenClaw handles this with truncation: if output exceeds a threshold, the agent receives the first N lines plus a note that output was truncated.

**The agent's response to truncation:** It typically writes output to a file instead:

```
Turn 2: exec("psql -d claims -c 'SELECT * FROM claims' > tmp/claims.csv")
```

Now the full data is in a file, not in context. The agent reads summary statistics, not every row. This is the correct pattern for large datasets anyway — you never want raw row data in your agent's context window.

---

## The Real Pattern: Write Script → Run Script → Read Output

For any non-trivial computation, the agent follows this pattern:

1. **Write** a Python/bash script to a temp file
2. **Exec** the script
3. **Read** or inspect the output file

```
Turn N:   write(tmp/process_npi.py, "#!/usr/bin/env python3\n...")
Turn N+1: exec("chmod +x tmp/process_npi.py && python3 tmp/process_npi.py npi_list.csv > tmp/npi_results.json")
Turn N+2: read(tmp/npi_results.json)
```

This pattern works because:
- The script is explicit and inspectable (you can open it)
- The output is a file (persists, can be read multiple times)
- The agent can fix the script with `edit` if the first run fails

## Enterprise Pattern: SQL → CSV → Report

A common enterprise workflow using tool composition:

```
Turn 1: read(AGENTS.md)
        → loads DB credentials, schema notes

Turn 2: write(tmp/query.sql,
         "SELECT claim_id, amount, denial_code, provider_npi
          FROM claims
          WHERE date_trunc('month', service_date) = '2026-02-01'
          AND status = 'denied'")
        → query on disk, auditable

Turn 3: exec("psql -h db.internal -d claims -f tmp/query.sql -o tmp/denied-claims.csv --csv")
        → denied-claims.csv written

Turn 4: read(tmp/denied-claims.csv)    [first few lines, verify schema]

Turn 5: write(tmp/summarize.py, "...")
Turn 6: exec("python3 tmp/summarize.py tmp/denied-claims.csv > tmp/denial-stats.json")

Turn 7: exec("python3 skills/reports/render_html.py tmp/denial-stats.json > reports/feb-denials.html")

Turn 8: message("February denial analysis complete: 847 denied claims, $2.1M total.\nreports/feb-denials.html")
```

Every intermediate file is preserved. You can rerun any step independently. The entire workflow is auditable.

## Common Mistakes

**Mistake 1: Giant one-liner commands**
```bash
# Bad — hard to debug, no intermediate files
exec("psql -d claims -c 'SELECT ...' | python analyze.py | pandoc -o report.html")
```
If anything in that chain fails, you don't know where. Break it into steps.

---

**Mistake 2: Not reading intermediate files**

If you generate `stats.json` but never read it, you're trusting that the script did the right thing. The agent should read key intermediate files to catch errors early.

---

**Mistake 3: Ignoring exit codes**

The agent can see exit codes but might not always check them carefully if your AGENTS.md doesn't emphasize it. Add a note like: "Always verify exit code 0 before proceeding to next step in report workflows."

---

**Mistake 4: Putting credentials in scripts**

Write credentials to separate files or use environment variables. Your AGENTS.md should contain connection info; scripts should read from env or config files, not have hardcoded values.
