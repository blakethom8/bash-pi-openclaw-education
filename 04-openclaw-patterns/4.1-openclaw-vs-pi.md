# 4.1 OpenClaw vs. Pi

**How OpenClaw extends Pi's minimal architecture for production use**

---

## Pi in One Sentence

> "Pi is a while loop that calls an LLM with four tools."

That's the whole system. A loop. An LLM. Four tools. Everything an agent needs to read files, write files, edit files, and run bash commands.

Pi is deliberately minimal. It was designed as a coding harness — something you run in a terminal to help build software. It has no authentication, no message routing, no persistence between sessions, no multi-user support. Just the loop.

---

## What OpenClaw Is

OpenClaw is Peter Steinberger's framework that takes Pi's philosophy and wraps it in production infrastructure. The core loop is **identical** — same while loop, same four tools, same "LLM decides what to do next" philosophy. What OpenClaw adds is everything you need to run that loop as a service that you can reach from your phone.

```
Pi:         Terminal → While Loop → 4 Tools → Your filesystem

OpenClaw:   Telegram/Web/WhatsApp
                 ↓
            Gateway (routes messages, manages auth)
                 ↓
            Agent Session (Pi's while loop, verbatim)
                 ↓
            4 Tools + Skills + Memory → Your filesystem
```

The agent loop itself did not change. OpenClaw built a building around it.

---

## Side-by-Side Comparison

| Feature | Pi | OpenClaw |
|---------|----|-----------|
| **Core loop** | While loop | Same while loop |
| **Tools** | read, write, edit, bash | read, write, edit, exec (+ extras) |
| **Interface** | Terminal, you run it | Telegram, web chat, WhatsApp |
| **Sessions** | Single session, terminal lifetime | Persistent, named, multi-turn |
| **Users** | You, locally | Multiple users via auth |
| **Memory** | Files you manage manually | MEMORY.md + semantic search |
| **Skills** | Markdown files you write | Same, plus skill discovery |
| **Deployment** | Run it yourself | Runs as a service |
| **Message routing** | N/A | Gateway handles it |

**Same foundation. Different operational context.**

## What OpenClaw Adds

### The Gateway

Pi has no concept of "where did this message come from." You type into a terminal, it responds in the terminal. OpenClaw adds a Gateway layer that accepts messages from Telegram, a web chat UI, WhatsApp, or any other surface you configure. The Gateway normalizes all those inputs into the same format the agent session receives.

This is what lets you send "run the monthly provider report" from your phone at 7am and come back to a finished HTML file.

### Persistent Sessions

In Pi, when your terminal session ends, so does the agent's context. OpenClaw sessions persist. You can start a conversation Monday, pick it up Wednesday. The agent still has the full history. You can have multiple named sessions — one for reports, one for data work, one for research.

### Skills System

Pi's skill pattern (write markdown docs, write bash scripts, read them as needed) is formalized in OpenClaw. Skills live in a discoverable directory structure. The agent learns about available skills from the system prompt (one-line descriptions), then reads the full SKILL.md when it needs to actually use one.

### Memory

OpenClaw's memory has two layers:
1. **File memory** — MEMORY.md, USER.md, daily notes in memory/ — same as Pi's file-based approach
2. **Semantic search** — the `memory_search` tool lets the agent find relevant past context even when it doesn't know exactly which file to look in

### Additional Tools

```
Core tools (same as Pi):
  read           — cat file, list directory
  write          — write file contents
  edit           — search/replace in file
  exec           — bash -c "command"   (renamed from bash)

Added tools:
  message        — send a message back to user (Telegram/web)
  memory_search  — semantic search across memory files
  browser        — web automation (optional, not always enabled)
```

The `message` tool is the key addition for production use. In Pi, the agent prints to your terminal when done. In OpenClaw, the agent calls `message("Done! Report saved to reports/q1.html")` and you get a Telegram notification.

## The Workspace: Your Files Explained

OpenClaw's workspace lives at `~/.openclaw/workspace/` (sometimes `~/openclaw/workspace/`).

**AGENTS.md** — Instructions for the agent. This is where you put:
- What this agent does ("You help generate insurance claims reports")
- Connections and credentials ("DB host: postgres.internal, DB name: claims")
- Output conventions ("Save all reports to workspace/reports/ as HTML")
- Recurring task descriptions ("Monthly claims report = run query_claims.sh for the month")

**SOUL.md** — The agent's persona. Tone, personality, how it communicates. Most users keep this simple.

**USER.md** — Information about you that helps the agent. Your role, your preferences, your timezone, what you care about.

**MEMORY.md** — Persistent facts the agent has learned. The agent appends to this when it learns something important about your preferences or workflow. You can also edit it directly.

**memory/** — Daily notes directory. The agent writes `memory/2026-02-09.md` as a log of what it did each day. Useful for picking up context in future sessions.

> **Key insight:** The workspace is just a directory. You can `cat AGENTS.md`, `ls reports/`, `open reports/february.html`. It's transparent by design. Files are truth.

## Why This Matters for Enterprise Work

If you're running agents for business workflows — generating reports, querying claim databases, looking up NPI numbers, summarizing provider search results — OpenClaw's production features matter:

1. **You don't have to be at a terminal.** Fire off "generate February claims report" from Telegram, get the finished HTML file notification when it's done.

2. **Sessions survive long jobs.** A complex report workflow might take 8 turns and several minutes. OpenClaw's persistent sessions handle this without you watching.

3. **AGENTS.md is your ops manual.** Put your database connection strings, report templates, query patterns, and naming conventions in AGENTS.md. The agent reads it at the start of every session. You update it once and all future sessions benefit.

4. **Skills encapsulate tribal knowledge.** Your claims query patterns, your NPI lookup scripts, your PowerPoint generation code — all of it becomes a skill. The agent knows it exists, reads the docs when needed, runs the scripts.

---

## The Core Loop Is Still Pi

It's worth being explicit: if you stripped away the Gateway, the session management, the memory search, and the message tool, what remains is Pi's while loop running against your filesystem with four tools. OpenClaw did not invent a new kind of agent. It made Pi's agent operational at work.

This means everything you learned about Pi's decision-making — how the LLM figures out what to do, how tool results accumulate in context, how the loop ends — applies directly to OpenClaw. The model is the same.
