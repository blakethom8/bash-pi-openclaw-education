# 4.2 Multi-Turn Conversation Flow

**How OpenClaw builds up work across multiple turns to complete complex tasks**

---

## What "Multi-Turn" Means

A single request to an agent rarely finishes in one step. When you ask "summarize this podcast episode," the agent doesn't just call one tool and respond. It has to find the video, extract the transcript, clean the text, run it through an LLM, format the output, and write the file. That's six or seven distinct operations — six or seven turns of the while loop.

**A turn** = one cycle of: LLM decides what to do → calls a tool → sees the result → decides what to do next.

Multi-turn is the normal mode. It's not an edge case — it's how every non-trivial task works. The agent accumulates context across turns, each result informing the next decision.

```
User request
    │
    ▼
Turn 1: LLM → tool call → result appended to context
Turn 2: LLM sees turn 1 result → new tool call → result appended
Turn 3: LLM sees turns 1+2 → new tool call → result appended
    ...
Turn N: LLM sees all prior results → no more tool calls → message("Done!")
```

## Walkthrough: Podcast Summarization

### The request:
> "Summarize the Syntax.fm podcast about Pi — the one with Peter and Armin"

---

### Turn 1: Find the episode

**LLM thinks:** I need to find the specific episode. I'll search for it.

**Tool called:**
```json
{
  "tool": "exec",
  "command": "web_search 'syntax.fm podcast Pi agent Peter Steinberger Armin Ronacher'"
}
```

**Result added to context:**
```
Found: Syntax Episode 976 - "Building an AI Agent That Runs Your Computer"
URL: https://syntax.fm/976
YouTube: https://www.youtube.com/watch?v=...
```

---

### Turn 2: Extract captions

**LLM thinks:** I have the YouTube URL. Use yt-dlp to get auto-generated captions.

**Tool called:**
```json
{
  "tool": "exec",
  "command": "yt-dlp --write-auto-subs --skip-download --sub-lang en -o 'podcasts/syntax-976' 'https://youtube.com/watch?v=...'"
}
```

**Result added to context:**
```
[download] Writing video subtitles to: podcasts/syntax-976.en.vtt
Exit code: 0
```

---

### Turn 3: Clean the VTT transcript

**LLM thinks:** VTT files have timestamps and duplicate lines. I need to clean it into readable text.

**Tool called (write):**
```json
{
  "tool": "write",
  "path": "podcasts/clean_vtt.py",
  "content": "#!/usr/bin/env python3\n..."
}
```

**Tool called (exec):**
```json
{
  "tool": "exec",
  "command": "python3 podcasts/clean_vtt.py podcasts/syntax-976.en.vtt > podcasts/transcript.txt"
}
```

**Result:** `Exit code: 0` — transcript.txt written

---

### Turn 4: Read and summarize

**LLM thinks:** I have clean text. Read it, then generate the summary.

**Tool called:**
```json
{
  "tool": "read",
  "path": "podcasts/transcript.txt"
}
```

**Result:** Full transcript added to context. LLM now has all the material it needs.

---

### Turn 5: Write the output

**LLM thinks:** Generate summary and write both markdown and HTML versions.

**Tool called (write markdown):**
```json
{
  "tool": "write",
  "path": "podcasts/syntax-976-summary.md",
  "content": "# Syntax 976: Building AI Agents\n\n## Key Points\n..."
}
```

**Tool called (write HTML):**
```json
{
  "tool": "exec",
  "command": "pandoc podcasts/syntax-976-summary.md -o podcasts/syntax-976-summary.html --standalone"
}
```

---

### Turn 6: Done

**LLM decides:** No more tool calls needed. Send the completion message.

**Tool called:**
```json
{
  "tool": "message",
  "text": "Done! Podcast summarized in 8 minutes.\n\nFiles:\n• podcasts/syntax-976-summary.md\n• podcasts/syntax-976-summary.html"
}
```

You get a Telegram notification. The session ends.

## Steering Mid-Execution

Because OpenClaw sessions are persistent and the Gateway queues messages, you can send a message while the agent is still working.

**Scenario:** Agent is on Turn 3 processing a long transcript. You send:
> "Also add a section on timestamps for the key moments"

OpenClaw queues your message. When the current turn finishes, the agent sees your steering message in context and adjusts its plan. The next turn incorporates your instruction.

This is fundamentally different from Pi (where there's no message channel mid-run) and it's what makes the enterprise use case practical — you can course-correct without waiting for completion.

---

## Session Persistence: Picking Up Tomorrow

OpenClaw sessions do not expire. If you come back the next day:

```
You: "Can you add a Spanish translation of that podcast summary?"

Agent: [reads session history]
       "I see I already created syntax-976-summary.md yesterday.
        Let me read that and create a translation."

Turn 1: read(podcasts/syntax-976-summary.md)
Turn 2: write(podcasts/syntax-976-summary-es.md)
Turn 3: message("Done! Spanish translation saved.")
```

The agent remembers all prior work in the session. You don't have to re-explain context.

---

## Context Window Management

Context accumulates across turns. A long session eventually approaches the LLM's context limit. OpenClaw handles this with summarization:

1. When context grows large, older turns are **compressed** — the detailed tool call/result pairs get replaced with a summary paragraph
2. Recent turns stay in full detail
3. Important facts get written to MEMORY.md before compression

This means very long sessions gradually lose fine-grained early detail, but key information persists via memory files.

**Practical implication:** For long multi-day projects, the agent should write intermediate results to files (which it will do naturally). Files outlast context; context is transient.

## Enterprise Walkthrough: Monthly Claims Report

### The request:
> "Generate the Q1 claims report for February"

---

### Turn 1: Load agent context

**LLM thinks:** Read AGENTS.md to understand what format this report should be, where the database is, and what scripts exist.

**Tool called:**
```json
{"tool": "read", "path": "AGENTS.md"}
```

**Result:** DB connection strings, report format, preferred output directory, notes about the claims schema.

---

### Turn 2: Query the database

**Tool called:**
```json
{
  "tool": "exec",
  "command": "bash skills/claims-query/scripts/query_claims.sh --month 2026-02 > workspace/tmp/claims-raw.csv"
}
```

**Result:**
```
Exported 14,832 rows to workspace/tmp/claims-raw.csv
Exit code: 0
```

---

### Turn 3: Analyze and calculate statistics

**Tool called:**
```json
{
  "tool": "exec",
  "command": "python3 skills/claims-query/scripts/analyze_claims.py workspace/tmp/claims-raw.csv workspace/tmp/stats.json"
}
```

**Result:**
```
Processing 14,832 claims...
Output written to workspace/tmp/stats.json
Exit code: 0
```

---

### Turn 4: Verify the data

**LLM thinks:** Before generating the report, spot-check the numbers.

**Tool called:**
```json
{"tool": "read", "path": "workspace/tmp/stats.json"}
```

**Result:** Key numbers in context. LLM checks they look reasonable (total claims, average amounts, top denial codes, etc.).

---

### Turn 5: Render the report

**Tool called:**
```json
{
  "tool": "exec",
  "command": "python3 skills/claims-query/scripts/render_report.py workspace/tmp/stats.json workspace/templates/claims-report.md > workspace/tmp/report.md"
}
```

---

### Turn 6: Convert to HTML

**Tool called:**
```json
{
  "tool": "exec",
  "command": "pandoc workspace/tmp/report.md -o workspace/reports/2026-02-claims.html --standalone --css templates/report.css"
}
```

---

### Turn 7: Archive the report

**Tool called:**
```json
{
  "tool": "exec",
  "command": "cp workspace/tmp/stats.json workspace/reports/2026-02-claims-stats.json"
}
```

---

### Turn 8: Notify

**Tool called:**
```json
{
  "tool": "message",
  "text": "February claims report complete.\n\n• 14,832 claims processed\n• Report: reports/2026-02-claims.html\n• Raw stats: reports/2026-02-claims-stats.json\n\nTop denial code: CO-97 (22% of denials)"
}
```

You get the notification. The report is in your workspace.

---

## What Builds Up in Context

After 8 turns, the context contains:
- Your original request
- AGENTS.md contents (DB info, format preferences)
- Script outputs (row counts, processing logs)
- The stats.json data
- Paths to all generated files

The agent has a full audit trail of everything it did. If something looks wrong in the report, you can ask "what query did you run for the claims data?" and the agent can answer from context.

## Practice

**Map out the turns for:** "Summarize last week's provider search results"

Think about:
- What data source does the agent need to read?
- Does it need to write a script, or run an existing one?
- What intermediate files get created?
- What does the final output look like?

<summary>Suggested answer</summary>

**Turn 1:** `read(AGENTS.md)` — learn where provider search results are stored and what format they're in
