# 4.5 Workflow Patterns

**Repeatable patterns for building reliable OpenClaw workflows**

---

## Why Patterns Matter

Ad hoc workflows work once. Patterns work every time. The five patterns below cover the majority of enterprise agent workflows. Each pattern has a clear shape: which tools get called, what intermediate files exist, where things can go wrong.

Once you recognize which pattern a task fits, you can predict how the agent will execute it and how to set it up for success (the right skill, the right AGENTS.md entries, the right template files).

---

## Pattern 1: Read → Process → Write

**Shape:**
```
read(input file)
  → exec(processing script)
  → write(output file)
```

**When to use:** Any data transformation task. You have a file in one format; you need it in another.

**Example:** Convert a raw claims CSV to a cleaned, deduplicated CSV:

```
Turn 1: read(AGENTS.md)                                → load schema info
Turn 2: read(workspace/tmp/claims-raw.csv)             → inspect first N lines
Turn 3: write(workspace/tmp/clean_claims.py, "...")    → write cleaning script
Turn 4: exec(python3 workspace/tmp/clean_claims.py
              workspace/tmp/claims-raw.csv
              > workspace/tmp/claims-clean.csv)
Turn 5: exec(wc -l workspace/tmp/claims-clean.csv)     → verify row count
Turn 6: message("Cleaned: 14,832 → 14,791 rows (41 duplicates removed)")
```

**What makes it reliable:**
- Input file exists before processing starts
- Processing script is written to disk (auditable, rerunnable)
- Output file is a real file (not just printed output)
- Row count verification catches silent failures

## Pattern 2: Fetch → Extract → Format

**Shape:**
```
exec(curl or web_search or API call)
  → exec(extract/parse)
  → write(formatted output)
```

**When to use:** API integrations, web data, external services. You're pulling data from outside your filesystem.

**Example:** Fetch NPI data and produce a formatted provider list:

```
Turn 1: read(skills/npi-lookup/SKILL.md)               → load skill docs
Turn 2: exec(bash skills/npi-lookup/scripts/
              batch_lookup.py npi_list.txt
              > tmp/npi_raw.json)                       → fetch from API
Turn 3: read(tmp/npi_raw.json)                         → verify response shape
Turn 4: write(tmp/format_providers.py, "...")           → write formatter
Turn 5: exec(python3 tmp/format_providers.py
              tmp/npi_raw.json
              > reports/provider-directory.md)
Turn 6: exec(pandoc reports/provider-directory.md
              -o reports/provider-directory.html)
```

**What makes it reliable:**
- Raw API response saved to file before any processing
- Extract and format are separate steps (if formatting fails, raw data is preserved)
- Verification of the raw response catches API errors early

## Pattern 3: Multi-Step Pipeline with Checkpoints

**Shape:**
```
exec(step1) → write(checkpoint1.json)
exec(step2) → write(checkpoint2.json)
exec(step3) → write(checkpoint3.json)
...
```

**When to use:** Long-running jobs, expensive computations, jobs where you want to resume from the middle if something fails.

**Example:** Monthly claims analysis pipeline:

```
Turn 1:  exec(query_claims.sh --month 2026-02 > checkpoints/01-raw.csv)
Turn 2:  exec(python3 deduplicate.py checkpoints/01-raw.csv
               > checkpoints/02-deduped.csv)
Turn 3:  exec(python3 enrich_npi.py checkpoints/02-deduped.csv
               > checkpoints/03-enriched.csv)            ← slow: NPI API calls
Turn 4:  exec(python3 calculate_stats.py checkpoints/03-enriched.csv
               > checkpoints/04-stats.json)
Turn 5:  exec(python3 render_report.py checkpoints/04-stats.json
               > checkpoints/05-report.md)
Turn 6:  exec(pandoc checkpoints/05-report.md -o reports/feb-claims.html)
Turn 7:  message("Done! Feb claims report complete. reports/feb-claims.html")
```

**Key benefit:** If Turn 3 fails midway through (network timeout on NPI lookups), you don't have to re-query the database. Restart from checkpoint 2.

**Directory structure:**
```
workspace/
└── checkpoints/
    ├── 01-raw.csv           # DB query output
    ├── 02-deduped.csv       # After deduplication
    ├── 03-enriched.csv      # After NPI enrichment (may be partial)
    ├── 04-stats.json        # Calculated statistics
    └── 05-report.md         # Rendered markdown
```

**In AGENTS.md, add:** "For multi-step pipelines, write checkpoints to workspace/checkpoints/ with numbered prefixes so steps can be rerun independently."

## Pattern 4: Template + Data → Document

**Shape:**
```
read(template file)
  → read(data file)
  → exec(rendering script)
  → write(output document)
```

**When to use:** Report generation. You have a fixed document structure and variable data. This is the core enterprise pattern — it's how you generate the same report format month after month with fresh data.

**Example:** Monthly claims report from template:

```
Turn 1: read(templates/claims-report.md)               → understand template structure
Turn 2: read(tmp/feb-stats.json)                       → load this month's data
Turn 3: write(tmp/render.py, "...")                    → write template renderer
Turn 4: exec(python3 tmp/render.py
              templates/claims-report.md
              tmp/feb-stats.json
              > tmp/feb-report.md)
Turn 5: exec(pandoc tmp/feb-report.md
              -o reports/2026-02-claims.html
              --standalone --css templates/report.css)
Turn 6: message("Report generated: reports/2026-02-claims.html")
```

**Template file example** (`templates/claims-report.md`):
```markdown
# Claims Report — {{month_name}} {{year}}

## Summary

| Metric | Value |
|--------|-------|
| Total Claims | {{total_claims}} |
| Total Amount | ${{total_amount_formatted}} |
| Denial Rate | {{denial_rate_pct}}% |

## Top Denial Codes

{{#each top_denial_codes}}
- **{{code}}**: {{count}} claims (${{amount_formatted}})
{{/each}}

## Recommendations

{{recommendations}}
```

**Why templates matter:** When the report format needs to change (new section, different table structure), you update the template — not the agent. The agent is just a renderer.

## Pattern 5: Query → Enrich → Report

**Shape:**
```
exec(query_db.sh) → [raw data]
  → exec(enrich.py) → [enriched data]
  → exec(make_report.py) → [report]
  → write(final output)
```

**When to use:** Your claims and provider workflows specifically. You have data in a database, it needs to be joined with external data (NPI registry, provider directory, reference tables), then reported.

**Example:** Provider denial analysis with NPI enrichment:

```
Turn 1: read(AGENTS.md)                                → load DB config, report prefs
Turn 2: read(skills/claims-query/SKILL.md)             → query patterns
Turn 3: read(skills/npi-lookup/SKILL.md)               → enrichment patterns

Turn 4: exec(bash skills/claims-query/scripts/
              query_claims.sh
              --month 2026-02
              --status denied
              --group-by provider
              > tmp/denied-by-provider.csv)

Turn 5: exec(python3 skills/npi-lookup/scripts/
              batch_lookup.py
              tmp/denied-by-provider-npis.txt
              > tmp/provider-details.json)

Turn 6: exec(python3 tmp/join_and_analyze.py
              tmp/denied-by-provider.csv
              tmp/provider-details.json
              > tmp/enriched-analysis.json)

Turn 7: exec(python3 skills/claims-query/scripts/
              render_report.py
              tmp/enriched-analysis.json
              templates/denial-analysis.md
              > reports/2026-02-denial-analysis.html)

Turn 8: message("Provider denial analysis ready. Top 3 providers by denial rate attached.")
```

**What this pattern produces that a raw SQL report can't:**
- Provider names and addresses (from NPI)
- Taxonomy codes (specialty information)
- Contact information
- Geolocation (from NPI address)

The query gives you what's in your database. Enrichment gives you context from external sources. The report combines both.

## Anti-Patterns to Avoid

### Anti-Pattern 1: The Giant One-Liner

```bash
# Don't do this:
exec("psql -d claims -c 'SELECT ...' | python analyze.py | pandoc -o report.html && notify")
```

When this fails, you don't know where. You can't inspect intermediate state. You can't restart from the middle. Break it into steps.

---

### Anti-Pattern 2: No Intermediate Files

```
Turn 1: exec(query + analyze + format — all piped)
Turn 2: message("Done")
```

If any part of Turn 1 fails silently (produces wrong output with exit code 0), you won't know. Intermediate files give you checkpoints to verify and restart from.

---

### Anti-Pattern 3: Ignoring Exit Codes

The agent should verify exit code 0 at each step before proceeding. Add this to AGENTS.md:
```
In report workflows, verify each exec step returns exit code 0 before
proceeding to the next step. If a step fails, stop and message the user
with what failed and what the error was.
```

---

### Anti-Pattern 4: Monolithic Scripts

```python
# bad: one script that does everything
def main():
    data = query_database()
    cleaned = clean_data(data)
    enriched = enrich_with_npi(cleaned)
    report = render_report(enriched)
    save_html(report)
    send_notification()
```

If `enrich_with_npi` fails at item 847 of 2000, you lose all prior work and can't restart. Split into separate scripts that each do one thing and write their output to a file.

---

### Anti-Pattern 5: Putting Everything in One Turn

```
Turn 1: [write 5 scripts, run all of them, generate final report]
```

The agent should work incrementally. Each major step deserves its own turn so the agent can verify the output before proceeding. Doing too much in one turn makes errors harder to locate and recover from.

---

## Composability Principle

> **Each script should do ONE thing. Compose via files.**

A script that takes one file as input and produces one file as output is maximally composable. You can test it in isolation, replace it, or reuse it in a different pipeline. Scripts that do multiple things are harder to debug, harder to test, and harder to reuse.

```
query_claims.sh → claims.csv       (one thing: query)
clean_claims.py → clean.csv        (one thing: clean)
enrich_npi.py   → enriched.csv     (one thing: enrich)
calc_stats.py   → stats.json       (one thing: summarize)
render.py       → report.md        (one thing: format)
pandoc          → report.html      (one thing: convert)
```

Six small scripts are easier to maintain than one big script. They're also easier for the agent to compose in different ways for different reports.

---

## Quick Reference

| Pattern | Use case | Key files |
|---------|----------|-----------|
| Read → Process → Write | Data transformation | input, script, output |
| Fetch → Extract → Format | APIs, web data | raw response, parsed, formatted |
| Multi-step with checkpoints | Long pipelines, resumable | checkpoints/01, 02, 03... |
| Template + Data → Document | Recurring reports | template.md, data.json, rendered.html |
| Query → Enrich → Report | DB + external data | raw.csv, enriched.csv, report.html |
