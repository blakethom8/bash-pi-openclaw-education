# 4.4 Skills System

**How OpenClaw's skills extend the agent without adding tools**

---

## What Skills Are

A skill is a directory containing a markdown documentation file and a `scripts/` folder with bash or Python scripts. That's it. No compilation, no registration, no API — just files.

The agent discovers skills by reading their one-line description in the system prompt. When it needs to use one, it reads the full SKILL.md. When it needs to run the skill's code, it calls `exec` on the scripts in `scripts/`.

This is identical to Pi's skill pattern. OpenClaw formalizes the directory structure and adds a discovery mechanism, but the concept is unchanged: **skills are documentation that teaches the agent how to use scripts you've already written.**

---

## Directory Structure

```
skills/
├── github/
│   ├── SKILL.md
│   ├── scripts/
│   │   ├── create_pr.sh
│   │   ├── list_issues.sh
│   │   └── comment_issue.sh
│   └── examples/
│       └── create-pr-example.md
│
├── openai-whisper-api/
│   ├── SKILL.md
│   └── scripts/
│       └── transcribe.sh
│
├── weather/
│   ├── SKILL.md
│   └── scripts/
│       └── get_weather.sh
│
└── your-custom-skill/
    ├── SKILL.md
    └── scripts/
        └── your_script.py
```

**Global skills:** `/usr/local/lib/node_modules/openclaw/skills/` — built-in skills, updated with OpenClaw

**Project skills:** `~/.openclaw/workspace/skills/` (or wherever your workspace is) — your custom skills

Project skills take precedence. You can override a built-in skill by creating one with the same name in your workspace.

## How the Agent Discovers Skills

OpenClaw builds a skills index when it starts. Each skill gets a one-line description injected into the system prompt. Something like:

```
Available skills:
  - github: Create PRs, manage issues, and interact with GitHub repositories
  - openai-whisper-api: Transcribe audio files using OpenAI's Whisper API
  - weather: Get current weather and forecasts for any location
  - claims-query: Query the claims database with standard report patterns
  - npi-lookup: Look up provider information from the NPI registry
  - powerpoint-generator: Create formatted PowerPoint presentations from data
```

That one-line description is all the agent sees at session start. It uses this to decide **whether** to load a skill. When it decides to use `claims-query`, it then calls `read(skills/claims-query/SKILL.md)` to get the full documentation before proceeding.

> **Write your one-line description carefully.** If the description doesn't match how you'll ask for things, the agent won't know the skill exists for that use case. "Query the claims database" is better than "Claims tool".

## Anatomy of a SKILL.md

A well-written SKILL.md answers three questions:
1. What does this skill do?
2. What are the prerequisites / configuration needed?
3. How do I use each script?

```markdown
# Claims Query Skill

Query the claims database for reports and analysis.

## Prerequisites

- PostgreSQL client (`psql`) must be installed
- Set environment variable `CLAIMS_DB_URL=postgresql://user:pass@host/claims`
  (or configure DB_HOST, DB_USER, DB_PASS, DB_NAME separately)
- See AGENTS.md for connection details

## Scripts

### query_claims.sh

Queries claims data for a given month and outputs CSV.

Usage:
  bash skills/claims-query/scripts/query_claims.sh --month YYYY-MM [--status denied|approved|all]

Output: CSV to stdout with columns: claim_id, service_date, provider_npi, amount, status, denial_code

Example:
  bash skills/claims-query/scripts/query_claims.sh --month 2026-02 > tmp/feb-claims.csv
  bash skills/claims-query/scripts/query_claims.sh --month 2026-02 --status denied > tmp/feb-denied.csv

### analyze_claims.py

Takes a claims CSV and produces summary statistics as JSON.

Usage:
  python3 skills/claims-query/scripts/analyze_claims.py input.csv output.json

Output JSON structure:
  {
    "total_claims": 14832,
    "total_amount": 2847293.50,
    "denial_rate": 0.22,
    "top_denial_codes": [{"code": "CO-97", "count": 843, "amount": 412000}],
    "by_provider": [...]
  }
```

## Writing a Custom Skill: Step by Step

### Step 1: Create the directory

```bash
mkdir -p ~/.openclaw/workspace/skills/powerpoint-generator/scripts
mkdir -p ~/.openclaw/workspace/skills/powerpoint-generator/examples
```

---

### Step 2: Write the script

```python
#!/usr/bin/env python3
# skills/powerpoint-generator/scripts/make_pptx.py
"""
Generate a PowerPoint presentation from a JSON data file and a template spec.

Usage:
  python3 make_pptx.py data.json spec.json output.pptx

data.json format:
  {
    "title": "Q1 Claims Report",
    "slides": [
      {"type": "title", "text": "February 2026"},
      {"type": "stats", "stats": [{"label": "Total Claims", "value": "14,832"}]},
      {"type": "chart", "chart_type": "bar", "data": {...}},
      {"type": "text", "heading": "Key Findings", "bullets": ["..."]}
    ]
  }
"""

import json
import sys
from pptx import Presentation
from pptx.util import Inches, Pt

def main():
    data_file, spec_file, output_file = sys.argv[1], sys.argv[2], sys.argv[3]

    with open(data_file) as f:
        data = json.load(f)

    prs = Presentation()

    for slide_data in data["slides"]:
        if slide_data["type"] == "title":
            slide = prs.slides.add_slide(prs.slide_layouts[0])
            slide.shapes.title.text = slide_data["text"]
        # ... etc

    prs.save(output_file)
    print(f"Saved: {output_file}")

if __name__ == "__main__":
    main()
```

---

### Step 3: Write SKILL.md

```markdown
# PowerPoint Generator

Create formatted PowerPoint presentations from JSON data. Uses python-pptx.

## Prerequisites

  pip install python-pptx

## Scripts

### make_pptx.py

Generates a .pptx file from structured JSON input.

Usage:
  python3 skills/powerpoint-generator/scripts/make_pptx.py data.json spec.json output.pptx

Input JSON structure (data.json):
  {
    "title": "Report Title",
    "slides": [
      {"type": "title", "text": "Slide title"},
      {"type": "stats", "stats": [{"label": "Metric", "value": "123"}]},
      {"type": "bullets", "heading": "Key Points", "bullets": ["Point 1", "Point 2"]},
      {"type": "chart", "chart_type": "bar", "title": "Chart title", "data": {...}}
    ]
  }

Example:
  python3 skills/powerpoint-generator/scripts/make_pptx.py \
    tmp/report-data.json \
    skills/powerpoint-generator/templates/standard.json \
    reports/q1-presentation.pptx
```

---

### Step 4: Test it

Have the agent use it:
> "Create a PowerPoint with 3 slides: title slide 'Q1 Report', a stats slide with total claims, and a bullet slide with key findings"

The agent will:
1. See "powerpoint-generator" in its skills list
2. Read the SKILL.md
3. Write a `tmp/pptx-data.json` with the slide structure
4. Run `make_pptx.py` via exec
5. Message you with the output path

## Enterprise Skill Examples

### `claims-query/`

Encapsulates everything about querying your claims database:
- Connection details (or reference to AGENTS.md for credentials)
- Schema documentation (what tables exist, what columns mean)
- Standard query scripts for common report types
- Analysis scripts for summarizing claim data

**Why this matters:** The agent doesn't have to figure out your schema every session. SKILL.md explains it once.

---

### `npi-lookup/`

Encapsulates NPI registry lookups:

```markdown
# NPI Lookup

Look up provider information from the CMS NPI Registry.

## Scripts

### lookup_npi.sh

Look up a single NPI number.

Usage:
  bash skills/npi-lookup/scripts/lookup_npi.sh NPI_NUMBER

Output: JSON with provider name, address, taxonomy, phone

Example:
  bash skills/npi-lookup/scripts/lookup_npi.sh 1234567890

### batch_lookup.py

Look up multiple NPI numbers from a file, with rate limiting.

Usage:
  python3 skills/npi-lookup/scripts/batch_lookup.py npi_list.txt output.json

Input: one NPI per line
Output: JSON array of provider records, with failed lookups noted
Rate limit: 5 requests/second (API limit)
```

```bash
#!/bin/bash
# skills/npi-lookup/scripts/lookup_npi.sh
NPI="$1"
curl -s "https://npiregistry.cms.hhs.gov/api/?number=${NPI}&version=2.1" \
  | python3 -m json.tool
```

## Skills vs. Extensions

OpenClaw supports two ways to add capability:

| | Skills | Extensions |
|--|--------|------------|
| **What they are** | Markdown docs + bash/Python scripts | TypeScript code |
| **Who writes them** | You, in any language | TypeScript developers |
| **Adds new tools?** | No — uses existing exec tool | Yes — registers new tool functions |
| **Agent interface** | Agent reads SKILL.md, runs scripts via exec | Agent calls tool by name |
| **Complexity** | Low | High |
| **When to use** | Almost always | When you need native async, streaming, or tight integration |

**Use skills for enterprise workflows.** They're easier to write, easier to debug (it's just bash/Python), and easier to update. Extensions are for when you're building OpenClaw itself or need capabilities that truly can't be expressed as shell scripts.

## Composing Skills

The agent can use multiple skills in a single task. The system prompt lists all available skills; the agent loads whichever SKILL.md files it needs.

**Example:** "Generate a PowerPoint from this month's NPI lookup results"

```
Turn 1: read(AGENTS.md)
Turn 2: read(skills/npi-lookup/SKILL.md)
Turn 3: read(skills/powerpoint-generator/SKILL.md)
Turn 4: exec(batch_lookup.py npi_list.txt tmp/npi_data.json)
Turn 5: write(tmp/pptx_data.json)          ← structures NPI data for PowerPoint
Turn 6: exec(make_pptx.py tmp/pptx_data.json ... reports/npi-summary.pptx)
Turn 7: message("Done! reports/npi-summary.pptx")
```

Two skills, composed naturally. The agent is the orchestrator.

## Writing Good Skill Descriptions

The one-liner that appears in the system prompt is the most important documentation you write. The agent uses it to decide whether a skill is relevant to the current task.

**Too vague:**
```
claims: Claims processing tool
```

**Too specific:**
```
claims: Query claims.claims_fact table in PostgreSQL for monthly denial reports
```

**Just right:**
```
claims-query: Query the claims database for monthly reports, denial analysis, and provider summaries
```

Cover the main use cases in plain language. Match the vocabulary your users will actually use ("monthly report", "denial analysis", not "ETL pipeline" or "analytics").
