# 4.6 Case Study: Real Multi-Turn Sessions

**Tracing two complete OpenClaw workflows turn by turn**

---

## How to Read These Case Studies

For each turn you'll see:
- **User said / Agent state** — what triggered this turn
- **LLM thought** — the reasoning behind the tool choice
- **Tool called** — which tool, with what arguments
- **Result** — what came back
- **Context now contains** — what's accumulated so far

These are real workflow patterns. The podcast one is based on an actual OpenClaw session. The claims report one is a representative enterprise workflow.

## Case Study 1: Podcast Summarization (8 turns)

### The request:
> "Summarize the Syntax.fm podcast about Pi — the episode with Peter Steinberger and Armin Ronacher"

---

### Turn 1: Find the episode

**LLM thought:** "I don't have the URL. I need to find it. Web search."

**Tool called:**
```json
{
  "tool": "exec",
  "command": "web_search 'syntax.fm Armin Ronacher Peter Steinberger Pi agent 2025 2026'"
}
```

**Result:**
```
Syntax Episode 976: "Building an AI Agent That Can Actually Use Your Computer"
Guests: Armin Ronacher, Peter Steinberger
URL: https://syntax.fm/show/976/
YouTube: https://www.youtube.com/watch?v=abc123xyz
Published: 2026-01-15
```

**Context now contains:** Original request + search results with episode URL and YouTube link.

---

### Turn 2: Get the YouTube captions

**LLM thought:** "I have the YouTube URL. Use yt-dlp to extract auto-generated captions. OpenClaw has the whisper skill, but yt-dlp captions are faster if available."

**Tool called:**
```json
{
  "tool": "exec",
  "command": "yt-dlp --write-auto-subs --skip-download --sub-lang en -o 'podcasts/syntax-976' 'https://www.youtube.com/watch?v=abc123xyz'"
}
```

**Result:**
```
[youtube] Extracting URL: https://www.youtube.com/watch?v=abc123xyz
[download] Writing video subtitles to: podcasts/syntax-976.en.vtt
Exit code: 0
```

**Context now contains:** Prior + confirmation that `podcasts/syntax-976.en.vtt` exists.

---

### Turn 3: Clean the VTT file

**LLM thought:** "VTT subtitle files have timestamps, duplicate lines, and formatting markers. I need a script to strip those out and produce readable prose."

**Tool called (write):**
```json
{
  "tool": "write",
  "path": "podcasts/clean_vtt.py",
  "content": "#!/usr/bin/env python3\n\"\"\"Clean a VTT subtitle file into plain prose.\"\"\"\nimport re, sys\n\nwith open(sys.argv[1]) as f:\n    content = f.read()\n\n# Remove WEBVTT header\ncontent = re.sub(r'WEBVTT.*?\\n\\n', '', content, flags=re.DOTALL)\n# Remove timestamp lines\ncontent = re.sub(r'\\d{2}:\\d{2}:\\d{2}\\.\\d{3} --> .*\\n', '', content)\n# Remove duplicate blank lines\ncontent = re.sub(r'\\n{3,}', '\\n\\n', content)\n# Remove position tags\ncontent = re.sub(r'<[^>]+>', '', content)\n\nprint(content.strip())\n"
}
```

**Tool called (exec):**
```json
{
  "tool": "exec",
  "command": "python3 podcasts/clean_vtt.py podcasts/syntax-976.en.vtt > podcasts/transcript.txt && wc -w podcasts/transcript.txt"
}
```

**Result:**
```
18432 podcasts/transcript.txt
Exit code: 0
```

**Context now contains:** Prior + confirmation that 18,432-word transcript exists at `podcasts/transcript.txt`.

---

### Turn 4: Read and process the transcript

**LLM thought:** "18k words is long but fits in context. I'll read it, then generate the summary directly."

**Tool called:**
```json
{
  "tool": "read",
  "path": "podcasts/transcript.txt"
}
```

**Result:** Full transcript text (18,432 words) added to context.

**LLM thought:** "I now have the full transcript. I can identify key themes, quotes, and structure without another tool call. I'll generate the summary in the next response."

**Context now contains:** Original request + transcript + search metadata.

---

### Turn 5: Write the markdown summary

**LLM thought:** "Generate the summary and write it to disk."

**Tool called:**
```json
{
  "tool": "write",
  "path": "podcasts/syntax-976-summary.md",
  "content": "# Syntax 976: Building an AI Agent That Can Actually Use Your Computer\n\n**Guests:** Armin Ronacher (Astral/rye), Peter Steinberger (OpenClaw)\n**Date:** January 2026 | [Listen](https://syntax.fm/show/976/) | [YouTube](https://youtube.com/watch?v=abc123xyz)\n\n---\n\n## The Core Idea\n\nPi is a while loop with four tools. That's the entire agent...\n\n## Key Quotes\n\n> \"Code is truth. I don't need another place to maintain things.\" — Mario\n\n## Technical Architecture\n\n...\n\n## Key Takeaways\n\n1. Minimal tools enable maximum adaptability\n2. Bash is the universal escape hatch\n3. Files are memory\n..."
}
```

**Result:**
```
File written: podcasts/syntax-976-summary.md (2,847 bytes)
```

---

### Turn 6: Generate HTML version

**LLM thought:** "A formatted HTML version is more shareable. Use pandoc."

**Tool called:**
```json
{
  "tool": "exec",
  "command": "pandoc podcasts/syntax-976-summary.md -o podcasts/syntax-976-summary.html --standalone --metadata title='Syntax 976 Summary'"
}
```

**Result:**
```
Exit code: 0
```

---

### Turn 7: Verify output

**LLM thought:** "Quick sanity check — confirm both files exist and are reasonable sizes."

**Tool called:**
```json
{
  "tool": "exec",
  "command": "ls -lh podcasts/syntax-976*"
}
```

**Result:**
```
-rw-r--r--  1 blake  staff  247K podcasts/syntax-976.en.vtt
-rw-r--r--  1 blake  staff  113K podcasts/transcript.txt
-rw-r--r--  1 blake  staff  2.8K podcasts/syntax-976-summary.md
-rw-r--r--  1 blake  staff  4.1K podcasts/syntax-976-summary.html
```

---

### Turn 8: Notify

**LLM thought:** "Files look correct. Send the completion message."

**Tool called:**
```json
{
  "tool": "message",
  "text": "Done! Syntax Episode 976 summarized.\n\n📄 Markdown: podcasts/syntax-976-summary.md\n🌐 HTML: podcasts/syntax-976-summary.html\n\nKey topics: Pi architecture, minimal tools philosophy, why bash is enough, file-based memory."
}
```

---

### What made this work

1. **File persistence:** The VTT file existed in Turn 2. The transcript existed in Turn 3. Each turn built on prior output.
2. **Error-free path:** No command failed. In practice the agent might have had to retry yt-dlp or adjust the VTT cleaning script.
3. **Context carried forward:** By Turn 4, the agent had the URL, the word count, and the transcript. It had everything it needed without re-querying anything.
4. **The `message` tool closed the loop:** You got a Telegram notification with the file paths. You didn't have to watch the terminal.

## Case Study 2: Monthly Claims Report (8 turns)

### The request:
> "Generate the monthly claims report for February"

### AGENTS.md contains:
```markdown
## Database
DB_HOST=db.internal
DB_NAME=claims
DB_USER=reports_readonly
Use: psql -h $DB_HOST -d $DB_NAME -U $DB_USER

## Report format
Monthly claims reports go in workspace/reports/
File naming: YYYY-MM-claims.html
Use templates/claims-report.md as the base template
After generating, message the user with total claims and top denial code.
```

---

### Turn 1: Load agent context

**LLM thought:** "Read AGENTS.md to understand the DB, credentials, and format preferences."

**Tool called:**
```json
{"tool": "read", "path": "AGENTS.md"}
```

**Result:** DB connection details, report format conventions, output directory, notification preferences.

**Context now contains:** DB host, credentials, template path, output format.

### Where things can go wrong — and how the agent recovers

**Scenario A: Database connection fails (Turn 3)**
```
exec exit_code: 1
stderr: psql: error: connection to server at "db.internal" failed
```
Agent sees the error. It checks AGENTS.md for alternative connection strings, tries `psql -h localhost`, or messages you: "Database connection to db.internal failed. Is the VPN connected?"

**Scenario B: Analyze script produces unexpected output (Turn 4)**
```
Output: Processing 14,832 claims...
        Warning: 3 claims with negative amounts (claim IDs: 88821, 88834, 89012)
        Exit code: 0
```
Agent sees the warning in stderr. Rather than ignoring it, it notes it in the notification: "Report complete. Note: 3 claims had negative amounts — see ids 88821, 88834, 89012."

**Scenario C: Render template missing a field (Turn 6)**
```
KeyError: 'avg_paid_amount'
Exit code: 1
```
Agent reads the template to see what field is expected, reads the stats JSON to see what's available, edits the template to use the correct field name, retries.

This is the while loop in action: errors are information. The agent adjusts.

---

### How to make this workflow repeatable

1. **Add it to AGENTS.md.** When you ask "generate the monthly claims report for March," the agent should follow the same 8 turns automatically. AGENTS.md makes the workflow deterministic:
   ```
   ## Monthly Claims Report
   To generate: run claims-query skill for the requested month, analyze,
   render with templates/claims-report.md, save to reports/YYYY-MM-claims.html,
   message with total claims, total amount, denial rate, and top denial code.
   ```

2. **Keep the template file updated.** When the report format needs to change, edit `templates/claims-report.md`. The agent picks it up on next run.

3. **Version the stats files.** Saving `reports/2026-02-claims-stats.json` alongside the HTML means you can regenerate the report with a different template later without re-running the database query.
