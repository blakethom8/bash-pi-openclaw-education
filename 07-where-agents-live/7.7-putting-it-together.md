## The Recommended Enterprise Agent Stack

This is a concrete reference architecture for the Hetzner + Supabase setup covered throughout this module. It is not the only valid design — it is a solid starting point that you can adapt.

```

│  Your Interface (Laptop / Phone)                             │
│                                                              │
│  • Claude Code (interactive development, building agents)    │
│  • Telegram (message OpenClaw daemon for on-demand tasks)   │
│  • Cron (scheduled tasks, no human trigger needed)          │
└─────────────────────────────┬───────────────────────────────┘
                              │ (network)
                              ▼

│  Agent Server (Hetzner VPS)                                  │

│  │ OpenClaw daemon (systemd)                             │  │
│  │                                                       │  │
│  │  User:         openclaw (dedicated, non-root)         │  │
│  │  WorkingDir:   /home/openclaw/workspace/              │  │
│  │  Logs:         journald → journalctl -u openclaw      │  │
```

## What Lives Where

One of the most common points of confusion for developers new to agents: where exactly is each piece of data?

| Thing | Lives on | Access pattern | Persists? |
|-------|----------|----------------|-----------|
| Agent loop code | Hetzner server filesystem | Runs as a process | Yes — until you update it |
| Conversation context (messages array) | Agent process RAM | In-memory, each turn | No — cleared between sessions |
| AGENTS.md and system prompt | Hetzner server filesystem | Read at session start | Yes — you manage this file |
| Agent memory files (memory/*.md) | Hetzner server filesystem | Agent reads/writes | Yes — accumulates over time |
| LLM model weights (Claude) | Anthropic's servers | API call over network | N/A — not yours |
| Your business data | Supabase | SQL over network | Yes — database |
| Report output | Hetzner server filesystem | Written by agent | Yes — until cleaned up |
| Credentials and config | .env on Hetzner server | Environment variables | Yes — you manage this file |
| Agent skills/scripts | Hetzner server filesystem | Bash execution | Yes — until you change them |
| Logs | journald (systemd) or log files | journalctl or cat | Yes — until log rotation |

The two things that are **not** on your server: the LLM model itself (at Anthropic), and your Supabase data (at Supabase). Everything else is local to your Hetzner machine.

## Decision Guide: Where to Run Your Next Agent

Use this flow when you're deciding where a new agent should live.

```
Is this a one-time task or something you're building/testing?
  ├── YES → Run on your laptop. Use Pi-mono or Claude Code.
  └── NO  → Continue ↓

Does it need to run on a schedule (nightly, weekly, monthly)?
  ├── YES → Cron job on Hetzner. One-shot agent script.
  └── NO  → Continue ↓

Does it need to respond to messages or external events?
  ├── YES → Daemon on Hetzner. Add to OpenClaw or create a new systemd service.
  └── NO  → Continue ↓

Is it experimental, or might it modify important files?
  ├── YES → Docker container with limited filesystem mounts.
  └── NO  → Continue ↓

Does it need more RAM or CPU than your Hetzner server has?
  ├── YES → Cloud VM (Azure, Hetzner Cloud larger instance).
  └── NO  → Continue ↓

Do multiple people need to trigger it independently?
  ├── YES → Daemon with web or messaging interface (webhook, Telegram bot).
  └── NO  → Daemon or cron on Hetzner is fine.
```

Most enterprise agent decisions resolve in the first three branches. Development on laptop, scheduled tasks via cron, always-on work via daemon. The later branches apply when you've outgrown the basics.

## The Three Commands to Know

Once an agent is deployed, these three commands tell you everything you need to know.

```bash
# 1. Is my agent daemon running and healthy?
systemctl status openclaw

# Good output:
# ● openclaw.service - OpenClaw Agent Daemon
#      Active: active (running) since Mon 2026-03-15 07:00:00 UTC; 9h ago
#    Main PID: 1247 (python3)
#      Memory: 142.3M

# Bad output:
# ● openclaw.service - OpenClaw Agent Daemon
#      Active: failed (Result: exit-code)
# → Check logs immediately: journalctl -u openclaw -n 50


# 2. What is my agent doing right now?
journalctl -u openclaw -f
# -f = follow (like tail -f)
# Shows logs as they come in
# Ctrl+C to stop following


# 3. How much RAM and CPU is it actually using?
htop -p $(pgrep -f agent.py)
# or, without htop:
ps -o pid,ppid,%cpu,%mem,vsz,rss,comm -p $(pgrep -f agent.py)
```

Add a fourth for scheduled agents:

```bash
# 4. Did my cron job run and succeed?
grep "CRON" /var/log/syslog | grep "weekly_report" | tail -20
# or check the agent's own log file:
tail -50 /home/openclaw/workspace/logs/weekly_report.log
```

## Common Failure Modes and Fixes

A quick reference for the problems that actually happen in production.

**Agent daemon stopped (not running):**
```bash
journalctl -u openclaw -n 100   # find the error
sudo systemctl start openclaw    # restart it
```

**Agent running but not responding:**
```bash
journalctl -u openclaw -f        # is it stuck in a loop? waiting on something?
sudo systemctl restart openclaw  # restart if stuck
```

**Disk filling up:**
```bash
df -h                            # overall disk usage
du -sh /home/openclaw/workspace/*  # what's taking space in the workspace
# Clean old reports:
find /home/openclaw/workspace/reports -mtime +90 -delete
```

**Agent can't reach Supabase:**
```bash
# Test the connection manually as the agent user
sudo -u openclaw psql $SUPABASE_URL -c "SELECT 1"
# If that fails: check .env has correct SUPABASE_URL, check network
```

**Agent running with wrong working directory:**
```bash
# Find the process and its cwd
ls -la /proc/$(pgrep -f agent.py)/cwd
# Fix: update WorkingDirectory= in the systemd service file
sudo systemctl daemon-reload
sudo systemctl restart openclaw
```

**Cron job not running:**
```bash
# Check crontab is correctly configured
crontab -l
# Check syslog for cron execution
grep CRON /var/log/syslog | tail -20
# Test the script manually first
bash /home/blake/agents/weekly_report.sh
```

## The Module in One Paragraph

Your agent is a process. It runs as a user, from a working directory, with a set of environment variables, on a specific machine. What it can do is determined by those three things — its user, its machine, and its env vars. If you want it to run while your laptop is off, it belongs on your server as a daemon, managed by systemd. If you want it isolated from other processes, put it in a container. If it needs to run on a schedule, cron is sufficient. The LLM model runs at Anthropic — what runs on your machine is the loop, the context, and the tool execution. Credentials live in `.env`, never in code. Sandboxes are not about distrust — they are how you make mistakes recoverable. The correct mental model for all of it: your agent is a smart intern. Set up their workspace correctly, give them the right access badge, put them in an appropriate office, and they'll do their job reliably.
