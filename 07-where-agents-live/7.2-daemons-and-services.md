# 7.2 Daemons: Agents That Never Sleep

**Background processes, server boot, and why your enterprise agent should outlive your terminal session**

---

## What is a Daemon?

You have probably heard the word "daemon" in a Unix context without getting a clear definition. Here it is:

> **A daemon is a process that runs in the background, continuously, without being attached to a terminal.**

That's it. No mythology required (though the name does come from Greek — the "Disk And Execution MONitor" backronym is a later invention and not the real origin).

You already use daemons constantly. Every time you connect to your Hetzner server over SSH, a daemon called `sshd` accepted your connection. PostgreSQL runs as a daemon. `nginx` is a daemon. The process that fires your cron jobs (`crond`) is a daemon. These processes started when the server booted, and they'll keep running until you stop them or the server shuts down.

**How to spot a daemon:**

```bash
ps aux

# Output (simplified):
USER       PID   TTY    COMMAND
root       312   ?      sshd
postgres   401   ?      postgres
www-data   512   ?      nginx: master process
blake     8823   pts/0  bash          ← this is YOUR terminal session
blake     9001   pts/0  python agent.py ← this agent is attached to your terminal
```

The `TTY` column tells you if a process is attached to a terminal. A `?` means it isn't — it's a daemon. `pts/0` means it's attached to your terminal session.

When you close your terminal, every process with your TTY in that column dies. Every process with `?` keeps running.

---

## Interactive Agent vs. Daemon Agent

This distinction determines whether your agent survives the moment you close your laptop.

**Interactive agent (Pi-mono, Claude Code):**
- You open a terminal, run `python agent.py`, have a conversation
- The process is attached to your terminal (TTY shows your session)
- You close the terminal: the agent dies
- Process lifecycle = your terminal session

**Daemon agent (OpenClaw, a webhook listener):**
- Starts when the server boots
- Runs indefinitely in the background
- Listens for incoming triggers: Telegram messages, webhook calls, cron schedules
- When a trigger arrives: handles it, responds, goes back to waiting
- Process lifecycle = server uptime

For your reporting agents and data pipeline agents, you almost certainly want a daemon. You should not have to SSH into your server and manually run a script every time you want a report. The daemon is running. You send it a message. It responds.

## How OpenClaw Works as a Daemon

OpenClaw's architecture is designed around the daemon pattern:

```
Server boot
    │
    ▼
OpenClaw process starts (systemd)
    │
    ▼
Connects to Telegram API (long-polling)
    │
    ▼
┌─────────────────────────────────────┐
│  Waiting for messages...            │◄──── stays here, always listening
└──────────────┬──────────────────────┘
               │ (message arrives)
               ▼
    Start/resume agent session
               │
               ▼
    Agent loop runs (turns 1..N)
               │
               ▼
    Send response via Telegram
               │
               ▼
┌─────────────────────────────────────┐
│  Waiting for messages...            │◄──── returns to waiting
└─────────────────────────────────────┘
```

OpenClaw is on your Hetzner server right now, if it's set up. Nobody is running it. It started on boot. When you message it on Telegram, it handles the message and goes back to waiting. If the server reboots at 3am, OpenClaw comes back up automatically.

This is what "always on" actually means in infrastructure terms. Not a laptop you leave running. A proper daemon managed by the OS.

## systemd: The Modern Daemon Manager

On Linux servers (including your Hetzner instances), `systemd` is the standard way to manage daemons. It handles starting on boot, restarting on crash, logging, and resource limits.

A systemd service file for an agent looks like this:

```ini
# /etc/systemd/system/reporting-agent.service

[Unit]
Description=Reporting Agent Daemon
After=network.target
After=postgresql.service

[Service]
Type=simple
User=reportingagent
WorkingDirectory=/home/reportingagent/workspace
ExecStart=/home/reportingagent/workspace/agent.py
Restart=always
RestartSec=5
EnvironmentFile=/home/reportingagent/.env

# Resource limits (covered in detail in 7.5)
MemoryMax=2G
CPUQuota=150%

[Install]
WantedBy=multi-user.target
```

Key fields to understand:

- `User=reportingagent` — the daemon runs as a dedicated user, not as you (see 7.5)
- `WorkingDirectory=` — the agent always starts from the right directory
- `EnvironmentFile=` — loads the `.env` file so env vars are available
- `Restart=always` — if the agent crashes, systemd restarts it after 5 seconds
- `After=network.target` — waits for the network to be up before starting

**The four commands you'll use:**

```bash
# Install and start
sudo systemctl daemon-reload           # tell systemd about the new service file
sudo systemctl enable reporting-agent  # start on boot
sudo systemctl start reporting-agent   # start now

# Day-to-day operations
systemctl status reporting-agent       # is it running?
journalctl -u reporting-agent -f       # follow logs in real time
journalctl -u reporting-agent --since "1 hour ago"  # recent logs
sudo systemctl restart reporting-agent # restart after config change
sudo systemctl stop reporting-agent    # stop it
```

**What good status output looks like:**

```
● reporting-agent.service - Reporting Agent Daemon
     Loaded: loaded (/etc/systemd/system/reporting-agent.service; enabled)
     Active: active (running) since Mon 2026-03-15 07:00:00 UTC; 8h ago
   Main PID: 1247 (python3)
     Memory: 142.3M
      Tasks: 3

Mar 15 07:00:00 hetzner-server reporting-agent[1247]: Starting agent...
Mar 15 07:00:01 hetzner-server reporting-agent[1247]: Connected to Supabase
Mar 15 07:00:01 hetzner-server reporting-agent[1247]: Listening for triggers
```

If you see `active (running)`, your daemon is healthy. If you see `failed`, check `journalctl -u reporting-agent -n 50` to see the last 50 log lines.

## Why Daemons Matter for Enterprise Agents

The difference between a daemon agent and a script-you-run-manually is the difference between infrastructure and a personal tool.

**Without a daemon:**
- Someone has to SSH into the server and run the script
- If nobody does it, the report doesn't happen
- If the person who knows the command is on vacation, it doesn't happen
- If the script fails silently, nobody notices

**With a daemon:**
- The agent is always running, always ready
- Triggers (cron, webhook, Telegram message) fire automatically
- Crashes are recovered automatically via `Restart=always`
- Logs accumulate in `journald` — you can investigate after the fact
- Monitoring scripts can check `systemctl is-active reporting-agent` and alert you

For reporting agents, data pipeline agents, and corporate automation — this is the target state. The agent as infrastructure, not as a personal script.

## The Three Agent Lifecycle Patterns

Every agent you build fits one of these three patterns. Knowing which one you're building determines your deployment approach.

**Pattern 1: One-Shot**

Run, complete a single task, exit.

```bash
# Runs, generates report, exits
python generate_monthly_report.py --month 2026-03

# Triggered by cron:
0 8 1 * *  /home/blake/agents/generate_monthly_report.py --month $(date +%Y-%m)
```

The process lives for seconds to minutes. No daemon needed — cron is the trigger.

Good for: scheduled reports, nightly data syncs, periodic cleanup tasks.

---

**Pattern 2: Interactive Session**

Run, engage in a back-and-forth conversation, exit when done.

```bash
# Pi-mono, Claude Code: you run it, have a session, close it
python agent.py
# ... conversation happens ...
# You type "exit" or Ctrl+C
# Process ends
```

The process lives for the duration of your session. You are the trigger. No daemon needed.

Good for: development, exploration, ad-hoc analysis, building things.

---

**Pattern 3: Persistent Daemon**

Start on boot, run forever, respond to triggers, repeat.

```bash
# OpenClaw, a webhook listener, a Telegram bot
# Started by systemd, never manually
# Listens for messages/webhooks/events
# Processes them and returns to waiting
```

The process lives as long as the server does. Triggers are external: messages, webhooks, scheduled events.

Good for: always-on assistants, webhook handlers, scheduled automation where the schedule may change or the agent needs to be triggered on demand.

---

**Which pattern should your enterprise agent use?**

| Need | Pattern |
|------|---------|
| Weekly report, nightly data sync | One-shot (cron) |
| Building and testing agents | Interactive |
| Always-on Telegram assistant | Daemon |
| React to new data arriving | Daemon (webhook listener) |
| Ad-hoc "run this analysis" | Interactive or one-shot |

Most enterprise automation ends up as a mix: a daemon that orchestrates one-shot agents.
