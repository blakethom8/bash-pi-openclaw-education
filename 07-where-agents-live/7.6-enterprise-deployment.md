# 7.6 Where to Actually Run Your Enterprise Agents

**A deployment decision matrix, your stack mapped to real decisions, the production checklist, and the intern mental model**

---

## The Deployment Decision Matrix

Every enterprise agent deployment is a combination of: what kind of task, how it gets triggered, what it needs access to, and what happens if it fails. This table covers the common patterns.

| Use case | Where to run | Trigger | Why |
|----------|-------------|---------|-----|
| Ad-hoc report, one-time analysis | Your laptop | Manual | Simple, interactive, full control while developing |
| Scheduled nightly/weekly reports | Hetzner server (cron + one-shot agent) | Cron | Always on, no laptop needed, reliable schedule |
| React to incoming data event | Hetzner server (webhook daemon) | Webhook from data source | Respond immediately when new data arrives |
| Interactive assistant | Hetzner server (OpenClaw daemon) | Telegram / web | Always ready, no startup lag |
| Isolated experimental agent | Docker container on Hetzner | Manual or cron | Mistakes don't affect the server |
| Heavy computation (large ML model, big data) | Azure VM or cloud instance | Manual or API | More RAM/CPU than your server; Windows tools |
| Multi-user agent (whole team triggers it) | Server daemon with web/messaging interface | Team members | Shared infrastructure, single source of truth |
| Data pipeline triggered by upstream system | Queue consumer daemon on Hetzner | Queue message | Decoupled from upstream, handles backpressure |

The short version: **if it needs to run while your laptop is closed, it belongs on your server.**

│   └── 30 17 * * 5 → end_of_week_summary.sh
│
└── Optional: Docker containers for isolated workloads
    └── docker-compose.yml with resource limits

Supabase (cloud-hosted PostgreSQL)
└── Your business data: agents read/write via connection string
    ├── Claims database
    ├── Provider records
    └── Agent session/output records (if you use Supabase for this)

Azure (if/when needed)
├── Windows VMs: Excel/PowerPoint COM automation

The thing to notice: Anthropic's API is not in this diagram because you don't manage it. It's a network call. Your agents call it the same way they call any external API.

## Making Agents Production-Ready: The Checklist

The difference between a development agent and a production agent is mostly the surrounding infrastructure. The checklist below captures the decisions that matter.

```
INFRASTRUCTURE
─────────────────────────────────────────────────────
☐ Agent runs as a dedicated user (not root, not your personal account)
☐ Workspace directory is isolated from the rest of the server
☐ Credentials are in environment variables, not hardcoded in scripts
☐ Resource limits are set: memory cap, CPU quota, execution timeout
☐ Daemon agents use systemd with Restart=on-failure or Restart=always
☐ Working directory is explicit in the service definition

OBSERVABILITY
─────────────────────────────────────────────────────
☐ Agent writes structured logs (timestamps, session IDs, error codes)
☐ Log rotation is configured (logs don't grow forever)
☐ You can check "is the daemon currently running?" in one command
☐ You can read recent logs without SSH-ing and hunting around
☐ Alerts: notify you if the agent hasn't run in the expected window

SAFETY
─────────────────────────────────────────────────────
☐ Agent runs against staging/dev database for initial deployment
☐ Output directory is separate from source data (agent can't overwrite inputs)
☐ Important files are git committed or backed up before agent runs on them
☐ AGENTS.md explicitly prohibits reading credential files
☐ Dry-run or plan mode available for new agent deployments

SECRETS MANAGEMENT
─────────────────────────────────────────────────────
☐ .env file exists and is not committed to git
☐ .gitignore includes .env
☐ All credentials reach the agent via environment variables
☐ For multi-person teams: credentials are in a shared secrets manager
```

You don't need to check every box before deploying an agent to production. But any unchecked box is a known risk. Be deliberate about which ones you're accepting.

## The .env Pattern: Your Most Important Habit

If you take one operational practice from this module, make it this one: every agent has a `.env` file with all its credentials and configuration, that file is never committed to git, and the agent reads everything via environment variables.

```bash
# .env — NEVER commit this to git
# Place this in the agent's working directory or home directory

SUPABASE_URL=postgresql://user:pass@db.supabase.co:5432/postgres
SUPABASE_ANON_KEY=eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...
SUPABASE_SERVICE_ROLE_KEY=eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...
ANTHROPIC_API_KEY=sk-ant-api03-...
HETZNER_API_KEY=abc123xyz...

OUTPUT_DIR=/home/openclaw/workspace/reports
WORKSPACE_DIR=/home/openclaw/workspace
LOG_LEVEL=INFO
```

Loading it:

```bash
# In a bash script
source .env
python agent.py

# In systemd service
EnvironmentFile=/home/openclaw/.env
```

```python
# In Python
from dotenv import load_dotenv
import os

load_dotenv()  # reads .env from current directory

supabase_url = os.getenv("SUPABASE_URL")
if not supabase_url:
    raise EnvironmentError("SUPABASE_URL not set — check .env file")
```

**The .gitignore entry is mandatory:**

```bash
# .gitignore
.env
.env.*
*.env
```

For teams working on shared infrastructure, a secrets manager replaces the `.env` file: Azure Key Vault, HashiCorp Vault, or AWS Secrets Manager. Agents fetch their credentials at startup rather than reading a local file. This matters when you have multiple team members or multiple deployment environments (dev, staging, prod) with different credentials for each.

## Scaling Up: When One Server Isn't Enough

Your Hetzner server handles most enterprise agent workloads. When it doesn't:

**Reports taking too long?**

First ask whether the bottleneck is CPU, RAM, or something else:

```bash
# During a slow agent run:
htop                    # CPU and RAM usage
iotop                   # disk I/O
nethogs                 # network bandwidth
```

If CPU or RAM is the limit: upgrade the Hetzner server type, or offload the heavy computation to a separate cloud instance for that task.

**Multiple agents competing for resources?**

Docker containers with resource limits let multiple agents share a server without stealing each other's RAM and CPU:

```yaml
# docker-compose.yml
services:
  reporting-agent:
    mem_limit: 2g
    cpus: 1.0
  data-pipeline-agent:
    mem_limit: 4g
    cpus: 2.0
  openclaw:
    mem_limit: 1g
    cpus: 0.5
```

**Need Windows tools?**

PowerPoint COM automation, Excel macros, and other Windows-specific tooling requires a Windows environment. Azure Windows VMs are the practical answer: you spin one up, install your tools, run the agent, shut it down. You pay only for the time it's running.

**Need GPU for ML workloads?**

Azure NC series VMs (or equivalent on other clouds) provide GPU instances. These are not cost-effective for general-purpose agent work but make sense for agents that run inference locally or process large models.

**Processing terabytes of data?**

Cloud object storage (Azure Blob, S3) combined with cloud compute (Azure batch jobs, AWS EMR) handles scale that no single server can. Your agent becomes an orchestrator that launches cloud jobs and collects results rather than doing the processing itself.

## The Intern Mental Model

When you're designing an enterprise agent deployment, the clearest mental model is this:

> **Think of your agent like a smart intern.**

This analogy holds surprisingly well across every deployment decision:

**Your laptop = the intern sitting at your desk.**

You're watching them work. They have access to your screen, your files, your login. You can see every command they run. If something goes wrong, you intervene immediately. This is development mode. High trust, high supervision, full capability. The cost: they stop working when you stop watching.

**Your Hetzner server = the intern with their own office.**

They work independently. They have their own desk, their own login, their own set of files. They can work while you're asleep or on vacation. They check in via Telegram or email when they have a result. They keep going until the job is done. The cost: you need to set up their office correctly (workspace, credentials, access to the right systems) before they start.

**A Docker container = the intern in a sandboxed workspace.**

They have everything they need for the task at hand and nothing else. They can't accidentally walk into another team's files. Their mistakes stay in their workspace. When the task is done, the workspace is cleaned up. The cost: a bit more setup to provision the workspace correctly each time.

**A daemon = the intern on 24/7 on-call rotation.**

They're always ready to respond. You don't need to tell them to start their shift — they're always on. If they fall asleep (crash), they wake themselves back up (systemd restart). The cost: you need to think about what happens when they crash, how you check on them, and how you update their instructions without a complete restart.

**The .env file = the intern's access badge.**

It holds the credentials they need to do their job. They carry it with them. You issue it, you can revoke it, you can change what it gives access to without changing anything about the intern themselves. The cost: if they lose it (the .env leaks to git), you need to rotate all the credentials immediately.

The intern framing helps because it separates the question of "what can the agent do" from "what environment have I given it to work in." A capable intern given a broken setup produces bad results. A capable agent given a clear workspace, the right credentials, appropriate resource limits, and explicit boundaries produces reliable results.
