# 7.3 VMs, Containers, and Serverless

**First-principles mental models for compute environments — and when each one makes sense for agents**

---

## The Problem This Solves

You could run all your agents on your laptop. For development and exploration, that's fine.

But eventually you hit real problems:

- Your laptop needs to sleep
- Two agents competing for the same Python environment cause dependency conflicts
- An agent fills up your disk, which is also your personal disk
- You want a teammate to trigger the agent without SSH-ing into your machine
- The workload needs more RAM than your laptop has

The question "where should my agent live?" is really asking: **what kind of isolated compute environment do I need?**

There are four answers, in increasing order of isolation and operational complexity.

---

## Level 1: Bare Metal

**What it is:** The agent runs directly on a physical machine — your laptop, your desktop, or a dedicated physical server.

No virtualization. The process touches real hardware directly.

**What you get:**
- Full access to all RAM and CPU
- Full access to all storage
- Maximum performance (no virtualization overhead)
- No isolation from other processes on the same machine

**Your Hetzner servers are effectively bare metal.** When you rent a Hetzner dedicated server (not a VPS), you get exclusive use of real physical hardware. When you rent a Hetzner VPS, you're on virtualized hardware, but from your perspective it behaves like a dedicated machine.

**When it's the right choice:**
- Development on your laptop
- Single-tenant servers where you control everything
- When you need maximum performance and don't need process isolation

**The risk:** One misconfigured agent can affect everything else on the machine. An agent that fills up the disk fills up your disk. An agent that consumes all RAM starves your other processes.

---

## Level 2: Virtual Machines

**What it is:** Software that simulates an entire computer.

A VM has its own complete operating system, its own RAM allocation, its own virtual disk, its own virtual network interfaces. From inside the VM, it looks and behaves like a real computer. But it's running as a process on a physical host machine, managed by a **hypervisor**.

```
Physical Server (your Hetzner server, or a datacenter machine)
├── Hypervisor (KVM, VMware, Hyper-V)
│   ├── VM 1: Ubuntu 22.04, 4GB RAM → runs your reporting agent
│   ├── VM 2: Ubuntu 22.04, 2GB RAM → runs your OpenClaw daemon
│   └── VM 3: Windows Server → runs legacy Excel automation tool
```

**The Azure mental model:** When you provision an Azure VM, you are renting a slice of Microsoft's physical hardware. Your VM is a process on their machine. From your perspective, you have a computer with the specs you chose — but that computer only exists as software.

**What you get:**
- Strong isolation: VM 1 cannot see VM 2's memory or processes
- Your own OS: you can run any OS version, any kernel
- Reproducible environments: snapshot the VM, restore it if something breaks
- Independent failure: one VM crashing doesn't crash others

**The cost:** Each VM uses its full RAM allocation even when idle. A VM allocated 4GB uses 4GB whether the agent is running or sleeping.

**When it's the right choice:**
- You need strong isolation between different workloads
- You need to run different operating systems (Windows agent + Linux agent on same server)
- Long-running services that need to be independent of each other
- When you want to snapshot and restore environments

---

## Level 3: Containers (Docker)

**What it is:** A lighter-weight isolation mechanism than a VM.

Containers share the host OS kernel but each gets its own isolated filesystem, network namespace, and process space. Think of it as a VM that skipped the "separate OS kernel" part.

```
Physical Server (or VM) — Linux OS
└── Linux kernel (shared by all containers)
    ├── Container: reporting-agent
    │   ├── Python 3.11 + libraries
    │   ├── Agent code
    │   └── Isolated filesystem (can only see what you mount in)
    ├── Container: openclaw
    │   ├── Node.js runtime
    │   ├── OpenClaw code
    │   └── Isolated filesystem
    └── Container: postgres
        ├── PostgreSQL
        └── Data volume
```

**Key concepts:**

**Image** — a frozen snapshot of a filesystem. Contains the OS base, libraries, your code, everything needed to run. Built once, run anywhere with Docker installed.

**Container** — a running instance of an image. Starts in seconds. Disposable. You can delete it and recreate it in 10 seconds.

**Volume** — a directory mounted into a container from the host. Data here persists even when the container is deleted. Your agent's workspace and output directories should be volumes.

```bash
# Run a reporting agent in a container
docker run \
  --name reporting-agent \
  --memory 2g \                              # max 2GB RAM
  --cpus 1.5 \                              # max 1.5 CPU cores
  -v ~/workspace:/app/workspace \           # mount workspace
  -v ~/reports:/app/reports \              # mount output directory
  --env-file .env \                        # pass credentials
  my-reporting-agent:latest
```

```yaml
# docker-compose.yml — for running multiple containers together
version: '3.8'
services:
  reporting-agent:
    image: my-reporting-agent:latest
    volumes:
      - ./workspace:/app/workspace
      - ./reports:/app/reports
    env_file: .env
    restart: always
    mem_limit: 2g

  openclaw:
    image: openclaw:latest
    volumes:
      - ./openclaw-workspace:/workspace
    env_file: .env
    restart: always
```

## VM vs. Container: When to Use Which

| | VM | Container |
|--|-----|-----------|
| Boot time | 1–5 minutes | 1–10 seconds |
| Image size | 5–50 GB | 100 MB – 2 GB |
| OS | Full separate kernel | Shared host kernel |
| Isolation strength | Very strong | Strong |
| Memory when idle | Full allocation used | Near zero |
| Good for | Different OS needs, strong isolation, legacy software | Reproducible deployments, fast scaling, microservices |
| Snapshot/restore | Yes (VM snapshots) | Yes (image versions) |

**For agents specifically:**

Containers win for most agent workloads because:
- Fast startup (10 seconds vs. 5 minutes matters for on-demand agents)
- Low overhead when idle
- Easy to update: build new image, stop container, start new one
- Easy to replicate: same image runs the same way on your laptop, your server, and your colleague's machine

VMs win when:
- You need a Windows environment (COM automation for Excel/PowerPoint generation)
- You need stronger isolation than containers provide
- You're running on shared infrastructure with untrusted workloads

## Level 4: Serverless Functions

**What it is:** Code that runs in response to an event, with no persistent machine to manage.

You write a function, deploy it to a cloud provider, and define what triggers it (an HTTP request, a queue message, a scheduled time). The cloud provider handles allocating compute, running your function, and tearing it down when done. You're billed for milliseconds of execution, not idle time.

Examples: AWS Lambda, Azure Functions, Google Cloud Functions.

```python
# An Azure Function that runs when triggered by HTTP
def main(req: func.HttpRequest) -> func.HttpResponse:
    # This code runs when someone calls your endpoint
    # It runs for seconds to minutes, then the environment disappears
    result = generate_report(req.params.get('month'))
    return func.HttpResponse(result)
```

**Why serverless is tempting for agents:**

- No server to manage
- Automatic scaling
- Pay only for what you use
- Triggered by webhooks automatically

**Why serverless is problematic for most agents:**

Agents are stateful and long-running. They make multiple LLM API calls (each one takes 3–30 seconds), accumulate context across turns, call tools, wait for results. A typical agent session might take 2–15 minutes.

Serverless functions have time limits: AWS Lambda caps at 15 minutes. Azure Functions Consumption plan at 10 minutes. Beyond that, your function is forcibly terminated.

Worse, serverless functions are stateless by design. Between invocations, nothing persists. Your conversation history, workspace files, and in-progress context disappear. You'd need to serialize all of this to external storage between calls — at which point you've rebuilt most of what a server gives you anyway.

**When serverless DOES work for agents:**

- Short, well-defined, one-shot agent tasks (generate a summary from pre-loaded data)
- Webhook handlers that receive a trigger and immediately push a job to a queue, where a real server processes it
- Tasks with predictable completion time under 5 minutes

**The pattern that works:**

```
Webhook arrives
    │
    ▼
Serverless function (fast, <5 seconds)
    │ puts job in queue
    ▼
Queue (SQS, Azure Service Bus)
    │
    ▼
Agent daemon on your server (picks up job, runs full agent loop)
    │
    ▼
Stores result in Supabase or sends via Telegram
```

The serverless function is just a fast intake mechanism. The real work happens on your persistent server.

## The Three Deployment Patterns for Your Stack

Given your Hetzner + Supabase setup, here are the three patterns you'll actually use:

---

**Pattern A: Agent as Daemon on Hetzner Server**

```
Hetzner Server (VPS or dedicated)
├── openclaw daemon (managed by systemd)
│   ├── Runs as: dedicated user
│   ├── Workspace: /home/openclaw/workspace/
│   ├── Logs: journald (accessible via journalctl)
│   └── Credentials: /home/openclaw/.env
└── Cron jobs: scheduled one-shot agents
```

Best for: your primary always-on setup. Full control, persistent storage, no moving parts.

---

**Pattern B: Agent in a Docker Container on Hetzner**

```yaml
# docker-compose.yml on your Hetzner server
services:
  reporting-agent:
    image: reporting-agent:2026-03-15   # versioned image
    volumes:
      - /home/blake/workspace:/app/workspace
      - /home/blake/reports:/app/reports
    env_file: /home/blake/.env
    restart: always
    mem_limit: 2g
    cpus: 1.5
```

Best for: when you want reproducible deployments (same image on dev and prod), easy rollbacks (just run the previous image tag), or isolation from other things on the server.

---

**Pattern C: One-Shot Agent via Cron**

```bash
# crontab -e
0 7 * * 1    /home/blake/agents/weekly_report.sh
0 0 * * *    /home/blake/agents/nightly_sync.sh
30 17 * * 5  /home/blake/agents/end_of_week_summary.sh
```

Best for: recurring tasks with a fixed schedule where you don't need an always-on daemon. Simple, inspectable, easy to test by running the script manually.

---

## The Mental Model

Think of the four levels as a tradeoff between control and isolation:

```
More control, less isolation
↑
Bare metal (your server, direct)
VM (your own OS, strong isolation)
Container (shared kernel, fast, lightweight)
Serverless (no server, but stateless and time-limited)
↓
Less control, more managed
```

For agents, containers on a persistent server is usually the right operating point: enough isolation to prevent accidents, fast enough to be practical, persistent enough to handle long-running sessions.
