# 7.5 Giving Agents a Sandbox

**Bounded environments where mistakes are recoverable — the five levers, practical setups, and the recovery question**

---

## Why Sandboxing Matters

Agents make mistakes. Not because they're malicious — because they misinterpret instructions, take literal actions when you meant something approximate, explore more aggressively than intended, or get caught in loops.

A sandbox is a bounded environment where those mistakes are recoverable. You can delete the sandbox and start fresh. You can restore from a snapshot. The mistake is contained.

> **Sandboxing is not about distrusting AI. It is good engineering practice. Humans use sandboxes too: staging environments, dev databases, test branches, paper-first workflows. The only difference is that agents move faster and make more tool calls per minute.**

Before deploying any agent to production, ask one question: "If this goes wrong, how do I recover?"

If the answer is "I'd have to manually restore from backup and spend a day figuring out what happened," your sandbox is not good enough.

---

## The Five Levers of Sandboxing

There are five independent dimensions you can constrain. Use more of them for higher-stakes agents; fewer for development.

---

### Lever 1: Filesystem Isolation

Limit what the agent can read and write.

**Minimal (instruction-based):** Tell the agent in `AGENTS.md` to stay within its workspace. Relies on the agent following instructions. Works for well-prompted agents on routine tasks.

**Better (dedicated workspace + restricted user):** The agent runs as a user that only has write access to its own home directory. Can't touch your files even if it tries. Details under Lever 2.

**Strongest (container with limited mounts):** The container's filesystem only contains what you explicitly mount in. Nothing else is accessible.

```bash
# Container with carefully scoped filesystem access
docker run \
  -v ~/workspace:/workspace \           # read + write: agent's working area
  -v ~/data:/data:ro \                  # read-only: source data
  -v ~/reports:/reports \               # write: output goes here
  --read-only \                         # rest of container filesystem: read-only
  --tmpfs /tmp \                        # /tmp is writable in RAM only
  my-agent
```

With this setup: the agent can write to `/workspace/` and `/reports/`. It can read `/data/` but not modify it. It cannot see anything outside these three directories. It cannot persist anything to `/tmp/` across restarts.

---

### Lever 2: User Permissions

Run the agent as a restricted OS user.

```bash
# Create a dedicated user for the agent
sudo useradd -m -s /bin/bash openclaw-agent

# Set up the workspace
sudo mkdir -p /home/openclaw-agent/workspace
sudo chown -R openclaw-agent:openclaw-agent /home/openclaw-agent/

# The agent's .env file
sudo nano /home/openclaw-agent/.env
sudo chmod 600 /home/openclaw-agent/.env
sudo chown openclaw-agent:openclaw-agent /home/openclaw-agent/.env
```

Now in your systemd service:

```ini
[Service]
User=openclaw-agent
WorkingDirectory=/home/openclaw-agent/workspace
```

**What this achieves:**

The `openclaw-agent` user can only read/write files it owns — primarily its own home directory. It cannot read your personal files. It cannot read other service accounts' files. It cannot read `/etc/shadow`. If the agent is somehow tricked into running `rm -rf ~/`, it deletes its own home directory, not yours.

**The important limit:** This user still has network access by default. It can still call the Anthropic API, Supabase, and anything else on the internet. Network isolation requires Lever 3.

---

### Lever 3: Network Isolation

Control what network destinations the agent process can reach.

**Default:** The agent can reach any IP address on any port. It can call your Supabase, Anthropic, external APIs, random internet hosts — no restrictions.

**Restricted via firewall rules:**

```bash
# iptables: allow agent user to reach only specific hosts
# Block all outbound traffic for the agent user
iptables -A OUTPUT -m owner --uid-owner openclaw-agent -j DROP

# Then allow only what's needed
iptables -I OUTPUT -m owner --uid-owner openclaw-agent -d api.anthropic.com -p tcp --dport 443 -j ACCEPT
iptables -I OUTPUT -m owner --uid-owner openclaw-agent -d db.supabase.co -p tcp --dport 5432 -j ACCEPT
```

**Restricted via Docker networking:**

```bash
# No network at all (fully isolated)
docker run --network none my-agent

# Custom network with specific allowlist
docker network create --internal agent-network
docker run --network agent-network my-agent
```

**When to use network isolation:**

- Agents processing sensitive data that should never leave your network
- Experimental agents where you don't want unauthorized external calls
- Agents that only need to talk to known internal services

For most enterprise reporting agents that legitimately need to call Anthropic and Supabase, full network isolation isn't practical. But restricting to known hosts is a reasonable middle ground.

---

### Lever 4: Resource Limits

Prevent a runaway agent from consuming all available RAM or CPU.

**Why this happens:** An agent processing a large CSV in memory, running in a loop due to a bug, or generating huge amounts of output can exhaust system resources. On a shared server, this crashes everything else.

**On Linux directly:**

```bash
# Set limits before running the agent
ulimit -v 2097152    # 2GB virtual memory limit
ulimit -t 600        # 600 CPU seconds total
ulimit -f 1048576    # 1GB max file size

# Then run the agent
python agent.py
```

Or in a systemd service:

```ini
[Service]
MemoryMax=2G
CPUQuota=150%           # can use at most 1.5 CPU cores
LimitNOFILE=1024        # max open file descriptors
```

**In Docker:**

```bash
docker run \
  --memory 2g \         # hard cap: 2GB RAM
  --memory-swap 2g \    # no swap (swap = slowness, not extra capacity)
  --cpus 1.5 \          # at most 1.5 CPU cores
  my-agent
```

**Practical limits to set:**

For a typical reporting agent that queries Supabase and generates HTML: 2GB RAM and 1–2 CPU cores is generous. If it's consistently hitting those limits, something is wrong with the workload, not with the limits.

---

### Lever 5: Time Limits

Prevent agents from running indefinitely.

An agent that gets stuck in a loop, waits forever for a slow API, or just takes longer than expected should be stopped and logged — not left running until someone notices.

**One-shot agents:**

```bash
# Kill the agent if it takes more than 10 minutes
timeout 600 python agent.py

# With a specific exit code
timeout --kill-after=30 600 python agent.py
echo "Exit code: $?"
# Exit code 124 means timeout was reached
```

**Daemon agents (systemd):**

```ini
[Service]
TimeoutStartSec=60    # fail if it doesn't start within 60 seconds
TimeoutStopSec=30     # wait 30 seconds after SIGTERM before SIGKILL
RuntimeMaxSec=3600    # restart service if it runs for more than 1 hour
```

**Why this matters for enterprise agents:** A nightly data sync that was supposed to take 10 minutes and is still running 3 hours later has likely hit a bug. You want it stopped and logged, not silently consuming server resources through the next day.

## Practical Sandbox Setups

Three real configurations for three real situations.

---

### Development (Your Laptop)

Goal: catch mistakes before they spread. Keep things simple.

```bash
# Create a dedicated sandbox directory
mkdir -p ~/agents/sandbox
cd ~/agents/sandbox

# Load credentials from .env
source .env

# Run the agent with a time limit
timeout 600 python ~/agents/agent.py

# Consequences of a mistake: limited to ~/agents/sandbox/
# Recovery: delete ~/agents/sandbox/ and start fresh
```

You don't need Docker for development. A dedicated directory and a time limit is usually enough.

---

### Production Reporting Agent (Hetzner Server)

Goal: the agent is unattended. Mistakes should not affect the server or other agents.

```bash
# /etc/systemd/system/reporting-agent.service
[Unit]
Description=Reporting Agent
After=network.target

[Service]
User=reportingagent
WorkingDirectory=/home/reportingagent/workspace
ExecStart=timeout 600 /home/reportingagent/agent.py --report-type monthly
EnvironmentFile=/home/reportingagent/.env
Restart=on-failure
RestartSec=60
MemoryMax=2G
CPUQuota=100%

[Install]
WantedBy=multi-user.target
```

The dedicated `reportingagent` user means mistakes stay in its home directory. The 600-second timeout kills runaway executions. The 2GB memory limit prevents disk-filling log dumps from causing a server crash.

---

### Experimental Agent (Docker)

Goal: you're trying something new and want full containment. Mistakes should affect nothing outside the container.

```dockerfile
# Dockerfile
FROM python:3.11-slim

# Create a non-root user inside the container
RUN useradd -m agentuser

# Install dependencies as root, then switch to restricted user
COPY requirements.txt /tmp/
RUN pip install -r /tmp/requirements.txt

USER agentuser
WORKDIR /workspace

COPY agent.py /home/agentuser/

CMD ["python", "/home/agentuser/agent.py"]
```

```bash
# Run it
docker run \
  --memory 1g \
  --cpus 1 \
  --network none \                     # no internet access
  -v $(pwd)/workspace:/workspace \     # only this directory is accessible
  --rm \                               # delete container when done
  my-experimental-agent

# Consequences of a mistake:
# - Contained to /workspace mount
# - No network access
# - Container deleted when done
# Recovery: nothing to clean up. Run again.
```

## What Gemini CLI Does for Sandboxing

For reference: Gemini CLI has two built-in sandboxing mechanisms that you can replicate manually.

**The approval gate** is a human sandbox: every mutating tool call (writes, deletes, shell commands) requires explicit user confirmation before execution. The user is the sandbox — mistakes require a human to approve them. You can replicate this with a `--dry-run` flag that shows what the agent would do without doing it.

**The shadow git repo** is a filesystem sandbox: before every file mutation, Gemini commits the current state to a hidden git repository. Every change is reversible. You can replicate this by running `git commit -am "before agent run"` before any session where the agent will modify important files.

Two lines of practice that give you the same safety net:

```bash
# Before any agent session that modifies important files
git add -A && git commit -m "checkpoint before agent run $(date +%Y%m%d-%H%M)"

# When reviewing what the agent did
git diff HEAD~1
git log --oneline -10
```

## The Recovery Question

Before running any agent in a new context, answer this question:

**"If this goes wrong in the worst plausible way, how do I recover?"**

| Scenario | Recovery if sandboxed | Recovery if not sandboxed |
|----------|----------------------|--------------------------|
| Agent deletes wrong files | Restore workspace from backup | Restore entire server from backup |
| Agent fills disk with output | Delete workspace directory | Emergency disk cleanup, other services affected |
| Agent overwrites source data | No source data was in agent's workspace | Restore from database backup |
| Agent exposes credentials in response | Credentials were not readable by agent user | Rotate all credentials, audit what was exposed |
| Agent loops for 6 hours | Timeout killed it after 10 minutes | Kill process manually, 6 hours of compute wasted |

The goal is not zero risk. The goal is recoverable risk. Mistakes that take 5 minutes to recover from are acceptable. Mistakes that take a day and cause data loss are not.

Set up the sandbox before you need it.
