# 7.4 What Can Your Agent Touch? Filesystem Access

**Permissions, the workspace pattern, four things that go wrong, and the rule of least privilege**

---

## The Filesystem is the Agent's World

Everything the agent does that persists goes to the filesystem. Reports, scripts, memory files, logs, intermediate data — all files. The conversation context is ephemeral (it's RAM, cleared between sessions). The filesystem is what survives.

This means filesystem design is agent architecture. Where files live, who can read them, how they're organized — these decisions shape what your agent can do and what happens when it goes wrong.

---

## Unix File Permissions in 60 Seconds

Every file and directory on a Linux system has an owner and a set of permission bits. Understanding these is prerequisite to deploying agents reliably.

```bash
ls -la ~/workspace/

# Output:
# drwxr-xr-x  5 blake blake  160 Mar 15 09:00 .
# drwxr-xr-x 18 blake blake  576 Mar 15 08:00 ..
# -rw-r--r--  1 blake blake 2341 Mar 15 09:00 AGENTS.md
# drwxr-xr-x  3 blake blake   96 Mar 15 09:00 memory
# drwxr-xr-x  2 blake blake   64 Mar 15 09:00 reports
# -rwxr-xr-x  1 blake blake  512 Mar 15 09:00 generate_report.sh
```

Reading the permission string:

```
-  rw-  r--  r--
│   │    │    │
│   │    │    └── others: read only
│   │    └─────── group: read only
│   └──────────── owner: read + write
└──────────────── type: - = file, d = directory
```

- `r` = read (open the file, list the directory)
- `w` = write (modify the file, create files in the directory)
- `x` = execute (run it as a program if a file; enter the directory if a dir)

**The agent process runs as a user.** It can only access files that user has permission to access. If your agent runs as `blake`, it can read/write any file `blake` can read/write.

```bash
# Check what user your agent will run as
whoami

# Check permissions on a specific file
ls -la /path/to/file

# Check if current user can read a file
cat /path/to/file   # if it fails with "Permission denied", you can't
```

## The Working Directory Convention

The single most important filesystem convention for agents: **always start from a known, fixed directory.**

```bash
# Bad: start wherever you happen to be
cd /tmp && python agent.py
# or
python ~/agents/agent.py   # working dir is wherever your shell is

# Good: always start from the agent's designated directory
cd /home/openclaw/workspace && python /home/openclaw/agent.py

# Better: make this explicit and non-negotiable in the service definition
[Service]
WorkingDirectory=/home/openclaw/workspace
ExecStart=/home/openclaw/agent.py
```

When the agent uses relative paths — `./reports/`, `./memory/`, `../data/` — they all resolve relative to the working directory. If the working directory is unpredictable, relative paths break unpredictably.

**The fix is a convention, not a rule enforcement mechanism.** Agree with yourself and your AGENTS.md that the agent always starts from a specific root. Then relative paths are reliable and portable.

## The Workspace Pattern

OpenClaw and Pi both use a workspace directory as the agent's "home." This pattern is worth copying for all your enterprise agents.

```
/home/openclaw/workspace/
├── AGENTS.md              ← injected into every session as system context
├── memory/                ← agent writes persistent notes here
│   ├── 2026-03-15.md      ← what happened today
│   └── known-issues.md    ← running notes the agent maintains
├── reports/               ← output: generated reports
│   ├── weekly-2026-03-10.html
│   └── weekly-2026-03-17.html
├── scripts/               ← reusable tools the agent builds and uses
│   ├── query_claims.py
│   ├── format_report.sh
│   └── send_to_slack.sh
└── data/                  ← input data (or symlinks to data)
    └── claims_2026_q1.csv
```

**Why this works:**

The agent always knows where it is. It doesn't need to search for files. It doesn't need hardcoded absolute paths. Everything is discoverable from the workspace root.

When you add a new script to `scripts/`, it's immediately available to the agent because the agent knows to look there. When you want to find an old report, you know exactly where it is.

**Hardcoded absolute paths are fragile:**

```bash
# Bad: breaks on any machine that isn't your exact laptop
bash("cat /Users/blake/Desktop/reports/claims.csv")

# Good: works wherever the workspace is
bash("cat ./data/claims.csv")
bash("cat $WORKSPACE_DIR/data/claims.csv")
```

Set `WORKSPACE_DIR` in your `.env` file. Now any script or agent can find the workspace regardless of server.

## Four Things That Go Wrong

### 1. The Agent Escapes the Workspace

The agent is told to analyze sales data. It calls `bash("ls ~/Documents/")`. That works — it has your permissions. It finds something interesting in `~/Downloads/`. It reads it. None of this was intended.

**Why it happens:** The agent has no built-in workspace boundary. It can explore the entire filesystem it has access to.

**What to do:**

First line of defense: put explicit guidance in `AGENTS.md`:

```markdown
## Workspace Boundaries
Your workspace is /home/openclaw/workspace/. All reads and writes should
happen within this directory unless explicitly told otherwise. Do not read
files in other users' home directories, /etc/, or system directories.
```

Second line of defense: run the agent as a restricted user who literally can't access those directories. (See 7.5.)

Third line of defense: Docker with a limited filesystem mount.

---

### 2. Disk Space Exhaustion

The agent generates a report. Then another. And another. Each report is 2MB. After six months, unnoticed, reports take up 180GB. Your server has a 200GB disk. Other processes start failing.

**Why it happens:** The agent writes files. Nobody set up cleanup. Disks don't signal "getting full" loudly.

**What to do:**

```bash
# Log rotation for agent logs
# /etc/logrotate.d/openclaw
/var/log/openclaw/*.log {
    daily
    rotate 14
    compress
    missingok
    notifempty
}

# Workspace cleanup script (run weekly via cron)
# cleanup_old_reports.sh
find /home/openclaw/workspace/reports -mtime +90 -name "*.html" -delete
find /home/openclaw/workspace/reports -mtime +90 -name "*.pdf" -delete

# Disk monitoring (add to crontab)
0 * * * *  df -h / | awk 'NR==2 && $5+0 > 85 {print "Disk >85% full on " $6}' | mail -s "Disk Alert" you@example.com
```

---

### 3. Sensitive File Exposure

Your agent is running. You ask it a question about the server setup. It decides to read `.env` to understand the configuration. The credentials are now in the conversation context — and potentially in the response it sends back.

**Why it happens:** `.env` is in the workspace directory. The agent can read it. There's nothing stopping it by default.

**What to do:**

Put this in `AGENTS.md`, explicitly and early:

```markdown
## Files You Must Never Read
- .env (contains credentials — you do not need to read this)
- Any file containing API keys, passwords, or tokens
- /etc/shadow, /etc/passwd (system credential files)

If a task seems to require reading these files, ask the user to confirm
instead of reading them automatically.
```

And use a dedicated user with restricted permissions so the agent process genuinely can't read your personal `.env` even if it tries. (Details in 7.5.)

---

### 4. Concurrent Access Corruption

You run two agent sessions simultaneously. Both are generating reports. Both write to `./reports/current-report.md`. The file ends up as corrupted interleaved content from both sessions.

**Why it happens:** Two processes writing to the same file without coordination will corrupt each other's output. The OS doesn't prevent this by default.

**What to do:**

```bash
# Session-specific workspaces: give each session its own output directory
REPORT_PATH="./reports/report-$(date +%Y%m%d-%H%M%S).md"
bash("python generate_report.py > $REPORT_PATH")

# File locking for truly shared resources
bash("flock -x ./locks/report.lock python generate_report.py > ./reports/report.md")

# Or use a database for shared state
# (Supabase handles concurrent writes safely via transactions)
```

The simplest solution: design agents so each session writes to its own uniquely-named file. Shared state should live in a database, not in a shared file.

## Remote Filesystem Access

Your Supabase database is not on your agent's filesystem. Neither is an S3 bucket, or a file on another Hetzner server. These are **network resources.**

The agent accesses them via network commands:

```bash
# Supabase: SQL over network
bash("psql $SUPABASE_URL -c 'SELECT * FROM claims WHERE status = $1' 'pending'")

# Supabase via Python client
bash("python query_claims.py --status pending --output ./data/pending_claims.json")

# Remote file on another server (via SSH)
bash("scp user@other-server:/data/export.csv ./data/export.csv")

# S3 / object storage
bash("aws s3 cp s3://my-bucket/reports/latest.csv ./data/")
```

The agent doesn't see these as files — it sees them as things it can reach via network commands. The results come back as command output or as local files after being downloaded.

**You CAN mount remote filesystems locally:**

```bash
# SSHFS: mount a remote directory as if it were local
sshfs user@other-server:/data /mnt/remote-data

# NFS: same idea, different protocol (common in corporate networks)
mount -t nfs other-server:/exports/data /mnt/remote-data
```

Once mounted, the agent treats `/mnt/remote-data/` exactly like a local directory. Reads and writes go over the network transparently. This is convenient but adds complexity and a dependency on network availability.

## The Rule of Least Privilege

There is a principle in security that applies directly to agents:

> **Grant only the access that is actually needed. Nothing more.**

For agents specifically:

```
Don't run your agent as root.
Don't run your agent as your personal user account if a dedicated user will do.
Don't mount your entire home directory if the agent only needs ./workspace/.
Don't give write access to directories the agent only needs to read.
The agent should be able to break its own workspace, not your whole server.
```

This is not paranoia. It's engineering discipline. Agents make mistakes — they misinterpret instructions, explore where they shouldn't, run destructive commands with good intentions. The job of the infrastructure layer is to make those mistakes recoverable.

If the agent can only write to `~/workspace/`, it can trash `~/workspace/`. You delete it and recreate it. If the agent can write to `~/` as your personal user, it can overwrite your `.ssh/authorized_keys`. That's a very different kind of mistake.

The mechanics of how to achieve this are covered in the next lesson.
