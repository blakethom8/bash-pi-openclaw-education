# 7.1 The Agent is a Process

**The foundation: everything your agent can do is determined by where its process runs**

---

## What is a Process?

When you type `python agent.py` and press Enter, the operating system does something specific: it creates a **process**.

A process is not just "a running program." It is a bundle of resources the OS allocates and tracks:

- **RAM:** A block of memory for the code, variables, buffers, and data
- **CPU time:** A slice of processor time on a rotating schedule with other processes
- **Open file table:** A list of files the process currently has open
- **Environment variables:** A copy of key=value pairs inherited from the shell that launched it
- **Working directory:** The directory the process was started from
- **User identity:** The OS user the process runs as (you, or a service account, or root)

Every one of these matters for your agent. Let's go through them.

---

## The Agent's User Identity

The most important thing about your agent process: **it runs as a specific OS user.**

If you open a terminal on your laptop and type `python agent.py`, that process runs as you — with every permission, every file access right, every network capability you have.

```bash
# See all running processes and their owners
ps aux | grep python

# Output:
blake  12847  2.1  0.8  python agent.py
blake  12901  0.0  0.1  grep python
```

The first column is the user. That agent is running as `blake`.

**What this means:**

When the agent calls `bash("cat /etc/passwd")`, that runs as `blake`. Can `blake` read `/etc/passwd`? Yes. So the agent can.

When the agent calls `bash("cat /etc/shadow")`, that runs as `blake`. Can `blake` read `/etc/shadow`? No — that requires root. So the agent gets a permission denied error.

When the agent calls `bash("rm -rf ~/Documents/")`, that runs as `blake`. Can `blake` delete `~/Documents/`? Yes. So the agent can — and will, if told to.

> **The agent has no special powers. It is exactly as powerful — and exactly as dangerous — as you running those same commands yourself in a terminal.**

This is the critical mental model. The agent is not sandboxed by default. It is not restricted by default. It is you, automated.

## The Agent's View of the Filesystem

The agent can read and write files its process has permission to access, relative to where it was started.

```bash
# Agent calls this:
bash("cat config.json")

# The OS runs:
bash -c "cat config.json"
# ...as the user who started the agent
# ...from the working directory the agent was started in
```

If the agent was started from `/home/blake/reports/`, then `cat config.json` looks for `/home/blake/reports/config.json`.

If the agent was started from `/`, then it looks for `/config.json`.

**This is why starting location matters.** An agent that works perfectly on your laptop might behave differently on your server if it was started from a different directory.

The fix is simple: always use a known, intentional working directory.

```bash
# Bad: start the agent wherever you happen to be
cd ~/random-place && python agent.py

# Good: always start from the agent's home directory
cd ~/openclaw/workspace && python agent.py

# Better: make the start location explicit in your systemd service or startup script
WorkingDirectory=/home/openclaw/workspace
ExecStart=/home/openclaw/agent.py
```

## Environment Variables: How Credentials Reach the Agent

The agent process inherits your shell's environment variables when it starts. This is the standard mechanism for passing credentials and configuration without hardcoding them.

```bash
# In your shell (or .env file):
export SUPABASE_URL="postgresql://user:pass@db.supabase.co/postgres"
export SUPABASE_ANON_KEY="eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9..."
export ANTHROPIC_API_KEY="sk-ant-..."

# The agent process inherits all of these
python agent.py
```

Inside the agent code, access them like this:

```python
import os

# Python
supabase_url = os.getenv("SUPABASE_URL")
api_key = os.getenv("ANTHROPIC_API_KEY")

if not supabase_url:
    raise ValueError("SUPABASE_URL not set — check your .env file")
```

Or from bash tools the agent calls:

```bash
# Inside a bash tool call, env vars are already available:
bash("psql $SUPABASE_URL -c 'SELECT COUNT(*) FROM claims'")
```

**The practical pattern is a `.env` file:**

```bash
# .env — sits in the agent's working directory, never committed to git
SUPABASE_URL=postgresql://user:pass@host/db
ANTHROPIC_API_KEY=sk-ant-...
OUTPUT_DIR=/home/blake/reports

# Load it before starting the agent:
source .env && python agent.py

# Or in Python:
from dotenv import load_dotenv
load_dotenv()
```

The agent never sees the credentials file directly — it sees the environment variables that were loaded from it. This is intentional. Keep the secrets out of the codebase and out of the conversation context.

## Where RAM Actually Gets Used

This is a common source of confusion. When your agent sends a request to the Anthropic API, where does the work happen?

```
Your Machine                          Anthropic's Servers
──────────────────────────────────    ─────────────────────────────────────
Agent loop code        → RAM here    Claude model weights → RAM there
Messages array         → RAM here    Inference compute   → CPU/GPU there
Tool output buffers    → RAM here
Python/Node runtime    → RAM here
```

**Your machine holds:**
- The agent loop code itself (small)
- The messages array — the accumulated conversation history (grows with every turn)
- The output of every tool call (this is where things get large)
- The Python or Node runtime and loaded libraries

**Anthropic's servers hold:**
- The actual Claude model (billions of parameters, hundreds of gigabytes)
- The compute resources that run inference

Every LLM API call sends your current messages array **over the network** to Anthropic, runs inference there, and sends back a response. The model never runs on your machine.

**Where you actually run out of memory:**

Not from the model — from tool outputs. If your agent runs `bash("cat huge_dataset.csv")` and that CSV is 500MB, that 500MB gets loaded into the messages array. Your machine's RAM holds it. And it gets sent to Anthropic with every subsequent API call.

This is why large tool outputs need to be handled carefully:

```bash
# Bad: dumps entire 500MB file into agent context
bash("cat /data/full_claims_export.csv")

# Good: give the agent what it needs, not everything
bash("head -20 /data/full_claims_export.csv")
bash("wc -l /data/full_claims_export.csv")
bash("python summarize_claims.py /data/full_claims_export.csv")
```

## The Three Things That Determine What an Agent Can Do

After everything above, it comes down to this:

**1. What user it runs as (permissions)**

Who is the process? What files can that user read and write? What commands can they run? Can they install software? Can they access other users' data? Can they sudo?

**2. What machine it runs on (filesystem and network access)**

Does this machine have the right files? Can it reach Supabase? Can it reach the Anthropic API? Is the `/data/` directory mounted? Does it have enough RAM for the workloads you're planning?

**3. What environment variables it has (credentials and config)**

Does the process know the database URL? Does it have the API key? Does it know where to write output? All of this flows in through environment variables.

Change any one of these three things and the agent's behavior changes completely. An agent that works on your laptop might fail on a server because the server uses a different user (different permissions), doesn't have a file your laptop had, or is missing an environment variable you forgot to set.

## Seeing It In Practice

When you run an agent and want to know exactly what you're dealing with:

```bash
# Find the agent process
ps aux | grep agent.py

# What user is it running as?
# (first column of ps output)

# What working directory?
ls -la /proc/$(pgrep -f agent.py)/cwd
# or on Mac:
lsof -p $(pgrep -f agent.py) | grep cwd

# What environment variables does it have?
cat /proc/$(pgrep -f agent.py)/environ | tr '\0' '\n'

# How much RAM is it using?
ps -o pid,vsz,rss,comm -p $(pgrep -f agent.py)
```

You don't need to run these regularly — but knowing you *can* demystifies the agent. It's a process. You can inspect it like any other process.

## Practice

Before moving on, trace through this scenario mentally:

**Scenario:** You run `cd /home/blake/reports && python agent.py` on your Hetzner server. The agent calls `bash("python generate_report.py ../data/claims.csv")`.

- What user does this run as?
- What directory does `../data/claims.csv` resolve to?
- If `SUPABASE_URL` wasn't exported before running the agent, what happens when the agent calls `bash("psql $SUPABASE_URL -c 'SELECT ...')`?

**Answers:** It runs as `blake`. The path resolves to `/home/blake/data/claims.csv`. The `psql` call gets `psql  -c 'SELECT ...'` — empty URL, immediate error.

These are the kinds of silent failures that waste hours. Understanding processes prevents them.
