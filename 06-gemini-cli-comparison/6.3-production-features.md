# 6.3 What Gemini CLI Adds for Production Use

**The features Pi doesn't have — and why they exist**

---

## Overview

Pi proves the loop works. These features answer the question: "What do you need when the loop runs against real infrastructure, on a team, with real consequences?"

Each feature addresses a specific production failure mode. Understanding *why* it exists is as useful as understanding *what* it does.

---

## 1. Plan Mode

**The problem it solves:** You ask an agent to "update all the Q4 reports with the corrected multiplier." It starts executing. You realize partway through it's touching files you didn't intend. No undo.

**What Plan Mode does:**

```
User: "Regenerate all Q4 provider reports with the updated rate table"

[Plan Mode — read-only enforcement active]

Agent (Pro model, planning):
  Step 1: Read current rate table from rates/2026-q4.csv
  Step 2: List all files matching reports/q4-*.md
  Step 3: For each report, recalculate amounts using new rates
  Step 4: Write updated versions (34 files)
  Step 5: Generate summary of changes

Plan saved to: .gemini/plan-2026-03-15-14-32.md
Edit the plan file, then approve to execute.
```

The plan is a markdown file you can edit. Remove steps, add constraints, change scope. When you approve, execution begins with a Flash model (cheaper, faster for mechanical execution) following the plan.

**The two-model strategy:**
- Pro model = expensive, best reasoning → use for planning (one call)
- Flash model = cheap, fast → use for execution (many calls)

**When to use:**
- Complex multi-step tasks touching many files
- Database migrations or mass updates
- Anything where "oops" has real consequences
- When you want an audit trail of what the agent intended to do

**Enterprise angle:** Before running a batch claims reprocessing job, plan mode gives you a reviewable list of steps to hand to a supervisor before execution starts.

## 2. Subagents with Isolated Context

**The problem it solves:** Complex tasks involve multiple phases — research, then write, then deploy. If all three happen in one context loop, by the time you're deploying, the context window is full of investigation artifacts from phase one.

**What subagents do:**

Each subagent is a fresh context loop. The main agent delegates a task, the subagent runs to completion, returns only its final result, and the main agent continues.

```
Main agent context:
  [user request] → delegate to codebase_investigator
                   ↓ (fresh context)
                   Subagent explores codebase, reads 40 files
                   Subagent summarizes findings
                   ↓ (only summary returned)
  [summary from investigator] → write report → done
```

**Built-in subagents:**
- `codebase_investigator` — reads and summarizes codebases
- `generalist_agent` — general task execution
- `browser_agent` — web browsing and scraping
- `cli_help` — explains CLI tools and commands

**Constraints:** Subagents cannot call other subagents. Delegation is one level deep.

**Why this matters for context window:**

```
Without subagents:
  Turn 1-20: explore codebase (fills context)
  Turn 21-30: write report (context getting crowded)
  Turn 31+: deploy (context near limit, accuracy degrades)

With subagents:
  Main: delegate investigation → subagent returns 500-token summary
  Main: write report (clean context)
  Main: delegate deployment → subagent handles it
```

**Enterprise angle:** "Analyze all Q1 claims data and produce a payer breakdown" delegated to a subagent. Main agent keeps a clean context for the presentation layer work.

## 3. Checkpointing

**The problem it solves:** Agent runs a batch operation, something goes wrong halfway through, files are in an inconsistent state. Your real git history is now polluted with partial changes.

**What checkpointing does:**

Before every file mutation, Gemini CLI creates a snapshot in a git shadow repo at:
```
~/.gemini/history/<project_hash>/
```

This is a complete git repo mirroring your project state. Each snapshot is a commit. Your actual `.git` repo is untouched.

```bash
# Agent is about to write 34 report files
# Gemini CLI does this automatically:
cd ~/.gemini/history/a3f9b2c1/
git add .
git commit -m "pre-mutation: 2026-03-15T14:32:00Z"

# Now executes the 34 writes
# If something goes wrong:
git checkout a3f9b2c1 -- reports/
# → All 34 files restored to pre-mutation state
```

**What you get:**
- Safety net for every file mutation, automatically
- Rollback without affecting your real git history
- Full audit trail of what the agent changed and when

**What you lose:**
- Disk space (shadow repo grows over time)
- It only covers files (not database mutations, not API calls)

**Enterprise angle:** Before a mass report regeneration job — automatic safety net with no setup required. Before a database migration — note that checkpointing won't cover DB changes, only files.

## 4. Policy Engine

**The problem it solves:** On a team, different people should have different levels of trust. Different projects have different risk profiles. You want `read_file` to always be allowed, `write_file` to require confirmation, and `run_shell_command` to be completely denied in sensitive contexts.

**What the policy engine does:**

Five priority tiers (each overrides the ones below it):
```
admin     > user     > workspace  > extension  > default
(org IT)    (you)      (project)    (extension)  (built-in)
```

Rules match on tool name and argument patterns:

```json
// workspace policy (in .gemini/policy.json)
{
  "rules": [
    {
      "tool": "read_file",
      "action": "allow"
    },
    {
      "tool": "write_file",
      "path_pattern": "reports/.*",
      "action": "allow"
    },
    {
      "tool": "write_file",
      "path_pattern": "src/.*",
      "action": "ask"
    },
    {
      "tool": "run_shell_command",
      "command_pattern": "rm -rf.*",
      "action": "deny"
    }
  ]
}
```

**Tier use cases:**
- **Admin:** IT locks down certain tools org-wide. Agent can't bypass.
- **Workspace:** Project-specific rules. Production repo = stricter than dev.
- **User:** Personal preferences. You always auto-approve your own scripts.
- **Extension:** Extension ships with default policies for its tools.
- **Default:** Gemini CLI's built-in defaults (confirm writes, auto-approve reads).

**Enterprise angle:** Compliance environment — admin policy prevents `run_shell_command` entirely. All execution must go through approved, audited scripts exposed as named MCP tools. The policy engine enforces this at the framework level, not just by convention.

## 5. MCP Protocol

**The problem it solves:** Every agent framework invents its own tool system. A Supabase integration built for Gemini CLI doesn't work in Claude Code. An integration built for Claude Code doesn't work in LangChain. You rebuild the same integrations repeatedly.

**What MCP (Model Context Protocol) does:**

MCP is a standardized protocol for tool servers. Any language, any runtime.

```
Agent (any MCP-compatible AI) ←→ MCP Protocol ←→ Tool Server (Python/Node/Go/etc.)
```

**Transport options:**
- `stdio` — local process, agent launches it
- `HTTP` — remote server, agent calls over network

**A Supabase MCP server in Python:**
```python
# supabase_mcp_server.py
from mcp.server import Server
import supabase

server = Server("supabase-tools")

@server.tool("query_claims")
def query_claims(status: str, date_from: str, date_to: str) -> str:
    """Query claims by status and date range"""
    client = supabase.create_client(os.environ["SUPABASE_URL"], os.environ["SUPABASE_KEY"])
    result = client.table("claims") \
        .select("*") \
        .eq("status", status) \
        .gte("submitted_date", date_from) \
        .lte("submitted_date", date_to) \
        .execute()
    return json.dumps(result.data)

@server.tool("lookup_provider")
def lookup_provider(npi: str) -> str:
    """Look up provider by NPI"""
    # ...

server.run()
```

Register it in GEMINI.md:
```json
{
  "mcpServers": {
    "supabase": {
      "command": "python",
      "args": ["supabase_mcp_server.py"]
    }
  }
}
```

**Now `query_claims` and `lookup_provider` appear as first-class tools in Gemini CLI.** Same server works in Claude Code, Cursor, or any MCP-compatible agent.

**Enterprise angle:** Build your integrations as MCP servers once. They're reusable across AI tools, versioned, testable, and don't require touching agent framework internals.

> MCP is the closest thing to a universal tool standard that exists right now. Worth investing in.

## 6. Hooks

**The problem it solves:** You need to do something every time a tool runs — log it, validate it, enrich context, filter output. Without hooks, you'd have to fork the agent framework.

**What hooks do:**

Intercept at 10+ lifecycle points:

```
SessionStart          → runs when agent session begins
BeforeAgent           → before agent loop starts
BeforeModel           → before each LLM call
AfterModel            → after each LLM call
BeforeToolSelection   → before model picks a tool
BeforeTool            → before each tool execution
AfterTool             → after each tool execution
AfterAgent            → after agent loop ends
SessionEnd            → when session terminates
Notification          → on agent status changes
PreCompress           → before context compression
```

**Example hook implementations:**

```python
# Audit logging — AfterTool hook
def after_tool(tool_name: str, args: dict, result: str):
    log_entry = {
        "timestamp": datetime.now().isoformat(),
        "tool": tool_name,
        "args": args,
        "result_preview": result[:200]
    }
    with open("~/.gemini/audit.jsonl", "a") as f:
        f.write(json.dumps(log_entry) + "\n")

# PII detection — BeforeTool hook
def before_tool(tool_name: str, args: dict) -> dict:
    if tool_name == "write_file":
        if contains_pii(args.get("content", "")):
            raise ValueError("PII detected in write_file content. Redact before writing.")
    return args  # pass through unmodified

# Context injection — BeforeAgent hook
def before_agent(context: list) -> list:
    # Inject current date/time into context before every session
    context.insert(0, {
        "role": "system",
        "content": f"Current date: {date.today().isoformat()}"
    })
    return context
```

**Enterprise angles:**
- **AfterTool** → compliance audit log of every agent action
- **BeforeTool** → PII/sensitive data detection before sending to model
- **BeforeModel** → rate limiting, cost tracking
- **AfterModel** → response filtering, toxicity screening
- **PreCompress** → preserve critical context before compression drops it

## 7. Model Steering

**The problem it solves:** An agent is running a long batch job. You realize mid-execution it's using the wrong data source, or you want to change scope. Without model steering, you have to cancel and restart.

**What model steering does:**

Inject a hint into the agent's context mid-execution without stopping it:

```
[Agent is generating 34 reports using Q3 data — turn 12 of ~40]

User injects: "Actually use Q4 data (rates/2026-q4.csv), not Q3"

Agent receives hint, adjusts:
  → Stops reading Q3 files
  → Switches to Q4 rate table
  → Continues from current state
```

The agent doesn't restart. It incorporates the hint and continues. Context accumulated so far is preserved.

**Practical limits:**
- The agent may already be committed to an approach that requires restart
- Works best for redirections ("use X instead of Y"), not reversals ("undo everything")
- Useful for long runs where context gives you visibility into what the agent is doing

**Enterprise angle:** Batch report generation running overnight. Monitoring shows it's using the wrong cohort. Inject a correction without canceling a multi-hour run.

## Summary: Which Features Apply to You

| Feature | When it matters | Priority |
|---------|----------------|---------|
| Plan Mode | Before risky batch operations | High |
| Checkpointing | Any batch file mutation | High |
| MCP servers | Building reusable integrations | Medium-High |
| Policy engine | Team use, compliance requirements | Medium |
| Hooks | Audit logging, PII detection | Medium |
| Subagents | Long tasks hitting context limits | Low-Medium |
| Model steering | Long-running batch jobs | Low |
