# 6.4 What to Take Back to Your Own Agent Design

**Actionable patterns from Gemini CLI for Pi and OpenClaw users**

---

## The Core Principle

You don't need Gemini CLI to use Gemini CLI's ideas. Every production feature it ships has a DIY equivalent that works with Pi, OpenClaw, or any bash-based agent.

The goal isn't to replicate Gemini CLI. The goal is to understand *why* it built each feature and apply that reasoning to your own setup.

---

## Concept 1: Domain-Native Context Is Your Competitive Advantage

**What Gemini has:** GCP extensions that bake infrastructure knowledge into the agent.

**What you have:** AGENTS.md with Hetzner server details and Supabase schemas.

These are the same thing. The difference is investment level.

**Steal this:**

Treat your AGENTS.md as a first-class engineering artifact. Every hour you spend making it more precise saves hours of agent trial-and-error.

Checklist for a high-investment AGENTS.md:
- [ ] Server hostnames, private IPs, and SSH access patterns
- [ ] Database connection environment variable names (not actual credentials)
- [ ] Table schemas for every table the agent touches regularly
- [ ] Column-level notes: what the values mean, what indexes exist, query performance notes
- [ ] Naming conventions (file naming, report naming, variable naming)
- [ ] Available scripts and exactly what parameters they take
- [ ] Constraints ("never query without a date filter", "never write to prod schema")
- [ ] Output format preferences (CSV for data exports, MD for summaries, HTML for reports)

Every item on this list is context the agent doesn't have to discover. Discovery costs turns. Turns cost tokens. Tokens cost money and time.

## Concept 2: On-Demand Skill Loading

**What Gemini has:** `activate_skill` tool — brief skill index in system prompt, full docs loaded only when needed.

**What Pi/OpenClaw does:** Loads all skill documentation into context at startup.

**The problem with eager loading at scale:**

```
3 skills:  ~3,000 tokens loaded at startup  ← fine
10 skills: ~10,000 tokens loaded at startup ← starting to matter
20 skills: ~20,000 tokens loaded at startup ← burning context budget
50 skills: ~50,000 tokens loaded at startup ← problem
```

**Steal this (DIY version):**

Keep your AGENTS.md skill section as a brief index:

```markdown
## Available Skills

| Skill | When to use |
|-------|-------------|
| supabase-query | Query claims, providers, payers tables |
| hetzner-ops | Deploy, restart services, check server health |
| pdf-report | Convert markdown report to PDF |
| npi-lookup | Look up provider details by NPI |

Full skill docs: workspace/skills/<skill-name>/SKILL.md
Read the relevant SKILL.md before using a skill.
```

The agent reads only the skills it needs, only when it needs them. Context window stays clean for the actual work.

## Concept 3: Plan Before You Act

**What Gemini has:** Plan Mode — read-only enforcement, editable plan file, Pro model for planning.

**What Pi/OpenClaw has:** Nothing. The agent just starts executing.

**The failure mode:** "Update all Q4 reports" → agent starts writing → you realize it's touching the wrong files → no undo.

**Steal this (DIY version):**

Add a planning step to your agent scripts:

```bash
#!/bin/bash
# run-agent.sh

MODE=${1:-execute}

if [ "$MODE" = "--plan" ]; then
  # Inject plan-only instructions
  EXTRA_INSTRUCTIONS="
  PLANNING MODE: Do NOT execute any writes or shell commands.
  Instead, output a numbered list of exactly what you would do,
  including which files you would write, which scripts you would run,
  and what changes you would make. Stop after producing the plan.
  "
fi

# Pass EXTRA_INSTRUCTIONS into system prompt...
```

Usage:
```bash
./run-agent.sh --plan "Regenerate all Q4 reports"
# Review the plan
./run-agent.sh "Regenerate all Q4 reports"
```

**Simpler version:** Add a WORKFLOW section to AGENTS.md that describes standard operating procedures. The agent follows the documented workflow instead of improvising.

```markdown
## Workflow: Report Generation

Standard steps for any report generation request:
1. Read AGENTS.md to confirm output format expectations
2. Check if required data source is available (query test)
3. Generate report to a tmp file first
4. Review tmp file for obvious errors
5. Move to final location with correct naming convention
6. Confirm to user with file path and row count
```

## Concept 4: Checkpointing = Git Commit Before Dangerous Operations

**What Gemini has:** Automatic git shadow repo snapshot before every file mutation.

**What Pi/OpenClaw has:** Nothing. File mutations are final.

**Steal this (DIY version):**

Add a checkpoint step to your agent wrapper scripts:

```bash
#!/bin/bash
# agent-with-checkpoint.sh

TASK="$1"

# Checkpoint before running
echo "Creating checkpoint..."
git add -A
git stash push -m "pre-agent: $(date -u +%Y-%m-%dT%H:%M:%SZ)"

# Run the agent
run-agent "$TASK"

EXIT_CODE=$?

if [ $EXIT_CODE -ne 0 ]; then
  echo "Agent failed. Restoring checkpoint..."
  git stash pop
else
  echo "Done. Checkpoint available via: git stash pop"
fi
```

For file operations without git:
```bash
# Copy workspace to temp before agent runs
BACKUP_DIR="/tmp/agent-backup-$(date +%Y%m%d-%H%M%S)"
cp -r ~/workspace "$BACKUP_DIR"
echo "Backup at: $BACKUP_DIR"
```

Not as elegant as Gemini's shadow repo. Gets you 90% of the safety net.

## Concept 5: Policy Engine → Explicit Permission Lists

**What Gemini has:** Five-tier policy engine with per-tool, per-argument allow/deny/ask rules.

**What Pi/OpenClaw has:** Nothing. The agent can do anything the user can do.

**Steal this (DIY version):**

Write your constraints directly in AGENTS.md. The model follows written rules well:

```markdown
## Constraints and Permissions

### Always Allowed (no confirmation needed)
- Read any file in ~/workspace/
- Query any table in the database (SELECT only)
- Run scripts in scripts/ directory
- Write to ~/workspace/reports/ and ~/workspace/tmp/

### Always Ask Before Doing
- Writing to any file outside ~/workspace/
- Running any command with `sudo`
- Deleting any file
- Making any external API call not listed in this document

### Never Do
- Modify the production database (prod schema)
- Run `rm -rf` or any recursive delete
- Expose NPI + name + amount in the same unredacted output
- Connect to any server not listed in the Infrastructure section
```

This is policy-as-text instead of policy-as-code. It works because LLMs follow written constraints reliably for well-defined rules. For hard security requirements (not just preferences), you want actual enforcement — but for operational guardrails, this gets you most of the way there.

## Concept 6: MCP Is Worth Watching

**What Gemini has:** MCP servers in any language, reusable across frameworks, HTTP transport.

**What Pi/OpenClaw has:** TypeScript extensions, bash scripts.

**Why this matters to you:**

If you build your Supabase integration as an MCP server today, it works with:
- Gemini CLI (now)
- Claude Code (now)
- Future MCP-compatible tools (whenever they arrive)

If you build it as a bash script in your skills directory, it only works with your current setup.

**Steal this (investment decision):**

For integrations you'll use long-term across multiple tools, build MCP servers. For quick one-off integrations, bash scripts are fine.

A minimal Python MCP server for your Supabase queries:

```python
# Install: pip install mcp supabase-py
from mcp.server.fastmcp import FastMCP
import os, json
from supabase import create_client

mcp = FastMCP("supabase-reporting")
client = create_client(os.environ["SUPABASE_URL"], os.environ["SUPABASE_KEY"])

@mcp.tool()
def query_claims(status: str, date_from: str, date_to: str) -> str:
    """Query claims by status and date range. Returns JSON."""
    result = client.table("claims") \
        .select("claim_id,npi,amount,status,submitted_date") \
        .eq("status", status) \
        .gte("submitted_date", date_from) \
        .lte("submitted_date", date_to) \
        .execute()
    return json.dumps({"count": len(result.data), "rows": result.data[:100]})

if __name__ == "__main__":
    mcp.run()
```

Register in Claude Code's MCP config or GEMINI.md's `mcpServers` section. Same server, both tools.

## Quick Reference: Gemini Feature → DIY Equivalent

| Gemini CLI Feature | DIY Equivalent for Pi / OpenClaw |
|-------------------|----------------------------------|
| Plan Mode | `--plan` flag that injects "list steps, don't execute" instruction |
| Checkpointing | `git stash` or `cp -r workspace /tmp/backup` before agent runs |
| `save_memory` tool | Agent writes to `memory/YYYY-MM-DD.md` (it already does this) |
| Policy engine | Constraints section in AGENTS.md |
| On-demand skills | Brief skill index in AGENTS.md, full docs in separate files |
| Subagents | Separate bash scripts called from main agent via `exec()` |
| MCP server | Python script called via bash tool (upgrade path: wrap in MCP) |
| Model steering | Keyboard interrupt → add new message → continue |
| GEMINI.md hierarchy | AGENTS.md + subdirectory NOTES.md files (read manually) |
| Hooks | Wrapper scripts around agent invocation |
| 3-level GEMINI.md | Global `~/.openclaw/AGENTS.md` + workspace `AGENTS.md` + skill docs |

## Prioritized Action List

**High value, low effort:**
1. Expand your AGENTS.md with full schema documentation, server details, and script inventory
2. Add a Constraints section to AGENTS.md
3. Add a brief skills index to AGENTS.md, move full docs to separate files

**Medium value, medium effort:**
4. Add `--plan` mode to your agent scripts
5. Add a checkpoint step to your most-used agent wrappers
6. Write standard workflow procedures in AGENTS.md

**High value, higher effort:**
7. Build your Supabase integration as an MCP server
8. Build your Hetzner operations as an MCP server

**Low priority (monitoring only):**
9. Watch MCP ecosystem growth — more client support coming
10. Watch for Gemini CLI features landing in OpenClaw or Pi equivalents
