# 6.1 The Same Loop, More Features

**Gemini CLI's architecture demystified**

---

## The Core Loop: Side by Side

**Pi's loop:**
```python
context = []

while True:
    response = llm.chat(system_prompt=AGENTS_MD, history=context)

    if response.tool_calls:
        for call in response.tool_calls:
            result = execute_tool(call)   # read / write / edit / bash
            context.append(result)
    else:
        print(response.message)
        break
```

**Gemini CLI's loop:**
```python
context = []

while True:
    response = llm.chat(system_prompt=GEMINI_MD, history=context)

    if response.tool_calls:
        for call in response.tool_calls:
            approved = approval_gate(call)   # confirm / autoEdit / yolo
            if approved:
                result = execute_tool(call)  # 17 named tools + MCP
                context.append(result)
    else:
        print(response.message)
        break
```

**The difference is one line: the approval gate.** Everything else is the same pattern.

---

## The Approval Gate

Pi trusts by default. Every tool call executes immediately.

Gemini CLI confirms by default. Before any write operation, it asks you:

```
✦ write_file reports/q1-summary.md
  [1,247 tokens]
  Allow? (y/n/always)
```

**Approval modes:**
- `default` — ask before writes, auto-approve reads
- `autoEdit` — auto-approve all file edits, ask before shell commands
- `plan` — read-only enforced, no execution until you approve the plan
- `yolo` — everything auto-approved (Pi behavior)

> **The Pi equivalent of `yolo` mode is how Pi always runs.** When you understand the tradeoff, you appreciate why Gemini defaults to confirmation — and why Pi's simplicity is a feature for local dev.

## Tool Count Comparison

**Pi: 4 tools**
```
read(path)
write(path, content)
edit(path, old, new)
bash(command)
```

**Gemini CLI: 17 tools**
```
# Navigation / search
list_directory(path)
glob(pattern)
grep_search(pattern, path)

# Reading
read_file(path)
read_many_files(paths[])

# Writing
write_file(path, content)
replace(path, old, new)

# Execution
run_shell_command(command)

# User interaction
ask_user(question)
write_todos(items[])

# Skills / memory / docs
activate_skill(name)
save_memory(content)
get_internal_docs(query)

# Planning
enter_plan_mode()
exit_plan_mode()

# Web
google_web_search(query)
web_fetch(url)
```

**Plus: unlimited via MCP** — any tool server you plug in adds to this list.

---

## Philosophy Comparison

**Pi:** `bash` is the escape hatch.

> Four tools is all you need because `bash` can do anything. The agent figures out which bash command to run. Discoverability is the model's job.

**Gemini CLI:** Named tools = discoverability + safety.

> Explicit tool names give the agent clearer affordances. `grep_search` is harder to misuse than `bash("rm -rf $(grep -r ...)")`. Named tools also enable the policy engine — you can't write rules against `bash()` if everything is bash.

**Both are correct.** Pi's philosophy is elegant for a personal dev tool. Gemini's philosophy scales to teams where you want per-tool approval rules.

## Context Files: Three Systems, Same Concept

All three systems use markdown files to inject context at agent startup.

**Pi:** Single `AGENTS.md` in the workspace root.

**OpenClaw:** Three files:
```
workspace/
├── AGENTS.md     # Instructions and capabilities
├── SOUL.md       # Persona
└── USER.md       # About the user
```

**Gemini CLI:** Three-level hierarchy:
```
~/.gemini/GEMINI.md          # Global — applies to all projects
./GEMINI.md                  # Workspace — project-specific
[any subdir]/GEMINI.md       # Just-in-time — loaded as agent enters dirs
```

The just-in-time loading is the interesting addition. When the agent reads files in `src/auth/`, it scans for a `GEMINI.md` there and loads it automatically. Context grows as the agent explores.

> **Practical equivalent:** Put a `NOTES.md` in key subdirectories of your workspace. Your agent already uses `read()` — you can instruct it to check for context files as it navigates.

## Skills: Eager vs. On-Demand Loading

**Pi / OpenClaw:** All skill documentation is loaded into the system prompt at startup.

```
System prompt = AGENTS.md + SKILL1.md + SKILL2.md + ... + SKILL_N.md
```

Works fine for 3–5 skills. At 20 skills, you've burned a large chunk of your context window before the conversation even starts.

**Gemini CLI:** Skills are described briefly in the system prompt. Full content loaded only when the agent calls `activate_skill(name)`.

```
System prompt = GEMINI.md + brief skill index

# Agent decides it needs a skill:
activate_skill("cloud-run-deploy")
# → Full SKILL.md loaded into context now
```

**The trade-off:**

| | Eager Loading (Pi) | On-Demand (Gemini) |
|---|---|---|
| Token cost at startup | High (all skills loaded) | Low (index only) |
| Token cost at use | None (already loaded) | Per-skill activation |
| Agent awareness | Agent knows all skills immediately | Agent must know to ask |
| Complexity | Simple | Requires skill index |

For personal use with a handful of skills, eager loading is fine. For enterprise use with a skills library, on-demand wins.

---

## Memory: Same Outcome, Different Ergonomics

**Pi / OpenClaw:** The agent writes to memory files directly.
```json
{
  "tool": "write",
  "path": "memory/2026-03-15.md",
  "content": "User prefers CSV exports. Hetzner DB is on cx21."
}
```

**Gemini CLI:** Dedicated `save_memory` tool.
```json
{
  "tool": "save_memory",
  "content": "User prefers CSV exports. Hetzner DB is on cx21."
}
```

Under the hood, `save_memory` writes to GEMINI.md or a designated memory file. Same outcome. The named tool makes it explicit in logs, enables hooks (`AfterTool: save_memory → audit_log`), and fits the policy engine.

## Full Feature Comparison Table

| Feature | Pi-mono | OpenClaw | Gemini CLI |
|---------|---------|----------|------------|
| Core loop | while + 4 tools | while + 4 tools + messaging | while + 17 tools + approval |
| Context files | AGENTS.md | AGENTS.md / SOUL.md / USER.md | GEMINI.md (3-level hierarchy) |
| Skills | Always loaded | Always loaded | On-demand (`activate_skill`) |
| Extensions | TypeScript via Jiti | TypeScript | MCP servers (any language) |
| Memory | `write()` to files | `write()` + `memory_search` | `save_memory` tool + GEMINI.md |
| Planning | Implicit in loop | Implicit in loop | Explicit Plan Mode (read-only) |
| Subagents | None | None | Built-in (isolated context loops) |
| Safety | Trust by default | Trust by default | Confirm by default + policy engine |
| Rollback | None | None | Git shadow repo checkpointing |
| Model routing | Single model | Single model | Pro for planning, Flash for execution |
| Hooks | None | None | 10+ lifecycle points |
| MCP | No | No | Yes (stdio + HTTP transport) |

---

## Key Observation

The columns don't represent three different approaches to building agents. They represent **three points on the same spectrum of production-readiness.**

Pi proves the concept. OpenClaw adds production features for real use. Gemini CLI adds enterprise features for teams, compliance, and scale.

The while loop at the center is identical in all three.
