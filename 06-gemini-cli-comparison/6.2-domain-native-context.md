# 6.2 The Secret Sauce: Domain-Native Context

**Why GCP extensions are Gemini's real advantage — and how your setup replicates it**

---

## What Makes Gemini CLI Different From "Just a CLI with AI"

The obvious answer is features: Plan Mode, subagents, checkpointing. But those are polish. The deeper answer is something most write-ups miss.

**Gemini CLI's GCP extensions don't just give the agent tools. They give it a pre-built understanding of an entire infrastructure domain.**

When you install the Cloud Run extension, the agent doesn't just get a tool called `cloud_run_deploy`. It gets:
- Knowledge of region codes (`us-central1`, `europe-west1`, etc.)
- Awareness of traffic splitting semantics (canary deploys, revision routing)
- Understanding of Cloud Run's resource model (services, revisions, jobs)
- Auth patterns (service accounts, ADC, IAM binding syntax)
- Common failure modes and how to diagnose them

The Cloud SQL extension similarly knows:
- Instance tiers and their tradeoffs
- Connection string formats (Unix socket vs. TCP, with Cloud SQL Auth Proxy)
- PostgreSQL-specific commands and pg_dump/pg_restore patterns
- Backup and maintenance window configuration

This isn't documentation that gets loaded into context. This is **baked into the model's weights** (Google trained Gemini on its own API surface) **and reinforced at runtime** by extension-provided GEMINI.md files with your specific project resources.

> **The phone book analogy:** Giving an agent `bash` access to `gcloud` is like handing someone a phone book. They can look things up if they know what to search for. Gemini with GCP extensions is like hiring someone who already speaks the language fluently and just needs your specific contacts.

---

## The "Baked-In Cloud" Concept

Think about what it would mean to have an AI that was:
1. Trained on the complete documentation of your cloud provider
2. Reinforced via RL on thousands of real workflows in that environment
3. Given runtime access to your specific resources via extensions

That's what Gemini has with GCP. It's home turf. The model doesn't need to derive how to deploy a Cloud Run service from first principles — it already knows.

**The Microsoft Azure equivalent would be:** an agent trained on all of Azure's ARM templates, CLI docs, and Bicep syntax, with extensions that inject your subscription's resource groups and service names. Every Azure command would just... work, because the agent already understands the environment at a deep level.

**The implication for you:** If you're using AWS/GCP/Azure, you're using an agent that has some of this implicit knowledge from training data. The more you use standard tooling and standard patterns, the more the agent's implicit knowledge helps you. The more you diverge from standard patterns, the more you need to fill in with explicit context.

## How Your OpenClaw Setup Is the Same Pattern

You have Hetzner servers and Supabase databases. You've given OpenClaw access via AGENTS.md and workspace instructions.

**That is domain-native context. The architecture is identical.**

```
Gemini CLI:
  Implicit knowledge (trained on GCP) + Explicit context (your project, via extension GEMINI.md)
  → Agent speaks GCP fluently for your specific project

Your OpenClaw setup:
  Implicit knowledge (Claude trained on Hetzner/Supabase docs) + Explicit context (your servers, via AGENTS.md)
  → Agent speaks your infrastructure fluently
```

The difference is in **depth and polish**:

| | Gemini + GCP | OpenClaw + Hetzner/Supabase |
|---|---|---|
| Implicit model knowledge | Deep (Google trained on own APIs) | Moderate (Claude knows general cloud patterns) |
| Explicit runtime context | Extension GEMINI.md (structured, versioned) | AGENTS.md (your own writing) |
| Available scripts/tools | Extension-provided MCP tools | Your own bash scripts |
| Maintenance | Google maintains extensions | You maintain AGENTS.md |

You don't have Google's training advantage. But you have the same architectural pattern — and you have the advantage of knowing your specific infrastructure better than any extension author.

## The Two Components of Domain-Native Context

Every domain-native context system has the same two layers:

### 1. Implicit Knowledge (trained into the model)

What the model already knows without being told. For Gemini + GCP: service names, API shapes, auth patterns, error codes, best practices. For Claude + your stack: general Linux administration, PostgreSQL syntax, REST API conventions, bash scripting patterns.

**You cannot change this.** You can only choose tools and frameworks that align with what the model was trained on. Standard tools (PostgreSQL, nginx, systemd) get better implicit support than niche tools.

### 2. Explicit Context (what you inject at runtime)

What you tell the agent about your specific environment. This is your AGENTS.md, your skill files, your pre-built scripts.

**You control this entirely.** This is your leverage point.

```
Implicit knowledge:  Claude knows PostgreSQL syntax generally
Explicit context:    Your AGENTS.md says "claims table has columns:
                     claim_id, npi, amount, status, submitted_date"

Result: Agent can write queries against your exact schema
        without trial and error
```

## How to Maximize Your Own Domain-Native Context

### In AGENTS.md

Document everything the agent would otherwise have to discover:

```markdown
## Infrastructure

### Hetzner Servers
- prod-db: cx21, Frankfurt DC4, 10.0.0.2 (private)
- prod-app: cx31, Frankfurt DC4, 10.0.0.3 (private)
- bastion: cx11, Frankfurt DC4, 123.45.67.89 (public)
- SSH: `ssh -J ubuntu@123.45.67.89 ubuntu@10.0.0.2`

### Supabase
- Project: acme-prod
- Region: us-east-1
- Connection: $SUPABASE_DB_URL (in .env)
- Direct connection for bulk ops: $SUPABASE_DIRECT_URL

## Database Schema (claims)

### claims table
| Column | Type | Notes |
|--------|------|-------|
| claim_id | uuid | PK |
| npi | varchar(10) | Provider NPI |
| amount | numeric(10,2) | Billed amount |
| status | enum | pending/approved/denied/paid |
| submitted_date | date | |
| payer_id | uuid | FK → payers.id |

### Available Scripts
- `scripts/query-claims.sh <status> <date-from> <date-to>` — paginated claims query
- `scripts/export-csv.sh <table> <where-clause>` — exports to ~/exports/
- `scripts/npi-lookup.sh <npi>` — NPPES registry lookup
```

Every line here is context the agent doesn't have to derive. Every derivation it skips is a turn saved, a mistake avoided.

### In Skills

Create skills for recurring workflows — not just capability documentation, but pre-built scripts the agent can call:

```
skills/
├── supabase-query/
│   ├── SKILL.md          # How to use, what parameters
│   └── scripts/
│       ├── run-query.sh  # Executes $1 as SQL, returns CSV
│       └── bulk-export.sh
├── hetzner-ops/
│   ├── SKILL.md
│   └── scripts/
│       ├── deploy.sh     # Deploy to prod-app
│       └── db-backup.sh  # Snapshot prod-db
```

The agent doesn't need to re-derive "how do I connect to Supabase?" on every request. It calls `run-query.sh` with a SQL string.

### The Explicit Context Investment Payoff

```
Low investment (AGENTS.md = nothing):
  Agent figuring out your schema → 4-6 turns of exploration → occasionally wrong

Medium investment (basic AGENTS.md):
  Agent reads schema section → writes correct query first try → 1-2 turns

High investment (AGENTS.md + scripts + skills):
  Agent calls pre-built script with parameters → 1 turn → done
```

The GCP extensions are Google's high-investment version of this. Your AGENTS.md is yours.

## Practical Example: A Well-Crafted AGENTS.md for Enterprise Reporting

This is what the "high investment" version looks like for a healthcare reporting use case:

```markdown
# Agent Instructions

## Identity
You are a reporting assistant for Acme Health. You help generate
claims reports, provider analytics, and payer summaries.

## Infrastructure

### Database
- Engine: PostgreSQL 15 via Supabase
- Connection: use $SUPABASE_DB_URL from environment
- For bulk operations (>10k rows): use $SUPABASE_DIRECT_URL
- Never modify production data. Read-only except staging schema.

### Reporting Output
- All reports go to ~/workspace/reports/
- Filename convention: YYYY-MM-DD_<type>_<scope>.{csv,md,html}
- Always generate both CSV (for data) and MD (for human review)

## Schema Reference

### claims
Primary claims table. Never JOIN without claim_id index.
```sql
-- Always filter by date range first (has index on submitted_date)
SELECT * FROM claims
WHERE submitted_date BETWEEN '2026-01-01' AND '2026-03-31'
  AND status = 'approved';
```

Key columns: claim_id, npi, amount, status, submitted_date, payer_id

### providers
NPI registry mirror. Updated nightly.
Key columns: npi, name, specialty, state, tin

### payers
Insurance payer directory.
Key columns: payer_id, name, payer_type (commercial/medicare/medicaid)

## Available Scripts

| Script | Purpose | Usage |
|--------|---------|-------|
| `scripts/claims-summary.sh` | Monthly claims summary by payer | `./claims-summary.sh 2026-01` |
| `scripts/npi-lookup.sh` | NPPES registry lookup | `./npi-lookup.sh 1234567890` |
| `scripts/export-csv.sh` | Export query to CSV | `./export-csv.sh "SELECT..."` |
| `scripts/pdf-report.sh` | Convert MD report to PDF | `./pdf-report.sh report.md` |

## Naming Conventions
- Payer codes: CMS = Medicare, MCD = Medicaid, prefix = commercial
- NPI: always 10 digits, no dashes
- Amounts: always in dollars (not cents), 2 decimal places

## Constraints
- Never query without a date range filter (table has 50M+ rows)
- Never expose NPI + amount + name in the same CSV without redaction flag
- Always use parameterized queries via scripts, not inline SQL
```

This AGENTS.md is doing what the Cloud Run extension does for GCP: giving the agent pre-wired knowledge of the domain so it can act without guessing.
