# 1.3 Pipes, Redirects, and Composition

**The most important concept for understanding how agents work.**

---

## The Power of Pipes

A **pipe** (`|`) connects the output of one command to the input of the next.

### Simple Example

```bash
# Without pipe (manual)
ls > files.txt
cat files.txt | wc -l
rm files.txt

# With pipe (automatic)
ls | wc -l
```

**Result:** Count files in directory (one command, no temp file).

---

## How Pipes Work

```
Command 1    |    Command 2    |    Command 3
  output  ──────>  input  ──────>  input
```

**Example:**
```bash
cat users.csv | grep "active" | wc -l
```

**Flow:**
1. `cat users.csv` → outputs all lines
2. `grep "active"` → filters lines containing "active"
3. `wc -l` → counts remaining lines

**Result:** "How many active users?"

---

## Why Agents Love Pipes

### 1. **Composability**

Small tools → complex workflows:

```bash
# Each step does ONE thing well
cat data.json \
  | jq '.users[]' \
  | grep '"email"' \
  | sed 's/.*: "//;s/".*//' \
  | sort \
  | uniq
```

**Result:** List of unique user emails (from JSON).

---

### 2. **No Intermediate Files**

```bash
# Bad: creates temp files
cat file1.txt > temp1
grep "error" temp1 > temp2
sort temp2 > result.txt
rm temp1 temp2

# Good: pipes (no cleanup needed)
cat file1.txt | grep "error" | sort > result.txt
```

---

### 3. **Stream Processing**

Pipes process data **as it flows**, not all at once:

```bash
# Processes 1GB file without loading it all into memory
cat huge-logfile.log | grep "ERROR" | head -10
```

## Redirects: Saving Output

### Output Redirect (`>` and `>>`)

```bash
# Overwrite file
echo "Hello" > file.txt

# Append to file
echo "World" >> file.txt

# Save command output
ls -la > directory-listing.txt
```

---

### Input Redirect (`<`)

```bash
# Read file as input
sort < unsorted.txt

# Same as:
cat unsorted.txt | sort
```

---

### Error Redirect (`2>`)

```bash
# Redirect errors to file
command 2> errors.txt

# Redirect errors to /dev/null (discard)
command 2> /dev/null

# Redirect both output and errors
command > output.txt 2>&1
```

## Real-World Agent Patterns

### Pattern 1: Extract → Transform → Load

```bash
# Extract data from API
curl https://api.example.com/users \
  | jq '.[] | {name, email}' \
  | while read line; do
      echo "$line" >> users.txt
    done
```

---

### Pattern 2: Filter → Aggregate → Report

```bash
# Analyze logs
cat /var/log/app.log \
  | grep "ERROR" \
  | awk '{print $4}' \
  | sort | uniq -c \
  | sort -rn \
  > error-summary.txt
```

**Result:** Most common errors, sorted by frequency.

---

### Pattern 3: Multi-Source → Combine → Process

```bash
# Combine data from multiple sources
cat data1.csv data2.csv data3.csv \
  | grep -v "^#" \
  | sort -t',' -k3 -n \
  > combined-sorted.csv
```

## Advanced: Command Substitution

### Using `$()` to capture output

```bash
# Capture command output as variable
file_count=$(ls | wc -l)
echo "Found $file_count files"

# Use in another command
echo "Report generated at $(date)" > report.txt
```

---

### Inline substitution

```bash
# Process files matching criteria
cat $(find . -name "*.log") | grep "ERROR"

# Same as:
find . -name "*.log" -exec cat {} + | grep "ERROR"
```

## Building Complex Workflows

### Example: Podcast Summarization

```bash
#!/bin/bash
# summarize-podcast.sh

# 1. Download audio
curl -L "$PODCAST_URL" -o podcast.mp3

# 2. Transcribe
whisper-cli -m model.bin -f podcast.mp3 -otxt -of transcript

# 3. Clean transcript
cat transcript.txt \
  | sed 's/\[.*\]//g' \
  | sed '/^$/d' \
  > clean-transcript.txt

# 4. Summarize (using LLM)
cat clean-transcript.txt \
  | llm "Summarize the key technical insights" \
  > summary.md

# 5. Generate HTML report
cat summary.md \
  | pandoc -f markdown -t html \
  --css style.css \
  > summary.html

echo "Done! See summary.html"
```

**Each step:** Read → Process → Output (via pipes)

## Debugging Pipelines

### Technique 1: Build incrementally

```bash
# Start simple
cat data.json

# Add one step
cat data.json | jq '.'

# Add another
cat data.json | jq '.users[]'

# Keep building
cat data.json | jq '.users[]' | grep "active"
```

---

### Technique 2: Use `tee` to inspect midstream

```bash
# Save intermediate output while continuing pipeline
cat data.json \
  | jq '.users[]' \
  | tee intermediate.json \
  | grep "active"
```

Now you can inspect `intermediate.json` to debug.

---

### Technique 3: Check exit codes

```bash
cat file.txt | grep "pattern" | sort
echo $?  # 0 = success, non-zero = error
```

## Common Pitfalls

### ❌ Piping to `cd` doesn't work

```bash
# WRONG
ls /path/to/dir | cd   # cd doesn't read stdin

# RIGHT
cd /path/to/dir && ls
```

---

### ❌ Forgetting quotes with variables

```bash
# WRONG (breaks with spaces in filenames)
cat $filename | grep "text"

# RIGHT
cat "$filename" | grep "text"
```

---

### ❌ Overwriting input file

```bash
# WRONG (empties file before reading)
cat file.txt | sort > file.txt

# RIGHT (use temp file or sponge)
cat file.txt | sort > temp && mv temp file.txt
# or
cat file.txt | sort | sponge file.txt  # requires moreutils
```

## Practice Challenges

### Challenge 1: Log Analysis
**Task:** Find the 10 most common IP addresses in an Apache access log.

<details>
<summary>Solution</summary>

```bash
cat access.log \
  | awk '{print $1}' \
  | sort \
  | uniq -c \
  | sort -rn \
  | head -10
```
</details>

---

### Challenge 2: CSV Processing
**Task:** Extract emails from CSV, remove duplicates, save sorted.

<details>
<summary>Solution</summary>

```bash
cat users.csv \
  | cut -d',' -f3 \
  | tail -n +2 \
  | sort \
  | uniq \
  > emails.txt
```
</details>

---

### Challenge 3: JSON to Markdown
**Task:** Convert JSON user list to markdown table.

<details>
<summary>Solution</summary>

```bash
cat users.json \
  | jq -r '.[] | "| \(.name) | \(.email) |"' \
  | sed '1i| Name | Email |' \
  | sed '2i|------|-------|' \
  > users.md
```
</details>

## Key Takeaways

> **"Small tools, composed well, solve big problems."**  
> — Unix philosophy

**For agents:**
1. **Build workflows from primitives** (not monolithic tools)
2. **Use pipes to chain operations** (no temp files)
3. **Each command does ONE thing** (composability)
4. **Bash is glue** between specialized tools
