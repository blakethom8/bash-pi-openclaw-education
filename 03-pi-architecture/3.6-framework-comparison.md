# 3.6 Pi vs Claude Code vs Cursor

**Choosing the right architecture for the job**

---

## The Landscape

In 2025–2026, several coding agent frameworks have matured:

- **Pi** (Armin Ronacher) — minimal, 4-tool, bash-first
- **Claude Code** (Anthropic) — ~20 specialized tools, polished coding UX
- **Cursor** (Cursor Inc.) — ~30 tools, IDE-integrated
- **Devin, SWE-agent, AutoGPT** — various multi-agent orchestration approaches

They're all LLM-in-a-loop systems. They differ on **how many tools**, **how opinionated**, and **who controls the workflow**.

---

## The Comparison Table

| Feature | Pi | Claude Code | Cursor |
|---------|----|-----------|----|
| **Core tools** | 4 (read/write/edit/bash) | ~20 specialized | ~30 specialized |
| **System prompt size** | ~1,000 tokens | ~5,000 tokens | ~10,000 tokens |
| **Extensibility** | Infinite (via bash) | Limited to built-in | Limited to extensions |
| **Self-modifying** | Yes (agent creates scripts/skills) | No | No |
| **Workflow** | You define it | Opinionated (git-centric) | Opinionated (IDE-centric) |
| **Setup complexity** | Minimal | Medium | High (IDE integration) |
| **Custom integrations** | Trivial (bash script) | Requires plugin system | Requires extension API |
| **Transparency** | High (you see every tool call) | Medium | Low (IDE-managed) |
| **Best for** | Custom workflows, enterprise | Coding projects | IDE-centric coding |

---

## Why Fewer Tools = More Flexible

This seems backwards. Surely more tools = more capable?

Not for enterprise workflows. Here's why:

### The specialized tools problem

Claude Code has a `search_files` tool. It works great for code search. But what if you need to:
- Search a PostgreSQL database?
- Search a Redis cache?
- Search a proprietary internal API?
- Search your Confluence wiki?

Claude Code has none of these. You're blocked unless the framework adds them — and they won't add your company's internal tools.

### The bash escape hatch

Pi has `bash`. You need to search Confluence?

```bash
bash("curl -s 'https://confluence.example.com/rest/api/content/search?cql=text~\"deployment+procedure\"' \
  -H 'Authorization: Bearer $CONFLUENCE_TOKEN' \
  | jq '.results[].title'")
```

Done. No framework changes needed. No extension API. Just bash.

> "The bash tool means Pi never needs a 5th tool. Whatever you can think of, bash can do it — or you can write a script that does it."

## The Specialization Trade-off

Specialized tools aren't without advantages:

**Where specialized tools win:**
- **UX polish:** Claude Code's diff display is prettier than raw unified diff
- **Safety:** A `git_commit` tool can enforce commit message format; `bash("git commit -m ...")` cannot
- **Discoverability:** A list of 20 named tools is easier to explore than "figure out the right bash command"
- **Reliability:** A dedicated `read_file` tool handles edge cases (encoding, permissions) better than a raw bash cat

**Where bash wins:**
- **Custom integrations:** Your internal systems, databases, APIs
- **Novel tasks:** Things the framework designers didn't anticipate
- **Composition:** Chain multiple tools together in a single command
- **Power user workflows:** If you know bash, you're never blocked

The Pi philosophy: trust the user to know bash. Optimize for power, not discoverability.

## System Prompt Size Matters

The system prompt is injected into every API call. It costs tokens every turn.

| Framework | System Prompt | Cost per turn |
|-----------|---------------|---------------|
| Pi | ~1,000 tokens | Low |
| Claude Code | ~5,000 tokens | Medium |
| Cursor | ~10,000 tokens | High |

At Claude API pricing (~$3/M tokens for input with Sonnet), a 100-turn session costs:

- Pi: ~$0.30 in system prompt tokens
- Claude Code: ~$1.50
- Cursor: ~$3.00

For high-volume enterprise use (running agents on thousands of files, nightly pipelines), this adds up. Pi's minimal prompt is a real advantage.

More importantly: smaller system prompts leave more room in the context window for actual work — file contents, command outputs, history.

## Self-Modification: Pi's Most Unusual Feature

Claude Code and Cursor do not modify themselves. Their tools are fixed at release.

Pi does. Because Pi's "tools" are just bash scripts, the agent can:

1. Write a new skill that codifies a procedure
2. Modify an existing skill when it discovers a better approach
3. Create project-specific tools as bash scripts
4. Update `AGENTS.md` with new domain knowledge it learned

This means a Pi agent gets better at your specific codebase over time. It learns your naming conventions, your data schemas, your deploy procedures — and writes that knowledge to files that persist across sessions.

```bash
# Agent writes this after learning your database schema
write(".pi/skills/query-database.md", "...")

# Agent updates preferences after you correct it
edit("memory/preferences.md", ...)

# Agent creates a project-specific utility
write("scripts/reset-test-db.sh", "...")
```

Claude Code and Cursor are static. Pi is adaptive.

## When to Use Each

### Use Pi when:

- **You need custom integrations:** Your company has internal APIs, databases, or tools that no framework has built-in support for
- **You're building data pipelines:** Moving data between systems, processing files, running reports on a schedule
- **You want full control:** You want to understand exactly what the agent is doing and why
- **You're an enterprise developer:** You need to integrate with existing infrastructure (JIRA, Confluence, custom APIs, internal databases)
- **You're building agent infrastructure:** Pi's simplicity makes it easy to embed in larger systems
- **You value transparency:** Every action is a visible tool call; no magic

### Use Claude Code when:

- **You want a polished coding experience:** It handles git, tests, PRs with good defaults
- **You're doing standard software development:** The built-in tools cover 90% of common coding tasks
- **You don't need custom integrations:** Standard dev workflow, public APIs, common languages
- **Discoverability matters:** Good for developers new to AI-assisted coding
- **You want safety guardrails:** Claude Code adds confirmation steps for destructive operations

### Use Cursor when:

- **IDE integration is essential:** You live in VS Code and want the agent there
- **You want inline suggestions:** Cursor's autocomplete is best-in-class
- **Your workflow is file-centric:** Open files, make edits, see results instantly

## Extensions vs Built-in Tools

Pi has a two-layer extensibility model:

```
Layer 1: Core tools (read/write/edit/bash)
         — always available, handle 90% of cases

Layer 2: Extensions (new tool implementations)
         — for capabilities bash genuinely can't provide
```

**Examples of legitimate extensions:**
- Web browser automation (headless Chrome via Playwright)
- Real-time audio/video input
- Hardware interfaces (GPIO on Raspberry Pi)
- Streaming large files without token overhead

**Not legitimate extensions (use bash instead):**
- Database queries → bash + psql/sqlite3
- HTTP requests → bash + curl
- File format conversion → bash + appropriate CLI tool
- Sending notifications → bash + curl to webhook

The question is: can bash do this? If yes, use bash. If bash genuinely can't (real-time streams, hardware, browser automation), consider an extension.

---

## The Enterprise Case for Pi

Enterprise environments often have:
- Proprietary internal APIs with non-standard authentication
- Legacy systems accessible only via CLI tools or internal SDKs
- Custom data formats and schemas
- Compliance requirements that require audit trails
- Workflows that don't map to "write code, run tests, commit"

Claude Code and Cursor were designed for public-facing software development workflows. Pi was designed for "whatever you need an agent to do."

For an insurance company running claims reports:
- Claude Code doesn't have a `query_claims_database` tool
- Cursor doesn't know your policy schema
- Pi can learn both via AGENTS.md and a query-database skill, in an afternoon

## Practice: Design Your Tool Set

Think about your actual workflow. Pick one of these scenarios (or your own):

1. Daily data pipeline: pull data from three sources, transform, load to warehouse, send summary email
2. Code review assistant: check PRs against your team's style guide, post comments
3. Customer support triage: read support tickets, categorize, route to right team

**Exercise:** List the tools you'd need. Then for each tool, determine: can bash do this? If yes, how? If no, what extension would you need?

<details>
<summary>Example: Data pipeline</summary>

**Tools needed:**
- Connect to Source A (internal PostgreSQL) → `bash` + psql
- Connect to Source B (REST API with OAuth) → `bash` + curl with token refresh
- Connect to Source C (SFTP file drop) → `bash` + sftp or scp
- Transform data (join, aggregate) → `bash` + Python script
- Load to warehouse (BigQuery) → `bash` + bq CLI or Python
- Send summary email → `bash` + curl to SendGrid API or `mail` command

**Extensions needed:** None. Bash handles all of it.

**Pi skill to write:** `daily-pipeline.md` — documents the sources, schemas, credentials pattern, and step sequence. Agent reads the skill, executes the pipeline, writes a summary to memory.

</details>
