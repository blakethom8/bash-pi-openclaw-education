# 3.4 The Skills System

**How Pi extends itself without adding tools**

---

## The Problem with Tool Proliferation

Every specialized agent framework faces the same pressure: "I need to send email, so add a `send_email` tool. I need to query Postgres, so add a `query_postgres` tool. I need to create Jira tickets…"

Six months later you have 47 tools, a 15,000-token system prompt, and a model that's confused about which tool to use when.

Pi's answer: **don't add tools. Add skills.**

> "Skills are instructions, not code. The agent already has bash. Skills just tell it what to do with bash."

---

## What a Skill Is

A skill is a **Markdown file with YAML frontmatter**. Not Python. Not TypeScript. Markdown.

### Structure

```markdown
---
name: generate-report
description: Generate a formatted markdown report from JSON data sources. Use when asked to create reports, summaries, or dashboards from data files.
---

# Generate Report Skill

## Overview
This skill creates formatted markdown reports from structured data.

## Steps
1. Read the source data file (JSON or CSV)
2. Identify key metrics
3. Create a markdown report with:
   - Executive summary (3-5 bullet points)
   - Data table
   - Trend analysis if date fields are present

## Output format
Save to `reports/YYYY-MM-DD-{name}.md`

## Example
```bash
# Read data
cat data/claims.json | python3 -c "
import json, sys
data = json.load(sys.stdin)
print(f'Total claims: {len(data[\"claims\"])}')
# ... etc
"
```
```

That's it. The skill is documentation. The agent reads it and follows it.

---

## Where Skills Live

```
~/.pi/agent/skills/       # Global skills (all projects)
.pi/skills/               # Project-local skills (this repo only)
```

**Global skills** are things you always want: your preferred coding style, how you like reports formatted, company-specific procedures.

**Project skills** are domain-specific: how to query this particular database, the schema for this system's JSON files, the deploy procedure for this service.

### File naming

```
.pi/skills/
├── generate-report.md
├── query-database.md
├── deploy-service.md
└── create-jira-ticket.md
```

One skill per file. Filename matches the `name` frontmatter field by convention.

---

## How the Agent Discovers Skills

Skills are not pre-loaded into the context. The process:

1. **Discovery:** At startup, Pi reads the `description` field from all skill files and includes them in the system prompt — one line per skill:
   ```
   Available skills:
   - generate-report: Generate a formatted markdown report from JSON data sources. Use when asked to create reports, summaries, or dashboards from data files.
   - query-database: Query the claims database. Use when asked about claims data, policy counts, or anything requiring database access.
   ```

2. **Decision:** When the user makes a request, the LLM decides whether any skill is relevant — based on the description alone.

3. **Loading:** If relevant, the agent calls `read(".pi/skills/generate-report.md")` to load the full skill content into context.

4. **Execution:** The agent follows the instructions in the skill, using the four tools.

**This means your skill description is critical.** The agent decides whether to load the full skill based only on the description. A vague description means the skill never gets used.

### Bad description:
```yaml
description: Report generation
```

### Good description:
```yaml
description: Generate a formatted markdown report from JSON data sources. Use when asked to create reports, summaries, or dashboards from data files.
```

The good description tells the agent: what it does, what inputs it expects, and when to use it.

## Why Markdown Beats Code

You could imagine skills as Python modules, TypeScript plugins, or bash scripts. Why Markdown?

| | Markdown Skills | Code Plugins |
|--|-----------------|-------------|
| Compilation needed | No | Yes |
| Human-readable | Yes | Depends |
| Self-documenting | Yes | Requires separate docs |
| Agent can modify | Yes (write tool) | Risky |
| Non-developer can write | Yes | No |
| Version control diff | Clean | Code diffs |

The killer advantage: **the agent itself can write and modify skills**.

If you ask Pi "remember how to generate expense reports my way," it can:
1. Write a new skill file documenting your preferred format
2. Save it to `.pi/skills/generate-expense-report.md`
3. Next time you ask, it loads and follows the skill

The system is self-extending.

## Example Skill: Generate Report

```markdown
---
name: generate-report
description: Generate a formatted markdown report from JSON or CSV data. Use when asked to create reports, summaries, weekly digests, or data dashboards.
---

# Generate Report

## Input
A data file path (JSON or CSV). If not provided, look for recent files in `data/` or `reports/input/`.

## Output
A markdown file at `reports/YYYY-MM-DD-{slug}.md`

## Steps

### 1. Read and understand the data
```bash
# For JSON
cat {file} | python3 -m json.tool | head -40

# For CSV
head -5 {file}
wc -l {file}
```

### 2. Compute key metrics
```bash
python3 << 'EOF'
import json
with open('{file}') as f:
    data = json.load(f)

# Adapt to actual schema
total = len(data)
# ... compute what makes sense
EOF
```

### 3. Write the report
Structure:
- `# Title` — descriptive, includes date
- `## Executive Summary` — 3-5 bullets, key numbers
- `## Detail Table` — markdown table of top records
- `## Notes` — data quality issues, caveats

### 4. Verify
Read the output file. Check that numbers make sense.
```

## Example Skill: Query Database

```markdown
---
name: query-database
description: Query the claims PostgreSQL database. Use when asked about claim counts, policy data, adjuster performance, or any question requiring database access.
---

# Query Database

## Connection
```bash
export PGPASSWORD=$DB_PASSWORD
psql -h $DB_HOST -U $DB_USER -d claims_db
```

## Common queries

### Claims by status
```sql
SELECT status, count(*)
FROM claims
GROUP BY status
ORDER BY count DESC;
```

### Recent claims
```sql
SELECT claim_id, policy_id, amount, status, created_at
FROM claims
WHERE created_at > NOW() - INTERVAL '7 days'
ORDER BY created_at DESC
LIMIT 20;
```

## Output
Format query results as a markdown table. Include the SQL used.

## Notes
- `claims` table has ~2M rows — always use LIMIT unless aggregating
- `policy_id` is a UUID, not an integer
- `amount` is in cents (divide by 100 for dollars)
- Read replica is at $DB_HOST_RO — use for SELECT queries
```

## The `disable-model-invocation` Option

Skills can include a frontmatter option to change how they're executed:

```yaml
---
name: run-deploy
description: Deploy the service to production. Use when asked to deploy, release, or push to production.
disable-model-invocation: true
---
```

When `disable-model-invocation: true` is set, the skill runs as a **direct bash script** rather than being interpreted by the LLM. This is useful for:

- **Deterministic operations:** Deployments, database migrations — you want exactly these commands, every time, with no LLM improvisation
- **Security:** Prevents the model from "helpfully" modifying the deploy steps
- **Reliability:** For production operations where consistency matters more than flexibility

Without this flag (the default), the model reads the skill and interprets it — it can adapt the steps to the situation, handle edge cases, etc.

## Skills vs Extensions

There are two ways to extend Pi:

| | Skills | Extensions |
|--|--------|-----------|
| **What they are** | Markdown instruction files | Code that adds new tools |
| **What they provide** | Instructions for using existing tools | New tool implementations |
| **Example** | "Here's how to query our database" | A new `web_search` tool backed by an API |
| **Written by** | Anyone (even the agent itself) | Developers |
| **Runtime** | Agent reads and follows instructions | New tool available in every turn |
| **When to use** | Domain knowledge, procedures, conventions | New capabilities bash can't easily provide |

**Most things should be skills, not extensions.** If bash can do it (and bash can do almost everything), a skill is the right answer. Extensions are for when you need a new, first-class tool — like adding streaming audio input or integrating a specialized SDK.

---

## Writing a Useful Skill

Checklist:

- [ ] **Name** is a short slug (`generate-report`, not `GenerateReportSkill`)
- [ ] **Description** includes: what it does, when to use it, what inputs it needs
- [ ] **Steps are concrete:** Include actual bash commands or code, not just descriptions
- [ ] **Schema documentation:** If it touches a database or file format, document the schema
- [ ] **Error handling:** Note common failure modes and how to handle them
- [ ] **Output format:** Specify exactly what the skill should produce

## Practice: Write a Skill

Write a skill for this use case:

> Your team runs a weekly expense report. It reads from `data/expenses.json` (schema: `[{date, amount_cents, category, description, employee_id}]`), computes totals by category, and saves a formatted markdown report.

<details>
<summary>Example answer</summary>

```markdown
---
name: weekly-expense-report
description: Generate a weekly expense report from expense data. Use when asked to create expense reports, summarize expenses, or analyze spending by category. Reads from data/expenses.json.
---

# Weekly Expense Report

## Input
`data/expenses.json` — array of expense objects with fields:
- `date`: ISO date string
- `amount_cents`: integer (divide by 100 for dollars)
- `category`: string (Travel, Software, Office, Meals, etc.)
- `description`: string
- `employee_id`: string

## Steps

### 1. Compute totals
```bash
python3 << 'EOF'
import json
from collections import defaultdict
from datetime import datetime

with open('data/expenses.json') as f:
    expenses = json.load(f)

by_category = defaultdict(int)
total = 0
for e in expenses:
    by_category[e['category']] += e['amount_cents']
    total += e['amount_cents']

print(f"Total: ${total/100:.2f}")
for cat, amt in sorted(by_category.items(), key=lambda x: -x[1]):
    print(f"  {cat}: ${amt/100:.2f}")
EOF
```

### 2. Write the report
Save to `reports/YYYY-MM-DD-expenses.md` with:
- Total spend
- Breakdown table by category
- Top 5 largest individual expenses

## Output format
```markdown
# Weekly Expense Report — YYYY-MM-DD

**Total:** $X,XXX.XX

## By Category
| Category | Amount |
|----------|--------|
| Travel   | $X,XXX |
...

## Top Expenses
| Date | Employee | Category | Amount | Description |
...
```
```
