# 3.2 The Four Tools (and why that's enough)

**Deep dive into read, write, edit, and bash**

---

## Why Only Four?

Most agent frameworks ship 20–50 specialized tools: `search_web`, `send_email`, `query_database`, `read_github_issue`, `create_jira_ticket`…

Pi ships four. This is deliberate.

> "The bash tool is the escape hatch for everything else. You don't need a `send_email` tool when you can write a bash script that sends email."

The four tools aren't a limitation. They're a foundation. Everything else is composition.

---

## Tool 1: `read`

**Purpose:** Read file contents into the LLM's context.

### Signature

```json
{
  "tool": "read",
  "path": "/absolute/or/relative/path"
}
```

### What it actually does

- Reads the file and returns its contents as a string
- For directories: returns a directory listing (like `ls -la`)
- Paginates large files: defaults to first **80 lines** or **100KB**, whichever comes first
- For images (PNG, JPG, etc.): decodes and sends as base64 to multimodal models
- Returns an error string if the file doesn't exist — the agent sees the error and can react

### Pagination

When a file exceeds the limit, the agent sees something like:

```
[File truncated at 80 lines. 1,240 lines remaining. Use offset=80 to continue.]
```

The agent then issues follow-up reads:

```json
{ "tool": "read", "path": "big-file.log", "offset": 80, "limit": 80 }
{ "tool": "read", "path": "big-file.log", "offset": 160, "limit": 80 }
```

### Real usage examples

```json
// Read a source file
{ "tool": "read", "path": "src/main.py" }

// Read a config
{ "tool": "read", "path": "config/settings.json" }

// List a directory
{ "tool": "read", "path": "src/" }

// Read a PNG screenshot
{ "tool": "read", "path": "screenshots/error.png" }

// Read with pagination
{ "tool": "read", "path": "logs/app.log", "offset": 200, "limit": 50 }
```

### Common pitfalls

- **Forgetting to paginate:** If the agent reads a 10,000-line file and only gets 80 lines, it may miss the relevant section. The LLM needs to recognize the truncation notice and continue reading.
- **Directory vs file:** Reading a directory path returns a listing, not an error. The agent uses this to explore.
- **Binary files:** Reading a non-image binary returns garbage. Use `bash` with `xxd` or `file` to inspect binaries.

## Tool 2: `write`

**Purpose:** Create or fully overwrite a file.

### Signature

```json
{
  "tool": "write",
  "path": "/path/to/file",
  "content": "full file contents here"
}
```

### What it actually does

- Creates the file if it doesn't exist
- **Overwrites completely** if it does exist — no merging, no diffing
- Auto-creates parent directories (no need to `mkdir -p` first)
- Returns the number of bytes written on success

### Real usage examples

```json
// Create a new Python script
{
  "tool": "write",
  "path": "scripts/process-claims.py",
  "content": "#!/usr/bin/env python3\nimport json\nimport sys\n\ndef main():\n    data = json.load(sys.stdin)\n    # ... process ...\n\nif __name__ == '__main__':\n    main()\n"
}

// Write a config file
{
  "tool": "write",
  "path": "config/database.json",
  "content": "{\n  \"host\": \"localhost\",\n  \"port\": 5432,\n  \"name\": \"claims_db\"\n}\n"
}

// Create a markdown report (deep directory — auto-created)
{
  "tool": "write",
  "path": "reports/2026/march/weekly-summary.md",
  "content": "# Weekly Summary\n\n..."
}
```

### Common pitfalls

- **Don't use write to edit:** If you write a 500-line file when you only needed to change one line, you've sent 500 lines of tokens for no reason. Use `edit` instead.
- **Overwrites silently:** The tool does not warn you before overwriting. The LLM needs to check if a file exists first (via `read`) if preservation matters.
- **Encoding:** Content is plain text. For binary output, use `bash` with appropriate commands.

## Tool 3: `edit`

**Purpose:** Surgical find-and-replace on an existing file.

### Signature

```json
{
  "tool": "edit",
  "path": "/path/to/file",
  "old_text": "exact text to find",
  "new_text": "replacement text"
}
```

### What it actually does

- Finds `old_text` in the file — **exact match, including whitespace and newlines**
- Replaces it with `new_text`
- **Requires the match to be unique** — if `old_text` appears more than once, the tool returns an error
- Returns a unified diff of the change on success

Example diff output:

```diff
--- a/src/config.py
+++ b/src/config.py
@@ -12,7 +12,7 @@
 DATABASE_CONFIG = {
-    "host": "localhost",
+    "host": "prod-db.internal",
     "port": 5432,
 }
```

### Why edit, not write?

If you need to change one line in a 1,000-line file:

- **write:** Send all 1,000 lines through the API. Expensive, error-prone.
- **edit:** Send just the old text and new text. Cheap, surgical.

This is not a full rewrite. It's a diff.

### Real usage examples

```json
// Fix a bug: wrong variable name
{
  "tool": "edit",
  "path": "src/processor.py",
  "old_text": "    result = proccess_claims(data)",
  "new_text": "    result = process_claims(data)"
}

// Update a config value
{
  "tool": "edit",
  "path": "config/settings.json",
  "old_text": "  \"timeout\": 30,",
  "new_text": "  \"timeout\": 60,"
}

// Add an import at the top of a file
{
  "tool": "edit",
  "path": "src/main.py",
  "old_text": "import json\nimport sys",
  "new_text": "import json\nimport sys\nimport logging"
}
```

### Common pitfalls

- **Non-unique match:** If `old_text` appears twice, the tool errors. Solution: include more surrounding context to make it unique.

  ```json
  // FAILS: "timeout" appears in multiple places
  { "old_text": "timeout = 30" }

  // WORKS: include surrounding context
  { "old_text": "# Database connection settings\ntimeout = 30" }
  ```

- **Whitespace mismatch:** Tabs vs spaces, trailing spaces — these must match exactly. If the edit fails, the agent should re-read the file to verify the exact whitespace.

- **Using edit on a non-existent file:** Returns an error. Use `write` to create new files.

- **Large replacements:** Edit can replace multiple lines at once. Include the full block in `old_text` if you're replacing a function body.

## Tool 4: `bash`

**Purpose:** Execute any shell command. This is the escape hatch.

### Signature

```json
{
  "tool": "bash",
  "command": "shell command here"
}
```

### What it actually does

- Runs the command in a shell subprocess
- Captures both stdout and stderr
- Returns exit code + output
- **Timeout:** Commands that run longer than the configured timeout (typically 30–120s) are killed
- **Output truncation:** Output is capped at **last 100 lines** or **30KB**, whichever comes first — tail, not head, so you see recent output
- Streams output to the terminal in real-time (you see it happening), but only the truncated result goes back to the LLM

### Output format

The agent sees:

```
Exit code: 0
Output:
[stdout/stderr content, last 100 lines or 30KB]
```

Or on failure:

```
Exit code: 1
Output:
Error: connection refused to prod-db.internal:5432
```

### Why bash is the escape hatch

Everything you can't do with read/write/edit, you can do with bash:

| Need | Bash command |
|------|-------------|
| Find files | `find . -name "*.py"` or `fd` |
| Search content | `grep -r "TODO" src/` or `rg` |
| HTTP requests | `curl` or `wget` |
| Database queries | `psql -c "SELECT ..."` or `sqlite3` |
| Send email | `sendmail` or a Python/curl one-liner |
| Process CSV | `awk`, `csvkit`, `python -c "..."` |
| Run tests | `pytest`, `npm test`, `go test ./...` |
| Git operations | `git log`, `git diff`, `git commit` |
| Install packages | `pip install`, `npm install`, `brew install` |
| Call APIs | `curl -X POST https://api.example.com/...` |

### Real usage examples

```bash
# Explore the codebase
bash("find . -type f -name '*.py' | head -20")

# Run tests and capture failures
bash("pytest tests/ -v 2>&1 | tail -30")

# Query a database
bash("psql $DATABASE_URL -c 'SELECT count(*) FROM claims WHERE status = \\'pending\\''")

# Download and inspect data
bash("curl -s https://data.example.com/claims.json | jq '.claims | length'")

# Run a data pipeline
bash("python scripts/process-claims.py < data/input.json > data/output.json")

# Check system state
bash("df -h && free -m && uptime")

# Git status
bash("git log --oneline -10")
```

### Common pitfalls

- **Output truncation — tail, not head:** If your command produces 500 lines, the agent sees the LAST 100, not the first. For build errors, this is usually fine (errors at the end). For log analysis, you may need to pipe through `grep` or `head` explicitly.

  ```bash
  # BAD: agent might miss errors in the middle
  bash("make build")

  # BETTER: filter for errors
  bash("make build 2>&1 | grep -E 'error|warning' | head -50")
  ```

- **No interactive input:** Commands that wait for user input will hang until timeout. Always pass flags to avoid prompts (`-y`, `--yes`, `--non-interactive`).

  ```bash
  # BAD: hangs waiting for confirmation
  bash("apt-get install postgresql")

  # GOOD
  bash("apt-get install -y postgresql")
  ```

- **Working directory:** The shell starts in the project root. If you need a different directory, use `cd` within the command.

  ```bash
  bash("cd frontend && npm test")
  ```

- **Environment variables:** The shell inherits the parent environment. Secrets set in the terminal are available. Don't hardcode credentials in commands.

- **Long-running commands:** For processes that take minutes (large builds, data imports), prefer running them and checking exit code. The agent can rerun if needed.

## Why Four Is Enough

Here's what the four tools can collectively do:

```
read     → understand anything (files, dirs, images)
write    → create anything
edit     → change anything (surgically)
bash     → do anything (every CLI tool on the system)
```

The key is **composition**:

1. `bash("pip install requests")` — install a library
2. `write("fetch-data.py", ...)` — create a script using it
3. `bash("python fetch-data.py > data.json")` — run it
4. `read("data.json")` — read the result
5. `write("report.md", ...)` — generate a report

Five tool calls. You've just built a data pipeline. No "download_data" tool needed.

> "The bash tool means Pi never needs a 5th tool. Whatever you can think of, bash can do it."

---

## Tool Selection Guide

| Situation | Tool |
|-----------|------|
| Read a file to understand it | `read` |
| List directory contents | `read` (on a directory path) |
| Create a new file | `write` |
| Change a few lines in an existing file | `edit` |
| Replace an entire file | `write` |
| Run a command / script | `bash` |
| Search for text across files | `bash` with `grep` or `rg` |
| Install software | `bash` |
| Call an API | `bash` with `curl` |
| Make a file executable | `bash` with `chmod` |
| Read a large file in chunks | `read` with `offset`/`limit` |

## Practice

### Exercise 1

The agent needs to change the database host in `config/settings.py` from `"localhost"` to `"prod-db.internal"`. The file is 800 lines long. Which tool should it use, and why?

<details>
<summary>Answer</summary>

Use `edit`. The file is 800 lines but only one value changes. `write` would require sending all 800 lines. `edit` with `old_text="localhost"` and `new_text="prod-db.internal"` — but only if "localhost" appears exactly once. If it appears multiple times, include surrounding context to make it unique.

</details>

---

### Exercise 2

An agent needs to find all Python files that import `requests` and count them. Write the bash command.

<details>
<summary>Answer</summary>

```bash
grep -rl "^import requests\|^from requests" . --include="*.py" | wc -l
```

Or with ripgrep:

```bash
rg -l "^import requests|^from requests" --type py | wc -l
```

</details>

---

### Exercise 3

The agent runs `bash("python process.py")` and gets back: exit code 1, output empty. What should it do next?

<details>
<summary>Answer</summary>

Rerun with stderr captured: `bash("python process.py 2>&1")`. Python errors go to stderr. The original command only captured stdout. The agent should always use `2>&1` when debugging failures, or check: `bash("python process.py 2>&1 | tail -20")`.

</details>
