# 3.1 The While Loop + 4 Tools Pattern

**Pi's architecture in one diagram**

---

## The Core Loop (Simplified)

```python
# Pseudocode - Pi's main loop
context = []

while True:
    # 1. Send context to LLM
    response = llm.chat(
        system_prompt="You have 4 tools: read, write, edit, bash",
        history=context
    )
    
    # 2. Check if LLM wants to use tools
    if response.tool_calls:
        for tool_call in response.tool_calls:
            # 3. Execute tool
            result = execute_tool(tool_call)
            
            # 4. Add result to context
            context.append({
                "role": "tool",
                "content": result
            })
    else:
        # LLM is done (no more tool calls)
        print(response.message)
        break
```

**That's the entire agent.** No complex orchestration.

## Step-by-Step Walkthrough

### Your request:
> "Create a script that counts errors in logs"

---

### **Turn 1:**

**LLM thinks:** "I need to see what files are available"

**Tool call:**
```json
{
  "tool": "bash",
  "command": "ls -la"
}
```

**Pi executes:** Runs `ls -la`, captures output

**Result added to context:**
```
[file listing showing logs/ directory]
```

---

### **Turn 2:**

**LLM thinks:** "I see a logs directory. Let me check what's inside"

**Tool call:**
```json
{
  "tool": "bash",
  "command": "ls logs/"
}
```

**Result:**
```
app.log
app.log.1
app.log.2
```

---

### **Turn 3:**

**LLM thinks:** "Let me check the format of one log file"

**Tool call:**
```json
{
  "tool": "read",
  "path": "logs/app.log"
}
```

**Result:** (first 100 lines of log)

---

### **Turn 4:**

**LLM thinks:** "I understand the format. I'll create a script"

**Tool call:**
```json
{
  "tool": "write",
  "path": "count-errors.sh",
  "content": "#!/bin/bash\ngrep -c 'ERROR' \"$1\"\n"
}
```

**Result:**
```
File created: count-errors.sh
```

---

### **Turn 5:**

**LLM thinks:** "Make it executable"

**Tool call:**
```json
{
  "tool": "bash",
  "command": "chmod +x count-errors.sh"
}
```

**Result:**
```
Exit code: 0 (success)
```

---

### **Turn 6:**

**LLM thinks:** "Test it"

**Tool call:**
```json
{
  "tool": "bash",
  "command": "./count-errors.sh logs/app.log"
}
```

**Result:**
```
42
```

---

### **Turn 7:**

**LLM response (no tool calls):**
> "Done! I've created `count-errors.sh` which counts ERROR lines in log files.
> 
> Usage: `./count-errors.sh <logfile>`
> 
> I tested it on `logs/app.log` and found 42 errors."

**Loop ends** (no more tool calls).

---

## Key Observations

### 1. **LLM decides what to do next**
Pi doesn't have a fixed plan. The LLM figures it out step by step.

### 2. **Each tool result → back to LLM**
LLM sees results immediately, can adjust course.

### 3. **Context grows**
```
Turn 1: system prompt
Turn 2: + tool result
Turn 3: + tool result
...
Turn N: full history
```

### 4. **Loop ends when LLM stops calling tools**
No tool calls = "I'm done"

---

## What Makes This Different?

### **Traditional agents:**
```
1. Plan entire workflow
2. Execute steps
3. Return result
```

**Problem:** Can't adapt if something goes wrong.

---

### **Pi:**
```
1. Try something
2. See result
3. Decide next step
4. Repeat
```

**Benefit:** Can adapt, explore, recover from errors.

---

## The Four Tools (Quick Reference)

### 1. **read(path)**
```bash
# Behind the scenes
cat /path/to/file
```

### 2. **write(path, content)**
```bash
# Behind the scenes
echo "$content" > /path/to/file
```

### 3. **edit(path, changes)**
```bash
# Behind the scenes (simplified)
# Apply diff or search-replace
```

### 4. **bash(command)**
```bash
# Behind the scenes
bash -c "$command"
```

---

## Why This Works

### **LLMs are trained on this pattern**
From the podcast:
> "Sonnet 3.5 and later models are specifically trained via reinforcement learning to keep going until tests pass, tasks complete, etc."

They've seen thousands of examples of:
1. Read file
2. Write code
3. Run tests
4. Fix errors
5. Repeat until success

---

## Limitations

### 1. **Context window**
Eventually, history gets too long. Pi truncates or summarizes.

### 2. **No guarantees**
LLM might get stuck, make wrong decisions, etc.

### 3. **Cost**
Each turn = API call (though small with compact tools).
