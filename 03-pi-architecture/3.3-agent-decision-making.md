# 3.3 How the Agent Decides What to Do

**Inside the LLM's reasoning loop**

---

## The Model Is Thinking

Between each tool call, the LLM generates a response. That response has two parts:

1. **Reasoning text** (often called "scratchpad" or "thinking") — the model works through what it knows and what to do next
2. **Tool call(s)** — zero or more structured tool invocations

Pi feeds the model the full conversation history on every turn. The model sees:

```
[system prompt]
[user: "Create a script that counts errors in logs"]
[tool result: ls -la output]
[tool result: cat logs/app.log output]
[tool result: file created]
...
```

At each step, the model decides: **what do I need to know, and what should I do?**

---

## Think Before Acting

Well-trained models follow a consistent pattern:

1. **Explore before touching:** Read files, list directories, understand the environment before writing anything
2. **Verify before assuming:** Don't guess what a file contains — read it
3. **Test after creating:** Run the script, don't just write it
4. **Confirm success:** Check the output, don't assume the command worked

This is why you typically see an agent do 2–3 read/bash calls before it writes anything. It's not being slow — it's avoiding costly mistakes.

> "The model that jumps straight to writing code without reading the codebase will make wrong assumptions about imports, naming conventions, existing utilities. The model that reads first writes less code and better code."

---

## Context Accumulates

Every tool result is added to the conversation history. The model on turn 8 sees everything from turns 1–7.

```
Turn 1: "ls -la" → [directory listing]
Turn 2: "cat config.json" → [config contents]
Turn 3: "cat src/main.py" → [source code]
Turn 4: write new file → [success]
Turn 5: "python new_file.py" → [error output]
Turn 6: edit fix → [diff applied]
Turn 7: "python new_file.py" → [success]
Turn 8: [model sees full history, knows task is done]
```

This accumulated context is how the agent "remembers" what it's done and what it learned. There's no separate memory system — the history IS the memory.

## Decision Tree Walkthrough: "Create a script that counts errors in logs"

Let's trace the full decision process, including the LLM's reasoning at each step.

---

### Turn 1: Orient

**LLM reasoning:** "I need to create a script. Before writing anything, I should understand the environment — what files exist, what the working directory looks like."

**Tool call:**
```json
{ "tool": "bash", "command": "ls -la" }
```

**Result:**
```
total 48
drwxr-xr-x  8 blake  staff   256 Mar 15 09:00 .
drwxr-xr-x 12 blake  staff   384 Mar 15 08:55 ..
-rw-r--r--  1 blake  staff   124 Mar 15 08:00 README.md
drwxr-xr-x  4 blake  staff   128 Mar 15 09:00 logs
drwxr-xr-x  2 blake  staff    64 Mar 15 08:00 scripts
```

**LLM learns:** There's a `logs/` directory and a `scripts/` directory. Will look at both.

---

### Turn 2: Explore logs

**LLM reasoning:** "I see a logs directory. I need to understand the log format before writing a counting script."

**Tool call:**
```json
{ "tool": "bash", "command": "ls logs/ && head -5 logs/app.log" }
```

**Result:**
```
app.log
app.log.1
app.log.2

2026-03-15 09:01:23 INFO  Server started on port 8080
2026-03-15 09:01:24 ERROR Failed to connect to database: timeout
2026-03-15 09:01:25 WARN  Retrying connection (attempt 1/3)
2026-03-15 09:01:26 ERROR Failed to connect to database: timeout
2026-03-15 09:01:27 INFO  Connection established
```

**LLM learns:** Log format is `YYYY-MM-DD HH:MM:SS LEVEL message`. Errors use the word `ERROR`. There are multiple rotated log files.

---

### Turn 3: Create the script

**LLM reasoning:** "I understand the format. The script needs to handle multiple log files (the rotated ones too). I'll write it to `scripts/` where existing scripts live."

**Tool call:**
```json
{
  "tool": "write",
  "path": "scripts/count-errors.sh",
  "content": "#!/bin/bash\n# count-errors.sh - Count ERROR lines in log files\n# Usage: ./count-errors.sh [logfile...]\n# Default: counts all logs/app.log*\n\nLOG_FILES=\"${@:-logs/app.log*}\"\n\necho \"Error counts:\"\nfor f in $LOG_FILES; do\n    count=$(grep -c ' ERROR ' \"$f\" 2>/dev/null || echo 0)\n    echo \"  $f: $count\"\ndone\n\necho \"\"\necho \"Total: $(grep -h ' ERROR ' $LOG_FILES 2>/dev/null | wc -l)\"\n"
}
```

**Result:**
```
File written: scripts/count-errors.sh (298 bytes)
```

---

### Turn 4: Make it executable

**LLM reasoning:** "Shell scripts need execute permission."

**Tool call:**
```json
{ "tool": "bash", "command": "chmod +x scripts/count-errors.sh" }
```

**Result:**
```
Exit code: 0
```

---

### Turn 5: Test it

**LLM reasoning:** "I should verify it works before telling the user it's done."

**Tool call:**
```json
{ "tool": "bash", "command": "./scripts/count-errors.sh" }
```

**Result:**
```
Error counts:
  logs/app.log: 42
  logs/app.log.1: 87
  logs/app.log.2: 103

Total: 232
```

---

### Turn 6: Done (no tool calls)

**LLM response:**
> "Done! I've created `scripts/count-errors.sh` which counts ERROR lines across all log files.
>
> ```
> $ ./scripts/count-errors.sh
> Error counts:
>   logs/app.log: 42
>   ...
> Total: 232
> ```
>
> You can also pass specific files: `./scripts/count-errors.sh logs/app.log`"

**Loop ends.** No tool calls = task complete.

## Recovery from Errors

When a tool returns an error (non-zero exit code, file not found, etc.), the agent sees the error and responds. It doesn't crash. It adapts.

### Example: Script fails on first run

**Turn 5 (bad result):**
```
Exit code: 1
Output:
./scripts/count-errors.sh: line 9: syntax error near unexpected token `done'
```

**LLM reasoning:** "There's a syntax error on line 9. I need to look at the script I wrote."

**Turn 6:**
```json
{ "tool": "read", "path": "scripts/count-errors.sh" }
```

**LLM sees the script, identifies the issue, fixes it:**

**Turn 7:**
```json
{
  "tool": "edit",
  "path": "scripts/count-errors.sh",
  "old_text": "for f in $LOG_FILES\n    count=",
  "new_text": "for f in $LOG_FILES; do\n    count="
}
```

**Turn 8:** Rerun the test. If it passes, done.

This is the recovery loop. The agent keeps trying until it succeeds or determines it can't.

## Parallel vs Sequential Execution

Some Pi implementations support **parallel tool calls** — issuing multiple tool calls in a single turn.

### Sequential (always safe):
```
Turn 1: read file A → result
Turn 2: read file B → result
Turn 3: write output → result
```

### Parallel (when reads are independent):
```
Turn 1: [read file A, read file B, read file C] → [result A, result B, result C]
Turn 2: write output → result
```

**When parallel is appropriate:**
- Reading multiple independent files to understand the codebase
- Running independent bash checks (disk space AND network status)
- Any set of reads where order doesn't matter

**When sequential is required:**
- When each step depends on the previous result
- When writes must happen before reads
- When the output of one command is the input to another

Most LLMs will naturally parallelize reads and serialize writes without being told to.

---

## When Does the Agent Stop?

The loop ends when the LLM produces a response with **zero tool calls**. This happens when:

1. **Task complete:** Everything requested has been done and verified
2. **Blocked:** The agent needs information it can't get (e.g., a password, a decision from the user)
3. **Giving up:** After repeated failures, the agent reports what it tried and why it failed
4. **Asking a clarifying question:** The agent realized the request is ambiguous

The last message (no tool calls) is the agent "speaking" to the user. This is the only message the user typically sees — everything before it was internal.

---

## The Decision at Each Turn

Simplified decision tree:

```
Do I have enough information to complete the task?
├── No → Read files / explore the environment
│
Do I need to create something new?
├── Yes → write()
│
Do I need to modify something existing?
├── Yes → edit() (small change) or write() (full rewrite)
│
Do I need to run something?
├── Yes → bash()
│
Did the last thing succeed?
├── No → Read the error, figure out why, fix it, retry
│
Is the task done and verified?
├── Yes → Respond with no tool calls
├── No → Continue the loop
```

## Practice: Trace a Multi-Step Scenario

**Scenario:** "I have a JSON file at `data/employees.json`. Create a CSV with just the name and email fields."

Trace the tool calls yourself before looking at the answer.

<details>
<summary>Answer</summary>

**Turn 1:** `bash("cat data/employees.json | head -20")` — understand the JSON structure

**Turn 2:** Based on the structure, either:
- `write("convert.py", ...)` — write a Python script to do the conversion
- Or directly: `bash("python3 -c \"import json,csv,sys; data=json.load(open('data/employees.json')); w=csv.DictWriter(sys.stdout, ['name','email']); w.writeheader(); [w.writerow({'name':r['name'],'email':r['email']}) for r in data] \" > data/employees.csv")`

**Turn 3:** `bash("head -5 data/employees.csv")` — verify the output

**Turn 4:** No tool call — report success, show sample output

The key decisions:
- Read first to understand the structure (what are the field names?)
- Verify the output (don't just report success — check it)
- Inline Python in bash is fine for simple transforms; for complex ones, write a script

</details>
