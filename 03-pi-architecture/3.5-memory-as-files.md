# 3.5 Memory as Files

**Persistent state without infrastructure**

---

## The Conventional Approach

Most agent frameworks treat memory as a special subsystem:

```
User: "Remember that I prefer PostgreSQL over MySQL"
Agent: [calls embed("PostgreSQL preference") → vector DB]

Later...
User: "What database should I use?"
Agent: [calls semantic_search("database choice") → retrieves embedding → context]
```

This requires:
- A vector database (Pinecone, Weaviate, Chroma, pgvector…)
- An embeddings model or API
- A RAG pipeline
- Infrastructure to run and maintain all of it

Pi's approach:

```bash
echo "User prefers PostgreSQL over MySQL" >> memory/preferences.md
```

---

## The File System IS Memory

Pi has no memory subsystem. The file system is the memory.

```bash
# Remember something
write("memory/preferences.md", "User prefers PostgreSQL over MySQL\n")

# Recall something
read("memory/preferences.md")

# Search memory
bash("grep -r 'database' memory/")

# Daily notes
write("memory/2026-03-15.md", "## Today\n- Fixed connection pool bug\n- Deployed v2.3\n")

# What happened last week?
bash("cat memory/2026-03-08.md")

# What have we been working on this month?
bash("ls memory/ | grep 2026-03")
```

This is the entire memory system.

## Memory File Patterns

### Preferences file

`memory/preferences.md` — persists user preferences across sessions:

```markdown
# User Preferences

## Code style
- Python over JavaScript for scripts
- Black formatter, 88-char line length
- Type hints required

## Workflows
- Always run tests before committing
- Reports go in reports/ directory
- Data files always in data/ directory

## Communication
- Concise responses preferred
- Show commands before running them
- Ask before deleting files
```

### Daily notes

`memory/2026-03-15.md` — what happened today:

```markdown
# 2026-03-15

## Completed

## Context Files: The Automatic Injection

Two special filenames get **automatically injected into the system prompt** at the start of every session:

- `AGENTS.md` (Pi convention)
- `CLAUDE.md` (Claude Code convention — Pi supports this too)

You don't have to ask the agent to read them. They're always in context.

```
[system prompt]
[contents of AGENTS.md or CLAUDE.md, if it exists]
[user message]
```

This is how you give the agent persistent domain knowledge about your project without repeating yourself every session.

## Writing a Good AGENTS.md

An effective `AGENTS.md` contains the things you'd tell a new developer on the first day:

### What to include

**Project overview** (1-2 sentences):
```markdown
# Claims Processing System

Backend service for insurance claims: intake, validation, routing, and reporting.
```

**Tech stack and versions:**
```markdown
## Stack
- Python 3.11, FastAPI, SQLAlchemy
- PostgreSQL 15 (claims_db)
- Redis for job queues
- Deployed on AWS ECS
```

**Directory structure (especially non-obvious parts):**
```markdown
## Key directories
- `src/` — application code
- `data/` — input files (never commit large files)
- `reports/` — generated reports (gitignored)
- `scripts/` — one-off scripts, utilities
- `memory/` — agent working memory
```

**Data schemas (the most valuable thing):**
```markdown
## Data schemas

### claims table
- `claim_id` UUID primary key
- `policy_id` UUID (FK to policies)
- `status` enum: pending|reviewing|approved|denied|closed
- `amount_cents` integer (NOT dollars)
- `created_at` timestamp with time zone

### JSON input format (data/claims-*.json)
[{"claim_id": "...", "policy_id": "...", "amount": 150.00, ...}]
Note: JSON uses dollars (float), DB uses cents (int)
```

**Naming conventions:**
```markdown
## Conventions
- Scripts: kebab-case.py (not snake_case)
- Reports: YYYY-MM-DD-description.md
- DB queries: always use read replica ($DB_HOST_RO) for SELECT
- Monetary amounts: always in cents in DB, always in dollars in user-facing output
```

**Common operations:**
```markdown
## Common tasks

### Run the test suite
```bash
pytest tests/ -v
```

### Generate a report
```bash
python scripts/generate-report.py --type weekly > reports/$(date +%Y-%m-%d)-weekly.md
```

### Connect to DB
```bash
psql -h $DB_HOST -U $DB_USER -d claims_db
```
```

A well-written `AGENTS.md` means you never have to re-explain your project. The agent knows it on session one and session one hundred.

## Search Without Semantic Search

The conventional argument for vector DBs is that they support "semantic search" — find documents similar in meaning even without exact keyword matches.

In practice, for developer workflows, keyword search is almost always sufficient:

```bash
# Find anything about database connections
grep -r "connection" memory/ --include="*.md"

# Find all files mentioning a specific claim
grep -r "claim_id: abc-123" . --include="*.json" --include="*.md"

# Find recent notes about a topic
grep -r "PostgreSQL" memory/ --include="2026-*.md" -l

# Find all TODO items across memory files
grep -r "TODO\|FIXME\|- \[ \]" memory/
```

`grep` and `rg` (ripgrep) are fast even on thousands of files. For the typical developer's memory directory (hundreds of notes), grep returns in milliseconds.

> "Code is truth. Code is the ground truth. It's also evolving and I don't need another place that I need to maintain."
> — Mario Zechner

## Contrast: Files vs Vector DB

| | Files | Vector DB |
|--|-------|-----------|
| **Setup** | None | Hours to days |
| **Infrastructure** | None (files already exist) | DB server, embedding API |
| **Search** | `grep`, `rg`, `find` | Semantic similarity query |
| **Transparency** | Open in any editor | Opaque embeddings |
| **Agent can write** | Yes (write tool) | Requires embed + insert calls |
| **Human can read/edit** | Yes | No |
| **Works offline** | Yes | No (usually) |
| **Scales to** | Millions of files | Billions of vectors |
| **Best for** | Developer workflows, structured data | Large-scale unstructured retrieval |

For most agent use cases — especially developer and enterprise workflows — files win. The cases where vector DBs win: when you have millions of documents and need fuzzy semantic search across all of them.

---

## Limitations

Files-as-memory isn't perfect. Be honest about when it breaks down:

**Large-scale retrieval:** If you have 500,000 email threads and need "find emails similar to this one," grep won't help you. Vector search is genuinely better here.

**Unstructured search:** "Find all memories where the user seemed frustrated" requires semantic understanding. Keyword search can't do this reliably.

**Deduplication:** If you append to memory files without structure, you'll eventually have the same information written multiple times. Needs occasional cleanup.

**Concurrent writes:** Multiple agents writing to the same file simultaneously can corrupt it. Fine for single-agent workflows; needs locking for multi-agent.

For the vast majority of developer and enterprise workflows, none of these are real problems.

## Context Compaction

The conversation history grows every turn. Eventually it hits the model's context window limit (128K, 200K tokens depending on the model).

Pi handles this with **context compaction**: when the history gets long, Pi summarizes older messages into a condensed form and replaces them with the summary.

```
Before compaction:
[system prompt]
[turn 1–50: detailed history, ~80K tokens]
[turn 51–60: recent turns]

After compaction:
[system prompt]
[summary of turns 1–50: ~5K tokens]
[turn 51–60: recent turns]
```

The summary preserves the key facts (what was decided, what was built, what worked, what failed) while discarding the detailed back-and-forth.

**What this means for you:**
- Very long sessions will lose detail on early turns
- Important decisions should be written to files, not just discussed in chat
- `AGENTS.md` and memory files survive compaction — they're re-injected fresh each time

This is why the "write things down" habit matters: files are permanent, conversation history is not.

## Practice: Write an AGENTS.md

Write an `AGENTS.md` for this scenario:

> You work at an insurance company. You're building an agent that helps generate quarterly reports from claims data. The claims data lives in a PostgreSQL database (claims_db). Reports go to a `reports/` directory. The team uses Python 3.11 and prefers pandas for data work.

<details>
<summary>Example answer</summary>

```markdown
# Claims Reporting Agent

Generates quarterly and ad-hoc reports from insurance claims data.

## Stack
- Python 3.11, pandas, psycopg2
- PostgreSQL 15 (claims_db)
- Reports: markdown format, saved to reports/

## Database

### Connection
```bash
psql -h $DB_HOST -U $DB_USER -d claims_db
# Read replica (use for SELECT):
psql -h $DB_HOST_RO -U $DB_USER -d claims_db
```

### Key tables
- `claims` — one row per claim
  - `claim_id` UUID
  - `policy_id` UUID
  - `status` enum: pending|reviewing|approved|denied|closed
  - `amount_cents` int (divide by 100 for display)
  - `adjuster_id` UUID (nullable)
  - `opened_at`, `closed_at` timestamps

- `policies` — one row per policy
  - `policy_id` UUID
  - `holder_name` text
  - `policy_type` enum: auto|home|life|health

- `adjusters` — adjuster roster
  - `adjuster_id` UUID
  - `name` text
  - `region` text

## Conventions
- Monetary amounts: always cents in DB, always dollars ($X,XXX.XX) in reports
- Report filenames: YYYY-MM-DD-{type}-report.md
- Always use read replica for SELECT queries
- Use pandas for any aggregation over >1000 rows

## Common report types
- **Quarterly summary:** total claims, approvals, denials, average processing time
- **Adjuster performance:** claims per adjuster, approval rates, avg resolution time
- **Policy type breakdown:** claims by policy type and region

## Output
Reports go to reports/ directory. Format: markdown with tables.
```

</details>
